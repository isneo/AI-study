2026-05-24 AI Agent 技术架构讨论纪要

背景

群内原定围绕三件事持续对齐：

  1. 横向认知到风控应用：泛化能力、延展性。
  2. Agent 业界调研、架构选型，以及什么适合风控场景。
  3. 差异化故事：为什么我们跑得快、怎么跑、和行业有什么不同。

2026-05-24 的讨论主要聚焦第 2 点，也延伸到第 1 点和第 3 点的表达框架。

核心共识

1. Agent 架构不是单一框架问题

讨论中把 Agent 相关能力拆成几个层次理解：

  • LangGraph 更像底层 orchestration runtime。
  • Pi / DeerFlow 更像完整 agent harness。
  • OpenAI / Anthropic SDK 更像模型厂商提供的 agent runner、tool、handoff、trace SDK。

这个拆法的价值是避免把所有产品或框架混在一起比较，而是先判断它解决的是运行时、工具调用、流程编排、观测追踪，还是业务 harness。

2. Memory 需要独立研究

大家认为「记忆」强依赖业务场景，不能只停留在泛泛概念。后续需要结合技术机制和风控业务一起展开，例如：

  • 哪些信息值得沉淀为长期记忆。
  • 哪些上下文只适合单次任务内使用。
  • 如何避免 memory 污染判断。
  • 如何验证 memory 对业务效果有真实收益。

3. Runtime、编排和观测要一起看

架构讨论涉及：

  • 编排：链式流程、handoff、可能的多 agent 协作。
  • 工具调用：模型如何调用业务工具、查询数据、写回结果。
  • 观测：trace、日志、效果评估，群内提到类似 Langfuse 这一类观测能力。
  • 上下文传递：对 a-to-a 交互能否有效解决上下文传递问题持保留态度。

4. 循环结束条件必须工程化

Agent 循环不能只依赖模型自觉停止。讨论中认为结束条件至少应包括：

  • 任务已完成。
  • 验证通过。
  • 达到系统上限，例如重试次数、上下文长度、预算或时间上限。
  • 进入兜底策略，例如人工介入、降级为确定性流程、记录失败原因。

5. 技术落地要区分「集成」和「借鉴」

大家明确区分两类策略：

  • 可直接集成：通用性强、和业务边界匹配的框架，例如 LangGraph、Spring AI 这类底层能力。
  • 只做思路借鉴：闭源、场景不贴合、工程边界不一致的方案，只参考设计思想，不强行接入。

自研部分需要说明为什么必须自研，以及自研解决的是业务约束、工程集成、效果验证还是组织协作问题。

对风控场景的启发

B 端风控场景很多时候存在 Ground Truth，更容易形成数据闭环。这个特点可以成为 Agent 落地的关键优势：

  • 更容易定义任务是否完成。
  • 更容易对输出做自动或半自动验证。
  • 更容易沉淀评估集和回归测试。
  • 更适合把 agentic workflow 与传统工程流程结合，而不是完全替代。

需要继续回答的问题

  • Memory 在风控场景中到底存什么、怎么更新、如何验证收益。
  • Agent 循环的结束条件和兜底策略如何落到系统设计。
  • 技术选型中哪些部分直接集成，哪些部分借鉴，哪些必须自研。
  • 如果重新做，为什么一开始不是 agentic workflow，演进路径会怎样变化。
  • 面试或对外表达时，如何把「横向认知 -> 风控应用 -> 差异化故事」串成一个完整叙事。
