# 近期代办

这份清单只保留近期最值得推进、能直接产出学习或面试材料的事项。

| 任务 | 解决什么问题 | 完成后的收益 | 具体内容 |
|---|---|---|---|
| Memory 专题拆解 | 避免只讲“Agent 有记忆”这种泛泛概念 | 能讲清楚风控场景中 memory 的价值、边界、风险和验证方式 | 按长期记忆、短期上下文、业务事实、用户偏好、案例经验、SOP 技能分别整理：存什么、谁能写、何时更新、如何回滚、如何验证收益 |
| Agent 循环结束条件和兜底策略 | 避免 Agent 无限循环、提前停止或失败后不可控 | 形成一套可落地的 runtime 设计 checklist | 明确任务完成、验证通过、重试上限、上下文上限、预算上限、人工介入、降级到确定性流程、失败记录这些条件 |
| 落地选型矩阵 | 区分直接集成、思路借鉴和自研，避免框架名堆砌 | 面试和方案评审时能解释为什么选、为什么不选 | 对 LangGraph、Spring AI、OpenAI/Anthropic SDK、Pi/DeerFlow、MCP/A2A、Langfuse 等按层级、适用场景、风险、集成成本做一张矩阵 |
| 风控 Agent 项目叙事 | 把历史资料变成可讲的项目故事 | 能回答“为什么做、怎么做、效果如何、和行业有什么不同” | 基于横向认知 -> 风控应用 -> 差异化故事三段式，整理 5 分钟版、15 分钟版、追问版 |
| 系统演进复盘 | 回答“为什么一开始不是 agentic workflow，如果重做会怎样” | 形成更可信的工程判断，而不是事后套概念 | 梳理传统 workflow 的合理性、触发 agentic workflow 的条件、迁移路径、风险控制和阶段性收益 |
| 资料短摘要补齐 | 现在资料多但入口仍偏重文件名 | 让后来者不用逐个打开大文件也能知道先看什么 | 给 HTML、Markdown、PDF 各补 3-5 句摘要、适合人群、阅读顺序，更新到 `resource-catalog.md` |
| 图片资料二次补充 | 当前图片只能拿到不可读缓存，缺少原始截图类材料 | 让图示、截图和讨论上下文更完整 | 从原始客户端手动另存确认为学习资料的图片，放入 `assets/images/`，避免导入生活照、聊天截图和隐私图片 |

## 推荐推进顺序

1. 先做“风控 Agent 项目叙事”和“落地选型矩阵”，最直接服务面试和方案表达。
2. 再做“Memory 专题拆解”和“循环结束条件”，补齐架构深度。
3. 最后补“资料短摘要”和“图片资料”，提升资料包可读性。
