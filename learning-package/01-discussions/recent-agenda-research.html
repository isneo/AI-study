AI Native 风控 Agent：Memory 调研、评估飞轮与落地架构

文档分两大部分：第一部分先做 Memory 和“为什么跑得快”的深度调研，体现行业认知和横向迁移能力；第二部分围绕四块普适架构展开风控整体设计和落地方案。生命周期、飞轮和业务流程均用图示表达。

版本：框架重排版 更新时间：2026-05-25 重点：Memory 行业认知 + 评估数据飞轮 + 风控架构落地

0. 结论先行

总判断：风控 Agent 的落地不应该从“套哪个 Agent 框架”开始，而应该从两个能力底座开始：一是 Memory，让系统能跨任务复用领域知识、案例经验和策略口径；二是评估与数据飞轮，让系统把每次运行、人工修正和申诉后验变成下一轮迭代资产。

推荐架构：第二部分采用四块普适架构：底层是可观测、评估和数据飞轮，负责让系统持续变好；中间层是编排和运行时，负责让系统跑起来；顶层左侧是 Memory & Context，负责让系统跑得好；顶层右侧是约束和验证，负责让系统别出错。

Memory

先建立行业认知

从 context、RAG、state 的边界讲起，再到 semantic / episodic / procedural / reflective memory。

Flywheel

跑得快靠评估闭环

每个 case 都能变成 trace、反馈、归因、评估样本、候选 memory 或策略 backlog。

Runtime

中间层是工程底座

技术框架、工具调用、模型调用、状态、停止条件、HITL 和系统执行链路都落在这里。

Domain

上层是业务技术交界

Memory/Context 与约束/验证同时包含业务领域知识、策略逻辑和技术实现。

第一部分 · 深度调研：Memory 与“为什么跑得快”

1. Memory 行业调研

Memory 的行业价值在于：让 Agent 不再是每次从零开始的模型调用，而是能跨任务保留状态、复用经验、吸收反馈、继承组织知识的长期系统。风控只是这套能力迁移到 B 端高风险场景后的一个应用。

1.1 Memory 的边界

概念 解决的问题 生命周期 典型实现 边界
Context 本次推理看什么。 一次调用或任务内。 Prompt、工具返回、检索片段、短期摘要。 多塞上下文不是长期记忆。
State 任务走到哪，已有变量是什么。 一次 workflow / thread。 状态机、checkpoint、任务表。 状态持久化主要服务恢复和回放。
RAG 从外部知识源取资料。 知识源长期存在，检索片段短期进入上下文。 向量库、BM25、GraphRAG、Rerank。 RAG 是检索机制，不天然包含经验写入和治理。
Memory 跨任务保留会影响未来行为的事实、经验和流程。 跨会话、跨任务、跨时间。 Memory store、case memory、profile、procedural skill、graph memory。 关键在抽取、治理、检索、更新、评估。

1.3 主流项目与研究路线

项目/研究 代表思想 对横向迁移的价值 迁移到风控的启发
CoALA 把 Agent 拆成 memory、action、decision-making 等认知部件。 提供跨业务分析框架，不被具体工具名绑住。 风控要区分语义事实、案例经验、审核流程和反思记忆。
Generative Agents / Reflexion 从观察、反馈和失败中形成反思，再影响后续行为。 说明不改模型参数，也能通过外部经验改进行为。 申诉成功、人工驳回、误判归因可形成候选 reflective memory。
Letta / MemGPT 把上下文窗口视为稀缺工作记忆，外部存储视为长期记忆。 强调 core memory 与 archival memory 的层级。 风控核心记忆应是边界和约束，历史 case 按需召回。
LangMem 面向 long-term memory 的抽取、管理、检索、版本历史。 把 memory 从聊天组件提升为工程模块。 适合借鉴候选记忆抽取、store、版本和管理机制。
Mem0 强调 memory layer 与 memory evaluation。 提醒 memory 要按任务收益评估，不是召回越多越好。 风控要看 memory 命中后的人工接受率、误引用率和处理时长。
Zep / Graphiti 用 temporal knowledge graph 组织实体关系和时间变化。 适合事实会演化、关系强的企业场景。 账号、设备、内容、举报、处罚、申诉天然适合时序图谱。
Databricks Memory Scaling 把 memory 视为新的 scaling 轴。 关注外部记忆增长后 Agent 是否更准、更快。 风控可用 case、trace、复核和申诉后验证明 memory scaling。

1.4 Memory 生产能力闭环

2. 为什么跑得快：评估体系与数据飞轮

“跑得快”不是说模型一次性就能审得更准，而是团队把反馈周期、错误归因和版本验证做短了。每个 case 都能产生可回放 trace、人工修正、申诉后验和业务结果；失败后能定位到知识、上下文、工具、Runtime、约束或评估缺口，再进入下一轮改进。

2.1 数据飞轮图

2.2 六层评估体系

评估层 回答的问题 典型指标 为什么提速
业务效果 风险有没有真变少，用户权益有没有受伤。 拦截率、漏放率、误杀率、申诉成功率、复核一致性。 让系统优化业务结果，而不是只优化模型输出观感。
Agent 判断 单个 case 判得准不准。 precision、recall、风险等级准确率、人工接受率、人审推翻率。 快速判断新模型、prompt、memory 是否值得灰度。
证据质量 证据是否取全，引用是否正确。 必要证据覆盖率、规则引用正确率、案例引用正确率、证据冲突率。 定位问题是证据缺失、工具缺失、规则解释错还是模型判断错。
放权治理 能不能扩大自治。 各风险档自决率、升级率、高风险转人工覆盖率。 放权用指标争取：先 shadow-run，再 canary，再自动化。
策略迭代 飞轮有没有真的转。 发现到上线耗时、回归集通过率、memory 命中收益、backlog 关闭率。 每次修复沉淀为 eval、memory、SOP、工具或策略改进。
工程审计 线上是否扛得住，出事能不能查清。 P95/P99 延迟、成本、工具成功率、case replay 覆盖率、回滚耗时。 让系统具备生产可信度，不停留在 demo。

2.3 评估集与反馈治理

分桶评估集

Golden Set 覆盖稳定规则，Boundary Set 覆盖灰区，Regression Set 防止重犯，Adversarial Set 防对抗，Fresh Sample 防分布漂移，Memory Eval Set 验证记忆收益。

Eval Asset

反馈先治理

资深审核员仲裁可信度高，申诉成功要先区分真误伤和规则变化，集中申诉和新账号批量反馈默认隔离，防止脏反馈带歪飞轮。

Anti Poisoning

版本可回滚

模型、prompt、memory、工具、规则和评估集版本绑定。任何改动都要能看到指标变化、影响样本、回滚路径和负责人。

第二部分 · 风控整体架构设计与方案落地

3. 四块总体架构

这套架构不是技术名词堆栈，而是一个生产系统的功能分层：底层让系统持续变好，中间层让系统跑起来，上层左侧让系统跑得好，上层右侧让系统别出错。

层级解释：中间层是工程底座，技术框架、工具协议、模型调用、状态、执行链路主要落在这里；上层两块是业务和技术的交界层，既包含风控领域知识、策略逻辑和证据规则，也包含 context building、memory retrieval、guardrail、validator 等技术机制。

4. 编排与运行时：让系统跑起来

运行时层负责把一个风控任务从进入系统到反馈回流跑通。它承载技术选型和系统执行链路：Spring AI 适合作为 Java 应用集成层，MCP 风格工具网关统一内部工具，LangGraph 的 state/checkpoint/HITL 思路可借鉴，OpenAI Agents SDK 可用于原型验证。

4.2 运行时层选型

能力 推荐 落位 理由
应用集成 Spring AI / 现有 Java LLM Adapter 中间层 如果风控后台是 Java/Spring，优先降低接入成本。

观测评估 Langfuse 自托管 / 内部 Trace & Eval 底层贯穿 生产必需，所有任务和改动都要可回放、可评分、可比较。

5. 上层领域能力：Memory & Context 与约束 & 验证

顶层左右两块不是纯技术层，而是业务与技术的交界层。左边负责把领域知识、当前案件和历史经验组织成可用上下文；右边负责把风控规则、权限、证据要求和人工责任链变成系统约束。

Memory & Context：让系统跑得好

包含策略口径、灰区解释、典型案例、申诉后验、审核 SOP、当前证据、用户画像、行为历史、设备关系和相似案例。技术上对应 context builder、memory retrieval、case retrieval、rerank、context budget。

领域认知层

约束 & 验证：让系统别出错

包含规则版本、证据引用、权限边界、高风险动作、转人工条件、输出 schema、停止条件和审计要求。技术上对应 guardrail、validator、policy engine、citation checker、HITL。

5.1 Memory 生命周期图

5.2 风控 Memory 分层

行业类型 风控映射 使用价值 治理边界
Semantic Memory 策略口径、灰区解释、业务线差异、质检标准。 减少口径漂移。 正式规则仍由策略系统发布。
Episodic Memory 举报 case、申诉成功、误伤/漏召、团伙处置轨迹。 支持相似案例召回和处置一致性。 单个案例不能直接泛化成规则。
Procedural Memory 审核 SOP、取证顺序、转人工条件、复核摘要模板。 让 Agent 按流程查证据。 不能绕过 runtime 的权限、预算和停止条件。
Reflective Memory 常见误判模式、工具缺口、规则冲突、评估回归结论。 让系统少犯重复错误。 必须先归因、评估、审批。
Graph / Temporal Memory 账号、设备、内容、举报、处罚、申诉的时序关系。 支持团伙、累犯、传播链和事实变化判断。 图谱是证据工具，不等同于全部 memory。

7. V1 实施计划

实施策略是先读后写、先辅助后自动、先评估后扩量。下面用路线图表达阶段推进，不用文字替代流程。

V1 验收标准

类别 标准 说明
功能 举报/申诉 case 能输出结构化证据链、规则引用、相似案例、建议结论、人工确认点。 结果必须可进入工作台，而不是聊天式回答。
可追溯 每条结论能追到工具返回、规则版本、案例来源和模型版本。 无来源结论不得进入正式建议。
可评估 每次改动能在固定 eval set 上跑回归，并产生 score 报告。 prompt、memory、工具、模型都必须版本化。
可控 高风险动作 100% 转人工；工具失败、证据冲突、预算上限均能停止。 不能出现无限循环或无记录降级。

8. 对外表达

5 分钟版本

我对这个项目的理解不是“给风控接一个 LLM”，而是把 Agent Memory、工具调用、Runtime、评估飞轮这些行业能力迁移到风控这样的高风险 B 端业务里。行业上 Memory 正从聊天历史和 RAG 组件，演进成 Agent 的长期状态层：它保存事实、案例、流程和反馈经验，并通过评估证明这些记忆真的让系统变好。
迁移到风控时，优势非常明显：风控天然有规则、case、证据、人工复核、申诉后验和策略版本，所以它不仅适合用 Agent 做跨系统取证和证据链整理，也适合验证 memory scaling 和 eval flywheel。我们的 V1 不做全自动处罚，而是从举报受理和申诉复盘切入，让 Agent 输出结构化证据链、风险解释、相似案例、处置建议和待人工确认点。
架构上我用四块来讲：底层是可观测、评估和数据飞轮，让系统持续变好；中间是编排和运行时，让系统跑起来；上层左边是 Memory & Context，让系统跑得好；上层右边是约束和验证，让系统别出错。中间层是工程底座，上层两块是业务和技术的交界层。
这个系统跑得快，不是因为模型一次性很强，而是第一天就设计了评估和数据飞轮。每个 case 都记录输入、工具调用、证据、规则版本、模型输出、人工修改和后续申诉结果。失败后能归因到模型、工具、规则、memory、Runtime 边界或评估缺口，再进入评估集、case memory、SOP、工具 backlog 或策略 backlog。
