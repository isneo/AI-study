0. 结论先行

总判断：风控适合建设 AI Native / Agent，但目标不是让模型绕过业务责任直接处罚，而是把证据链整理、规则引用、相似案例召回、风险解释、人工复核和后验归因做成可回放的生产系统。真正的护城河不只是单次模型效果，而是 Memory、评估集、Trace、工具和策略改进一起滚动的组织能力。

Memory 结论：Memory 不是把规则库、画像、图谱、案例库换个名字。规则、画像、图谱是 source of truth 或工具，原始案例是证据与回放材料；只有从 trace、人工修正、申诉后验中抽取出的稳定经验，经过版本、TTL、权限、citation、评估、灰度和回滚治理后，才应进入 Memory。

为什么跑得快：快来自评估和数据飞轮，而不是单次推理更激进。风控天然有人工复核、申诉结果、处罚后验、复犯与投诉等 ground truth 信号，可以把每个 case 转成 trace、反馈、错误归因、评估样本、候选 memory、工具缺口或策略改进项，让失败样本快速进入下一轮迭代。

推荐架构：四块分层最稳：底层是可观测、评估和数据飞轮；中间层是编排与运行时，承接工具调用、状态机、HITL、Trace 和版本发布；顶层左侧是 Memory & Context，组织长期经验与本次证据包；顶层右侧是约束与验证，把规则、权限、citation、schema、risk gate 和转人工条件工程化。

落地路径：V1 先做旁路建议和人工复核，优先跑通举报受理、申诉复盘、图片/多模态审核三条流程；验收看查证时长、证据覆盖率、引用正确率、人工接受率、申诉归因覆盖率、回归集通过率和回滚耗时。稳定后再进入 shadow-run、canary 和低风险半自动。

第一部分 · 深度调研：Memory 与“为什么跑得快”

1. Memory 行业调研

Memory 的行业价值在于：让 Agent 不再是每次从零开始的模型调用，而是能跨任务保留状态、复用经验、吸收反馈、继承组织知识的长期系统。风控只是这套能力迁移到 B 端高风险场景后的一个应用。

1.1 Memory 的边界

概念 解决的问题 生命周期 典型实现 边界 类比
Context 本次推理看什么。 一次调用或任务内。 Prompt、工具返回、检索片段、短期摘要。 多塞上下文不是长期记忆。 开会时桌面上摊开的材料。
State 任务走到哪，已有变量是什么。 一次 workflow / thread。 状态机、checkpoint、任务表。 状态持久化主要服务恢复和回放。 项目看板上当前做到哪一步。
RAG 从外部知识源取资料。 知识源长期存在，检索片段短期进入上下文。 向量库、BM25、GraphRAG、Rerank。 RAG 是检索机制，不天然包含经验写入和治理。 临时去档案室查资料带回会议室。
Memory 跨任务保留会影响未来行为的事实、经验和流程。 跨会话、跨任务、跨时间。 Memory store、case memory、profile、procedural skill、graph memory。 关键在抽取、治理、检索、更新、评估。 个人经验、组织知识库和复盘笔记。
一句话区分：Context 是“这次看什么”，State 是“任务走到哪”，RAG 是“从哪里找资料”，Memory 是“哪些经验值得长期留下并影响以后”。RAG 可以把 Memory 检索进 Context，State 可以沉淀为候选 Memory，但四者不是同一个东西。

1.3 主流项目与研究路线

项目/研究 代表思想 解决什么问题 实现原理 举例说明 对横向迁移的价值 迁移到风控的启发
CoALA 把 Agent 拆成 memory、action、decision-making 等认知部件。 解决 Agent 架构分析容易被具体框架名带偏的问题。 用认知架构拆分工作记忆、长期记忆、外部动作、内部推理和决策循环。 分析客服、风控、编码 Agent 时，都先问它的 memory、action、decision 分别是什么。 提供跨业务分析框架，不被具体工具名绑住。 风控要区分语义事实、案例经验、审核流程和反思记忆。
Generative Agents / Reflexion 从观察、反馈和失败中形成反思，再影响后续行为。 解决 Agent 每次都重复犯错、无法把历史反馈转成经验的问题。 把事件记录为观察，再由模型总结成反思或文字经验，后续任务检索这些经验。 一次任务失败后写下“下次先验证工具返回再下结论”，下一次运行时把这条经验放回上下文。 说明不改模型参数，也能通过外部经验改进行为。 申诉成功、人工驳回、误判归因可形成候选 reflective memory。
Letta / MemGPT 把上下文窗口视为稀缺工作记忆，外部存储视为长期记忆。 解决长对话和长期任务无法全塞进 context window 的问题。 把 memory 分成始终可见的 core memory、按需召回的 recall/archival memory，并让 Agent 管理读写。 个人助理始终记住用户身份和偏好，但历史项目细节只在相关任务里检索出来。 强调 core memory 与 archival memory 的层级。 风控核心记忆应是边界和约束，历史 case 按需召回。
LangMem 面向 long-term memory 的抽取、管理、检索、版本历史。 解决长期记忆工程化缺少抽取、管理和检索标准流程的问题。 从对话、运行轨迹或反馈中抽取候选 memory，写入 store，再按任务检索并维护版本历史。 从一次产品需求讨论里抽取“用户偏好短文档、先给结论”的记忆，未来写方案时自动召回。 把 memory 从聊天组件提升为工程模块。 适合借鉴候选记忆抽取、store、版本和管理机制。
Mem0 强调 memory layer 与 memory evaluation。 解决应用接入 memory 后不知道记忆是否真的有用的问题。 围绕用户、会话、Agent 抽取和检索记忆，并用专门评估指标衡量 memory 命中后的任务效果。 同一批任务分别开启/关闭 memory，对比成功率、响应质量、重复提问次数。 提醒 memory 要按任务收益评估，不是召回越多越好。 风控要看 memory 命中后的人工接受率、误引用率和处理时长。
Zep / Graphiti 用 temporal knowledge graph 组织实体关系和时间变化。 解决事实、人物、实体关系会随时间变化，纯向量检索难以表达关系和时序的问题。 把会话和业务事件抽取成实体、关系、时间边，再用图谱和时间维度检索相关事实。 用户换了公司、设备关系发生变化、某条事实过期，系统能按时间判断哪个事实仍有效。 适合事实会演化、关系强的企业场景。 账号、设备、内容、举报、处罚、申诉天然适合时序图谱。
Databricks Memory Scaling 把 memory 视为新的 scaling 轴。 解决单靠更大模型、更多 test-time compute 或长上下文仍无法复用组织经验的问题。 把外部记忆库规模和质量作为变量，观察 Agent 准确率、效率和可泛化能力是否随记忆增长提升。 客服 Agent 随着历史工单、成功解决方案和用户反馈增加，在新工单上更少重复探索。 关注外部记忆增长后 Agent 是否更准、更快。 风控可用 case、trace、复核和申诉后验证明 memory scaling。
一句话总结：技术领先点不在“接一个向量库”，而在把 Memory 做成 eval-driven 的长期状态层：从 trace 和人工反馈抽取候选记忆，按 semantic / episodic / procedural / graph memory 建模，叠加版本、TTL、冲突消解、权限和 citation 治理，再用 retrieval policy、rerank、context budget、回放评估和灰度实验验证命中后的任务收益；风控可迁移这套机制，把规则口径、相似案例、证据链、复核结论和申诉后验沉淀成可评估、可回滚、可持续变好的风险决策资产。

1.4 Memory 生产能力闭环

迁移原则：Memory 迁移到风控时，关键不是把规则库、画像、图谱等原始资产重新命名为记忆，而是把它们作为工具、检索源和权威数据源接入；只有从 trace、复核、申诉后验和人工反馈中抽取、蒸馏、治理并通过评估证明有效的经验层，才进入 Agent Memory。第一部分先建立这个行业判断，第二部分第 5 章再展开风控资产如何被 Memory 化并进入上下文。

2. 为什么跑得快：评估体系与数据飞轮

“跑得快”不是一句效率口号，也不是让模型直接替代审核员，而是把风控 case 从一次性处理变成可回放、可归因、可复用的数据生产过程。每个 case 都要留下 trace、证据快照、规则版本、Agent 建议、人工修改、申诉后验、错误归因和资产更新，这样才能把失败样本快速转成评估集、case memory、SOP、工具改进池或策略改进池。

落地判断：风控 Agent 先跑通“旁路建议 + 人工复核 + 后验回流”，再逐步进入 shadow-run、canary 和低风险半自动；价值不先押注全自动处罚，而是先证明查证提效、证据完整性、误判归因速度和版本回滚能力。

2.1 风控数据飞轮图

例子：欺诈导流举报如何进入飞轮。1. Agent 处理：拉取举报文本、图片 OCR、账号历史、设备关联、近 30 天处罚和当前规则版本，输出证据链、风险解释、缺失证据和建议动作。2. 人工复核：审核员把“建议处罚”改成“补证据观察”，并标注原因是证据不足。3. 申诉与后验：若后续申诉成立，系统回放原始证据、规则版本和 Agent 输出，确认当时高置信建议不成立。4. 错误归因：归因到“证据不足仍给高置信建议”或“OCR 漏识别关键文本”。5. 资产沉淀：补入 Regression / Boundary 样本，形成 prompt 约束、工具改进池或案例记忆候选。6. 离线评估：用新样本回放旧版和新版，验证新版会触发补证据或转人工。7. 灰度发布：先 shadow-run 观察同类 case 的补证据率、人工接受率和误伤回退，再决定是否 canary。

2.2 风控数据飞轮的最小闭环

飞轮环节 风控里怎么做 产出数据 进入哪里 为什么提速
Agent 处理 按任务 schema 输出证据链、规则引用、风险等级、置信度、不确定点和建议动作。 trace、工具调用记录、证据引用、模型输出、policy_version。 case replay、证据质量评估、模型/工具对比。 每个结果都能回放，不再靠复盘时猜模型当时为什么这么判。
人工复核 审核员只做接受、修改、驳回、补证据和转人工原因标注，不写散落的长文本反馈。 human_diff、复核标签、补证据项、驳回原因、最终处置。 评估标签、人工接受率、人审推翻率、偏差分析。 反馈变成结构化标签，能直接用于对比版本和定位问题。
申诉后验 回放原处罚时的证据、规则版本和 Agent 输出，对比申诉材料和最终复核结论。 appeal_outcome、误伤/漏放标签、规则争议、后验有效证据。 Regression Set、Boundary Set、策略复盘。 线上真实后果能回流成高价值样本，避免只用离线标注自嗨。
错误归因 把失败样本归因到模型判断、工具缺失、证据不足、规则口径、Memory 误用、Runtime 流程或评估缺口。 error_taxonomy、责任 owner、影响样本、修复建议。 评估集、Memory 候选、SOP、工具改进池、策略改进池。 每类问题都有去处，不会停留在“模型效果不好”的笼统结论。
资产沉淀 把可复用经验分别沉淀为评估样本、相似案例、反思记忆、流程模板、工具需求或策略修订。 eval case、case memory、SOP diff、tool ticket、policy diff。 版本化资产库和发布流水线。 一次错误变成长期资产，下一轮不用从零排查。
离线评估 用 Golden / Boundary / Regression / Adversarial / Fresh Sample 做业务样本回放，并按 Memory、Tool、Prompt、Retrieval、Policy、Runtime 做专项切片对比。 eval_report、分桶指标、失败样本、上线门槛、风险说明。 发布审批、灰度范围、回滚预案。 让评估成为发布前关口，而不是出事后的复盘。
灰度发布 新模型、prompt、规则、工具或 Memory 先 shadow-run，再 canary，对比旧版本指标。 版本对比、影响样本、指标变化、回滚记录。 发布决策、放量策略、回滚机制。 改动能被小流量验证，避免“凭感觉上线”。

2.3 六层评估体系

评估层 回答的问题 典型指标 不通过时怎么处理
1. 输入与证据层 证据是否取全、工具返回是否可靠、引用是否能支撑判断。 必要证据覆盖率、工具成功率、证据冲突率、证据引用正确率、OCR/视觉命中率。 补工具、补取证流程、补证据 schema，不能让模型用缺证据硬判。
2. 规则与约束层 规则版本、权限边界、citation、schema 和转人工条件是否正确执行。 policy_version 匹配率、规则引用正确率、schema 通过率、越权建议率、高风险转人工覆盖率。 修 validator、policy checker、权限配置和停止条件，禁止越权放量。
3. Agent 决策层 风险等级、建议动作、解释、不确定点和补证据建议是否准确。 precision、recall、风险等级准确率、建议动作准确率、过度自信率、解释质量。 修 prompt、模型选择、reasoning 约束、Memory/RAG 召回和 rerank。
4. 人机协同层 系统是否把合适的 case 留给人，把明确的 case 做成高质量建议。 人工接受率、人审推翻率、补证据率、升级率、handoff 准确率、审核员修改 diff。 调整置信度阈值、转人工策略、复核界面和解释模板。
5. 业务结果层 风险是否真的下降，用户权益是否被保护，策略目标是否达成。 拦截率、漏放率、误杀率、申诉成功率、复犯率、复核一致性、投诉率。 限制灰度或回滚，并进入业务归因：规则、证据、模型、人审还是分布漂移。
6. 工程运行层 线上是否稳定、可回放、可审计、可回滚，成本和延迟是否可控。 P95/P99 延迟、成本、trace 完整率、case replay 覆盖率、状态恢复成功率、回滚耗时。 修 runtime、trace schema、状态机、限流降级和版本发布流程。

2.4 评估集与反馈治理

分桶评估集

评估先按样本风险形态分桶：Golden 覆盖稳定规则，Boundary 覆盖灰区，Regression 防止重犯，Adversarial 防对抗，Fresh Sample 防分布漂移；再按 Memory、Tool、Prompt、Retrieval、Policy、Runtime 做能力专项切片。

Eval Asset

反馈先治理

资深审核员仲裁可信度高，申诉成功要先区分真误伤和规则变化，集中申诉和新账号批量反馈默认隔离，防止脏反馈带歪飞轮。

Anti Poisoning

版本可回滚

模型、prompt、memory、工具、规则和评估集版本绑定。任何改动都要能看到指标变化、影响样本、回滚路径和负责人。

Versioned Ops

评估样本怎么分桶

评估集 样本来源 覆盖什么风险 更新频率 上线门槛
Golden Set 稳定规则、资深审核员一致判定、历史高置信 case。 基础能力不能退化：规则引用、证据完整、风险等级和处置建议。 低频更新，规则大版本变化时同步调整。 新版本必须不劣于当前线上版本，否则不能放量。
Boundary Set 灰区 case、规则边界、不同业务线口径差异、需要人工讨论的样本。 防止模型把灰区说得过于确定，或越权给出高风险处置。 每周从人工复核和策略讨论中补充。 必须输出不确定点、补证据建议和转人工原因。
Regression Set 申诉成功、人工推翻、线上事故、重大误伤和漏放。 防止同类错误重复发生。 每次修复后立即补入，并绑定错误归因标签。 相关样本必须通过，失败时阻断发布或限制灰度范围。
Adversarial Set 对抗样本、规避敏感词、图文变体、团伙批量行为。 防止被绕规则、绕 OCR、绕相似案例召回。 按风险事件和策略对抗周期滚动更新。 高风险样本必须触发补证据或转人工，不能静默放过。
Fresh Sample 最近线上随机抽样、最新业务活动、新用户/新设备分布。 监控分布漂移，避免只在历史样本上好看。 每日或每周刷新，不进入长期固定分数榜。 发现明显漂移时暂停放量，并进入归因分析。

能力专项怎么评估

专项能力 评估问题 样本切片 典型指标 失败后进入哪里
Memory Eval 召回的历史经验是否真的帮助判断，而不是误引用或污染上下文。 需要相似案例、规则口径、申诉后验、历史误判经验才能判好的样本。 memory 命中收益、误引用率、人工接受率、开启/关闭 memory 对比。 memory 候选审核、TTL 调整、冲突消解、deprecation。

Runtime Eval 状态、回放、停止条件、handoff、版本绑定是否可靠。 多工具链路、长任务、需要补证据、需要转人工、需要回滚的样本。 case replay 覆盖率、状态恢复成功率、handoff 准确率、回滚耗时。 runtime 修复、trace schema、状态机、灰度和回滚流程。

第二部分 · 风控整体架构设计与方案落地

3. 四块总体架构

这套架构不是技术名词堆栈，而是一个生产系统的功能分层：底层让系统持续变好，中间层让系统跑起来，上层左侧让系统跑得好，上层右侧让系统别出错。

层级解释：中间层是工程底座，技术框架、工具协议、模型调用、状态、执行链路主要落在这里；上层两块是业务和技术的交界层，既包含风控领域知识、策略逻辑和证据规则，也包含 context building、memory retrieval、guardrail、validator 等技术机制。

4. 编排与运行时：让系统跑起来

运行时层负责把一个风控任务从进入系统到反馈回流跑通。它承载技术选型和系统执行链路：Spring AI 适合作为 Java 应用集成层，MCP 风格工具网关统一内部工具，LangGraph 的 state/checkpoint/HITL 思路可借鉴，OpenAI Agents SDK 可用于原型验证。

4.3 技术契约、数据模型与执行流程

运行时层要承接上层领域能力，关键是把 Memory、Context、约束和验证都变成 Runtime 可调用的稳定契约，而不是散落在 prompt 里的文字。建议把工具协议、Context Pack、Memory Item、Validation Result 和 Trace Event 都作为一等数据对象版本化。

技术契约 作用 核心字段 设计要点
Tool Gateway Protocol 统一规则、画像、设备、图谱、OCR、案例等内部工具。 tool_name、input_schema、output_schema、auth_scope、timeout、audit_id。 MCP-style，但要接入内部权限、审计、限流和脱敏。
Context Pack Contract 定义模型本次可以看到什么。 case_facts、evidence_items、policy_refs、similar_cases、memory_hints、action_constraints。 按槽位组织，不把所有文本拼成一个不可验证 prompt。
Memory Item Contract 定义可长期召回的经验资产。 memory_id、type、content、source_trace、confidence、owner、version、ttl、status。 必须有来源、责任人、有效期、状态机和回滚路径。
Policy Snapshot 绑定一次决策使用的规则版本。 policy_version、effective_time、business_line、clause_refs、risk_level。 所有建议必须能回放到当时规则，而不是只看当前规则。
Validation Result 记录约束校验结果和失败原因。 validator_name、passed、severity、failed_fields、repair_hint、review_required。 校验结果要能进入 trace 和评估集，不能只在日志里。
Agent Decision 定义工作台可消费的 Agent 输出。 risk_level、decision、evidence_refs、policy_refs、confidence、missing_evidence、review_points。 建议、证据、规则、置信度和待人工确认点必须结构化。
Trace Event 支撑回放、评估、归因和数据飞轮。 run_id、step_id、state_hash、tool_call、context_pack_id、validation_result、latency。 所有工具调用、Memory 命中、验证失败和人工修改都要记录。
Feedback Record 把人工复核和申诉后验转成可治理反馈。 label_source、human_decision、diff、appeal_outcome、error_taxonomy、owner。 反馈必须分可信度和原因，不能直接把人工修改当真值。

5. 上层领域能力：Memory & Context 与约束 & 验证

顶层左右两块不是纯技术层，而是业务与技术的交界层。左边不是把所有业务资产都叫 Memory，而是负责把领域知识、当前案件、工具事实和历史经验组织成可用 Context；右边负责把风控规则、权限、证据要求、输出格式和人工责任链变成可执行的约束与验证。第 5 章按“资产边界 → Memory 准入 → Context 构建 → 约束验证”的顺序展开。

Memory & Context：让系统跑得好

Memory 负责长期经验，Context 负责本次任务的工作台。权威事实走工具查询，文档和案例走检索，经过治理和评估的经验才进入 Memory；Context Builder 再把这些材料按证据、规则、相似案例、缺口和行动约束组装成可验证的输入包。

领域认知层

约束 & 验证：让系统别出错

约束不是一句 prompt，而是规则服务、权限系统、证据引用、输出 schema、风险分级、停止条件和人工复核共同组成的控制面。技术上对应 policy engine、validator、citation checker、risk gate、HITL 和 trace-bound release。

治理安全层

5.1 风控资产的 Memory 化分层

边界原则：规则库、案例库、画像、设备和图谱等原始资产默认不是 Memory，而是 source of truth、工具或检索源；只有当系统从 trace、人工反馈、复核和申诉后验中抽取出稳定经验，并经过版本、TTL、权限、citation、评估和灰度治理后，才成为 Agent Memory 层的一部分。
风控资产 默认角色 Memory 化后的形态 对应类型 边界原则
规则库 / 策略体系 / 敏感词库 Policy engine、规则服务、词典工具，是权威判断来源。 版本差异、灰区解释、误伤样例、规则冲突归因、策略口径摘要。 Semantic / Reflective Memory 只能辅助解释和召回，不能覆盖正式规则版本。
案例库 / 审核记录 / 处罚记录 / 申诉记录 Case DB、审计日志、处置流水，是证据和回放来源。 代表性案例、相似处置路径、申诉成功模式、误判和漏判经验。 Episodic / Reflective 原始记录是证据，Memory 是蒸馏后的经验，不能把单例直接泛化成规则。
审核 SOP / 复核模板 / 取证流程 流程文档和工作流配置，是执行规范来源。 常用取证顺序、任务拆解模板、转人工条件、复核摘要写法。 Procedural Memory 可以提示流程，实际执行仍受 runtime 权限、预算和停止条件控制。
用户画像 / 行为历史 / 设备信息 Feature service、画像 API、设备风控服务，是当前事实查询工具。 稳定行为模式、历史处置上下文、设备风险摘要、特征漂移观察。 Semantic / Temporal 实时事实必须回查源系统；Memory 只保存可解释、可过期、可追溯的摘要。
关系图谱 Graph query tool，是关系证据和团伙识别工具。 时序关系摘要、团伙模式、传播链经验、关系变化后的后验结论。 Graph / Temporal 图谱本体仍是工具；被治理、评估并进入召回策略的图谱经验才是 Memory。

5.2 风控 Memory 条目生命周期

与 1.4 的关系：1.4 讲的是组织级 Memory 生产能力闭环，回答“如何持续捕获、蒸馏、索引、检索、评估和治理”；5.2 讲的是单条风控 Memory 条目的准入生命周期，回答“什么样的候选经验可以上线、如何灰度、何时回滚”。因此 5.2 不是另一套 Memory 框架，而是 1.4 在风控高风险场景里的 Govern / Evaluate / Release 细化。

5.3 Context 构建与证据包

Context 不是“把材料塞进 prompt”，而是把本次任务需要的事实、证据、规则、相似案例和可用经验组装成一个可追溯、可裁剪、可验证的 Context Pack。它既要让模型看见足够信息，也要防止无关材料、过期事实和未经授权的数据进入决策。

Context 槽位 来源 关键字段 排序与预算 验证点
case_facts 当前案件、工具事实、业务系统。 case_id、subject_id、content_type、report_reason、event_time。 必须进入；事实字段不被摘要替代。 来源系统、时间戳、权限范围必须齐全。
evidence_items OCR、图片识别、设备、画像、图谱、历史处罚。 evidence_id、source_tool、value、confidence、citation。 按证据强度、时效、场景相关性排序。 无 citation 的证据不能支撑结论。
policy_refs 规则库、策略体系、敏感词库。 policy_id、policy_version、clause_id、effective_time。 当前有效规则优先，历史规则只用于回放。 结论必须绑定规则版本，不能引用过期规则。
similar_cases 案例库、审核记录、申诉记录。 case_ref、similarity_score、decision、appeal_outcome。 top-k + rerank，保留反例和边界样本。 相似案例只能辅助一致性，不能直接替代规则判断。
memory_hints 已准入的 semantic / episodic / procedural / reflective memory。 memory_id、type、confidence、valid_until、owner。 按任务收益、置信度、过期时间和冲突关系裁剪。 必须可追溯、可过期、可回滚。
action_constraints 权限、风险分级、HITL 策略、停止条件。 allowed_actions、forbidden_actions、review_required、stop_reasons。 必须进入；不受 token budget 影响。 高风险动作必须转人工或二次确认。

5.4 约束与验证控制面

约束与验证要从“模型输出后检查一下”前移为全链路控制面：任务进入时校验权限和场景，工具调用前校验授权和预算，生成前校验证据充分性，生成后校验 schema、citation、规则版本和高风险动作，发布前决定自动通过、补证据、转人工或停止。

控制点 验证什么 实现方式 失败动作 沉淀数据
权限与数据边界 当前任务是否允许访问画像、设备、图谱、处罚记录。 subject_scope、operator_role、purpose binding、字段级脱敏。 拒绝工具调用或返回脱敏摘要。 permission_check、redaction_trace。
规则版本 是否使用当前有效规则，是否存在业务线差异。 policy_snapshot、clause validator、effective_time check。 停止输出正式建议，转为规则冲突说明。 policy_version、clause_hits、policy_diff。
证据引用 每个判断是否有对应证据，证据是否来自可信工具。 citation checker、evidence graph、source whitelist。 降级为“证据不足”，触发补证据或人工复核。 unsupported_claim、missing_evidence。
输出结构 是否符合工作台可消费 schema。 JSON schema、枚举校验、字段完整性校验。 自动修复一次；仍失败则停止并记录。 schema_error、repair_attempt。
风险动作 处罚、封禁、降权、拒诉等动作是否允许自动化。 risk gate、action policy、confidence threshold、HITL rule。 高风险动作 100% 转人工，低风险可灰度半自动。 action_gate、review_required_reason。
一致性与冲突 Memory、相似案例、规则和工具事实是否互相冲突。 conflict detector、version bundle、反例召回。 标注冲突点并转人工，不允许强行下结论。 conflict_set、human_resolution。
停止条件 预算耗尽、工具失败、证据不足、循环调用是否出现。 stop controller、tool timeout、max_steps、degrade policy。 停止、降级、补证据或转人工。 stop_reason、degrade_path。

举报受理辅助 举报内容、被举报对象、内容 OCR/视觉标签、用户画像、设备信息、历史处罚、规则版本。 自动拉取证据、匹配规则口径、召回相似案例，输出风险等级、证据链、缺失证据和建议处置。 结构化 evidence bundle、case trace、人工修改 diff、相似案例命中记录。 必要证据覆盖率、引用正确率、人工接受率、平均查证时长、补证据率。

6.2 举报受理辅助流程图

用例目标是把一个举报工单变成可复核的证据包和处置建议。V1 不直接处罚，而是旁路生成建议，人工确认后才进入处置链路。

6.3 申诉复盘闭环流程图

用例目标不是重新“再判一次”，而是回放原始决策、对比新增材料和后验结果，把误判归因到规则、证据、工具、Memory、Runtime 或人工流程，并反哺评估集和资产治理。

6.4 图片/多模态审核流程图

用例目标是把图片识别结果转成可引用证据，而不是让视觉模型单独决定处罚。多模态结果必须和规则、上下文、相似案例、人工复核一起进入证据链。

6.5 Trace 写入与资产沉淀

三类业务场景都必须写同一套 trace，不然无法回放、评估和归因。Trace 不是普通日志，而是把一次 Agent 决策拆成可重放的数据契约。

Trace 事件 必传字段 写入位置 后续用途
run_start run_id、scenario、case_id、subject_id、operator_role、auth_scope、policy_version_request、input_hash。 Trace Store / Run Index。 串起一次完整运行，支持权限审计和 case replay。
tool_call step_id、tool_name、tool_version、input_ref、source_system、output_hash、latency_ms、error_code、redaction_rule。 Trace Store / Tool Audit Log。 定位工具失败、数据过期、字段脱敏和延迟问题。
context_pack_created context_pack_id、case_facts、evidence_ids、policy_refs、similar_case_refs、memory_ids、action_constraints、dropped_items。 Context Pack Store / Trace Store。 回放模型当时到底看见了什么，支持新旧版本 A/B 回放。
memory_retrieved memory_id、memory_type、owner、version、ttl、source_trace、retrieval_query、score、used_in_output。 Memory Store / Trace Store。 评估 Memory 是否真的带来收益，发现误用后回滚或下线。
validation_result validator_name、passed、severity、failed_fields、unsupported_claims、review_required、repair_hint、action_gate。 Validation Log / Trace Store。 判断系统为什么停止、转人工或不允许自动动作。
agent_decision decision_id、risk_level、suggested_action、evidence_refs、policy_refs、confidence、missing_evidence、review_points。 Decision Store / 审核工作台。 给人工复核使用，也作为后续申诉和质检的原始决策。
human_review reviewer_id、human_decision、human_diff、reject_reason、added_evidence、final_action、review_time。 审核工作台 / Feedback Store。 形成 ground truth 候选，衡量人工接受率和模型改动方向。
feedback_ingested label_source、feedback_weight、appeal_outcome、error_taxonomy、repair_path、owner、eval_bucket、candidate_memory_id。 Feedback Store / Eval Store / Memory Candidate Store。 进入 Regression、Boundary、Memory Eval、Tool Eval，或生成策略/工具改进项。
