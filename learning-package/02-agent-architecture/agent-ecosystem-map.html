0.2为什么"快手要自研 Agent"？给非技术读者的一段话

非技术版结论
快手做的是风控——决定要不要封一个账号、限流一段视频、把内容下架。每一步都是高风险动作：封错了平台被告，封漏了内容出事。任何"AI Agent 自动判罚"都必须满足 4 个硬要求：
① 每一步可解释——监管来查"为什么封这个号"要给出依据
② 每一步可追回——封错了 5 分钟内能撤销
③ 每一步可暂停——发现 Agent 跑偏立刻拉闸
④ 关键决定可人工签发——大 V、政治、涉黄涉政这种必须有人复核
市面上现成的 Agent 产品（Cursor / Claude Code / OpenHands 等）都是给开发者写代码用的，没有这套治理链路。买不到现成的，所以要自研。自研的核心不是"再造一个 LLM"，而是把这套 4 条审计治理链路真正建出来。

6.4三个争议项目的判定（codex 多轮推演）

① LangChain4j 究竟是"状态图范式"还是"Java LLM 应用层"？

判定：Framework Glue（Java LLM 应用层），不是 State Graph。

官方定位（docs.langchain4j.dev）是"简化 Java 应用集成 LLM"：核心抽象 ChatLanguageModel / AiServices / @Tool / ChatMemory / EmbeddingStore。它可以做 agentic

workflowworkflow = "工作流"，一连串有先后顺序、可能要等很久的步骤。比如"订单确认 → 通知仓库 → 配货 → 发货 → 跟踪"。durable workflow（持久工作流）能在任意一步挂掉重启接着跑。
，但底层并非显式状态图——状态由用户在 Java 代码里维护。要在 JVM 上获得 LangGraph 那种"画状态图"的体验，需要用 LangGraph4j（独立项目，langgraph4j/langgraph4j），它确实是 State Graph 范式。

不要因为名字像就把 LangChain4j 和 LangGraph4j 当兄弟——它们是不同范式 + 不同维护团队。

② Embabel + GOAP：算独立范式还是规划范式的子集？

判定：放在"Planner / Goal-Driven"族里，不单独升顶层。

GOAP（Goal-Oriented Action Planning）本质是 Symbolic Planner 的一个子型，强调"从目标反推动作序列"。Embabel（embabel/embabel-agent）由 Spring 创始人 Rod Johnson 2025 创建，把 GOAP 思路落到 JVM Agent 上。把 GOAP 当独立顶层会过细化（同理 PDDL、HTN 都是规划范式子型）；归到"Planner / Goal-Driven"族里更符合学术界对规划范式的分类。

风控启示：不一定上 Embabel，但"调查—判罚—复核"的动作目录 + 前置/后置条件思想，可以照搬到自研的 Agent action catalog。

③ LangChain（不是 LangGraph）应该作为独立"Chain/LCEL"范式吗？

判定：应该。补回独立一族。

用户问"为什么没有 LangChain，只有 LangGraph"——这就是第一版的缺漏。LangChain 自己（核心是 LCEL = LangChain Expression Language）代表"声明式管道"范式，和 State Graph 是两件事。LCEL 适合 RAG 这种"prompt → model → parser"线性场景；遇到分支、循环、人审就该升级到 LangGraph。把 LangChain 写成"已经被 LangGraph 取代"是夸张——LCEL 至今仍是 LangChain 生态的主流入口，只是 agentic 部分让位了。来源：LangChain 官方关于 LCEL 的博客。
