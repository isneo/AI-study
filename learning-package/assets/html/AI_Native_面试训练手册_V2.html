Chapter One
i · 背景

这件事，从哪个痛里长出来

方案显得"虚浮"的根本原因，是没有把真实的痛说透。 传统风控的问题不是"做得不够好"，而是这条路本身走到了头—— 规则体系正在爆炸式增长，证据链正在变得无法追溯，黑产对抗节奏正在压倒所有人工迭代速度。

这三道裂缝不是"传统风控做得不够好"，而是传统范式的结构性极限—— 越往后规则越多、证据越散、对抗越快，靠加人/加规则的线性手段已经无法对齐增长。 这是讲清楚"为什么必须做 AI Native"的起点。
业务背景口播 · 60 秒讲透痛点
在讲我们做了什么之前，/ 先讲为什么必须做。// 传统风控走到今天，/ 有三道结构性裂缝。// 第一道·规则爆炸——/ 策略从几十条长到几千条，/ 规则之间互相冲突、没人敢删、上线即遗忘。/ 这不是规模问题，是规则范式本身的极限。// 第二道·证据链碎片——/ 一个 case 的证据散落在账号、设备、行为、关联、历史、外部六类系统里，/ 人工拼一遍要半小时。/ 这是审核员经验难以规模化的根因。// 第三道·对抗节奏——/ 黑产是天级变种，/ 我们的迭代是月级。/ 节奏差一个数量级。// 这三道裂缝同时发生，/ 靠加人、加规则、加模型都补不上。/ 这才是AI Native 化的真正起点。
规则爆炸证据碎片对抗节奏必须做的根因
核心亮点
  • "结构性裂缝"而非"做得不够好"——把痛点定位成范式极限，而不是执行差距。这是讲"为什么必须重构"而不是"为什么要优化"的关键差别。
  • 三道裂缝是行业内行话——规则爆炸、证据链、对抗节奏，每一道都能让懂行的面试官点头。
  • "节奏差一个数量级"——天级 vs 月级的对比，用数量级的语言而不是百分比，更有冲击力。
  • "靠加人、加规则、加模型都补不上"——主动排除三种常见的"伪方案"，反向凸显 AI Native 的必要性。
常见误区
  • "传统风控效率不高，所以我们用 AI 提升效率。" 这是 AI Assisted 话术。改成"传统范式有结构性裂缝，必须换范式"。
  • 把痛点讲成"业务量大""人手不够"这类资源问题。 资源问题靠加人能解，结构性问题靠换范式才能解——前者是优化叙事，后者是重构叙事。
  • "规则太多了我们想简化它。" 不是"规则太多"，而是"规则范式本身到了极限"——这是认知层面的差别。
Chapter Two
ii · 演进

从机审到 AI Native · 四段路径

每一段路径都是上一段走到头之后的必然。这条演进线很重要—— 它把"AI Native"从一个时髦词变成历史的合理结果， 让方案不再像凭空冒出的创新，而是从问题里长出来的答案。

这张图的关键不是"四段名字"，而是每段为什么必然走向下一段—— 规则爆炸推着走向智能审核、规则难产推着走向 AI 驱动、协作断裂推着走向 AI Native。 这条因果链让 AI Native 不再像跳跃，而像合理的终点。
演进路径口播 · 90 秒讲完四段史
风控审核走过四段路径。/ 每一段都是上一段走到头的必然。// 第一段·机审——/ 人写规则、机器执行。/ 走到头是规则爆炸。// 第二段·智能审核——/ 引入 ML 模型给人当参谋。/ 走到头是规则难产——/ 模型只能识别旧风险，新规则还是人来写。// 第三段·AI First——/ 让 AI 驱动规则制定，/ 从 bad case 里自动归纳新规则。/ 走到头是协作断裂——/ AI 能写规则，但人不知道怎么和 AI 协作，/ 流程还是旧的。// 第四段·AI Native——/ 重新设计整条业务链路，/ 让 Agent 驱动识别、决策、流转、沉淀、进化，/ 人变成 Agent 的协作者与监督者。// 这四段的逻辑链是：/ AI 辅助人审 → AI 驱动规则 → AI 驱动人机协同 → AI 原生闭环。/ 每一步都在把 AI 推得更深。
四段必然性逻辑链条不是创新是终点
核心亮点
  • "每一段都是上一段走到头的必然"——这句话把演进讲成因果链而不是时间线，立刻区别于"AI 越来越厉害"这种浅层叙事。
  • 把"走到头"具体化为三个症状——规则爆炸 / 规则难产 / 协作断裂。每个症状都能让面试官接得上。
  • "我们站在哪里"——主动定位行业大多数在 ERA 02→03 的过渡期，我们要直接做到 ERA 04。既体现野心，也体现自知。
  • 逻辑链一句话总结——"AI 辅助人审 → AI 驱动规则 → AI 驱动人机协同 → AI 原生闭环"，金句版本可单独甩出。
常见误区
  • 直接说"我们做了 AI Native 项目"，不交代演进背景。 先把四段路径铺出来，让 AI Native 成为合理终点而不是孤立跳跃。
  • 把演进讲成"技术升级史"（规则 → ML → DL → LLM）。 演进的核心不是技术，是"AI 和人的关系"的变化——参谋 → 驱动者 → 主导者。
  • 说自己已经在 ERA 04，把状态吹得太满。 承认部分主线还在 ERA 03 向 04 过渡，整体方向是 ERA 04——更可信。
  • 把"AI First"和"AI Native"混为一谈。 AI First 是"AI 进入决策路径"，AI Native 是"AI 重塑业务链路"——前者是工具升级，后者是范式重构。
Chapter Three
iii · 差异

AI Native 与 AI Assisted 的分水岭

前两章铺完背景与演进，这一章要让面试官真正"听懂"AI Native 在说什么。 99% 的同行做的是 AI Assisted——把 AI 塞进旧流程当助手。 AI Native 的本质是生产关系的变化：谁发起决策、谁编排流程、谁审核谁。

这张图建议练到能 60 秒徒手画出来。AI Native 不是"AI 做得更多"， 而是角色互换——人从执行者变成审核者，Agent 从工具变成发起者。
差异化口播 · 30 秒杀手锏
传统做法是：人写规则 → 机器执行 → 出问题 → 人改规则。/ 我们做的是：让规则的生产、合并、迭代本身都由 AI 驱动。// 审核员从"写规则的人"变成"审核 AI 生产的规则的人"——/ 这才是 AI Native，/ 而不是 AI Assisted。// 换个角度说，/ 整个业务链路里，人的角色从执行者变成协作者和监督者，/ 这是生产关系层面的变化。
金句一·生产关系金句二·审核 AI追问时甩出
核心亮点
  • "生产关系层面的变化"——这句话能立刻让面试官知道你不是在做技术堆砌，而是在做范式转移。
  • 角色互换的具象化——"从写规则的人变成审核规则的人"，用一个具体角色变化代替抽象概念。
  • 反例锚定——主动提"AI Assisted"作为对照，让你的定位自动浮现。
常见误区
  • "我们用了 LLM 帮审核员提高效率，让他们少看一些 case。" 这就是 AI Assisted 话术。改成"Agent 主导审核决策，人审核 Agent 的决策"。
  • 把 AI Native 等同于"全自动化"或"无人化"。 AI Native 不是去掉人，而是重排人的角色——人不消失，但人的角色升级。
  • 只用"识别更准、效率更高"这类增量改进描述差异。 用"谁发起、谁审核、谁迭代"这种结构性变化描述差异。
Chapter Four
iv · 技选

为什么是Trigger-based Reasoning，而不是 Workflow 编排

这一章是技术选型的"分水岭"——风控审核不是固定流程，而是自主寻证。 策略理解、证据收集、推理判定这些任务，本质上模拟的是人类审核员的动态决策过程。 Workflow 编排适合"开会请假"那样路径清晰的任务，风控不是。

风控审核员的真实工作流是"看到一个信号 → 决定查什么 → 查到结果 → 决定下一步查什么"—— 这是触发器驱动的推理，不是预设的流程。Trigger-based Reasoning 是对人类审核员认知过程的工程化复制。
技选口播 · 为什么选 Trigger-based
很多人问我们为什么不用 Workflow 编排，/ 毕竟 Workflow 工具链更成熟、更可控。// 我们想过这个问题，/ 最后选了触发器驱动的推理——Trigger-based Reasoning。/ 原因有三个。// 第一·任务本质不允许。/ 风控审核不是固定流程。/ 看到一条内容、一个账号，/ 要查什么、不查什么，取决于看到了什么。/ 这是审核员每天在做的事——/ 看到敏感词查上下文、看到陌生设备查关联、看到金额异常查历史。// Workflow 编排把这种动态过程固化成步骤，/ 要么漏查、要么过度查询。// 第二·黑产对抗需要。/ 固定流程一旦被黑产摸透，/ 就能针对性绕过。/ 触发器驱动的推理路径每次都不一样，/ 天然有对抗性。// 第三·模拟人类认知。/ 这是最本质的——/ 我们是在把资深审核员的"经验"工程化，/ 而审核员的经验本身就是触发器结构：/ 看到 X 就去想 Y，看到 Y 就去查 Z。/ 这才能真正复用专家经验。
三个理由任务本质对抗安全经验工程化
核心亮点
  • "看到 X 就去想 Y"——用一句话概括触发器机制，比讲技术架构更直观。
  • 对抗安全维度——主动提"固定流程被摸透就能绕过"，体现你不只看效率还看安全。
  • "经验工程化"——把技术选型上升到"专家经验复用"的业务价值，避免被定位为纯技术讨论。
  • 承认 Workflow 也有优点——开场说"Workflow 工具链更成熟"，反映出你做过真实权衡，不是 ideologically driven。
常见误区
  • "Workflow 编排太老了，我们用 Agent 才先进。" 技术选型不讲新旧，讲任务适配性。请假审批用 Workflow 反而合适。
  • 把 Trigger-based 和 ReAct 混为一谈。 ReAct 是单 Agent 的推理-行动循环；Trigger-based 是多触发器并行 + 证据回流再触发的领域设计。
  • 把"触发器"讲成简单的 if-else 规则。 触发器是"特征→调研动作"的语义映射，由 LLM 在上下文里动态决定，不是硬编码。
  • 所有任务都用 Trigger-based，不区分场景。 明确分层：低风险高频走规则/Workflow，高风险复杂 case 走 Trigger-based reasoning。
Chapter Five
v · 母图

五条主线 × 三大支柱 · 能徒手画的母图

这张图是整场面试的母图。任何被追问的环节，都可以回到这张图定位。 练到闭眼能画的程度——面试时如果有白板，主动起身画一遍，效果胜过所有 PPT。

母图——你的"白板武器"。讲解顺序：先指上层五个卡片说"业务链路视角"， 再指下层三个柱子说"底层能力视角"，最后指底部虚线框说"主线松耦合、支柱强复用"。 整套讲解 60 秒以内。
母图讲解口播 · 60 秒
我把项目分成三个视角看。// 业务链路视角——/ 五条改造主线，覆盖风险事件的完整生命周期：/ 从识别、决策、流转、沉淀到进化。/ 每条主线都是Agent 驱动的，/ 而不是给传统流程加 AI。// 底层能力视角——/ 三大支柱是所有主线共用的基础能力：/ 知识管理负责领域知识的生产和演化，/ Harness 负责给 Agent 划定决策边界，/ 行为上下文负责为 one-off 调用提供用户档案。/ 这三大支柱是最重要的架构决策——避免每条主线各造轮子。// 设计原则——/ 主线之间松耦合、独立可验证；/ 支柱层强复用、共享基础设施。
白板手绘版母图记忆点展开钩子
核心亮点
  • 三视角分解——业务链路 / 底层能力 / 设计原则。这是操盘者级别的架构表达方式。
  • "避免各造轮子"——能立刻打动技术 VP，体现工程复用与横向治理的思考。
  • 风控特有的"行为上下文"——区别于"对话历史"，是风控 Agent 与通用 Agent 的本质差异。
常见误区
  • 把架构讲成"流水线四层"：输入 → 规划 → 执行 → 输出。 这是通用 Agent 模板，没体现风控特色。改用"五主线 + 三支柱"结构。
  • 把"行为上下文"等同于"对话历史"或"Memory"。 风控 Agent 是 one-off 调用，没有对话。上下文是用户的数据档案。
  • 不解释"Harness"，让面试官以为是沙箱。 主动定义：Harness 不是沙箱，是 Agent 在风控领域"守规矩"的约束机制。

飞轮口播 · 项目节奏陈述
五条主线如果同时推进，会平摊资源、什么都做不深。/ 我的节奏是"先建飞轮、再做规模"。// 第一阶段·MVP——/ 先打透「审核决策 + 知识沉淀」这两条最关键的主线，/ 把闭环跑通、效果验证、专家信任建立。// MVP 已经能交出几组关键数据：/ 人审覆盖率下降 XX%、/ 单 case 处理时长提升 XX 倍、/ 新型风险发现从月级缩短到天级、/ 策略库新增中XX% 由 Agent 提案。// 第二阶段——/ 扩展「风险识别 + 案件流转」，/ 把上层业务能力全面 Agent 化，现在在灰度。// 第三阶段——/ 攻坚「自我进化」，/ 让系统从"人驱动"走向"AI 驱动"。// 这套节奏的本质是——/ 用最少的资源先证明飞轮能转，/ 再用飞轮自身的产出去支撑后续主线。
三段式节奏MVP 量化资源管理叙事
核心亮点
  • "先建飞轮、再做规模"——这句话能立刻让面试官认出你的项目操盘心法。
  • 四个量级占位指标——人审下降、时长提升、发现速度、自动占比——四个角度刚好覆盖效率、效果、对抗、自治四个维度。
  • 诚实分级——已验证 / 灰度中 / 攻坚中。区别于浮夸地"全都做完了"。
  • "新型风险发现从月级到天级"——直接呼应第一章"对抗节奏裂缝"，闭环讲述的高级感。
常见误区
  • 用"功能清单"汇报进度：我们做了 A、B、C、D、E、F…… 改用"飞轮验证度"：哪个环节闭环了，哪个还没闭环，闭环依靠什么。
  • 量化数据虚高或编造，被追问"怎么计算的"答不上来。 数据要能讲清楚基线是什么、口径是什么、抽样多大——讲不清楚的数据宁可不讲。
  • 把所有主线说成"已完成"，把所有数据说得很美。 主动暴露"进化主线还在攻坚"，反而更可信。
  • 只讲"做了什么"，不讲"为什么按这个顺序做"。 操盘者的核心价值在顺序设计上——讲清楚为什么先 A 后 B。
Chapter Seven
vii · 评测

Agent 怎么评？面试官最在意的三个维度

会议纪要里特别提到：面试官会深入探讨评测方法。这是技术面的"杀手锏问题"—— 很多人能讲清楚 Agent 怎么搭，但讲不清楚 Agent 怎么评。这一章把三类评测说透： 性能、稳定性、效果。

这张图建议背成肌肉记忆——被问到"你怎么评测 Agent 的"时，第一句话就是 "我们从性能、稳定性、效果三个维度评测"，再分别展开。 先给框架，再给细节，是技术面的稳赢节奏。
评测口播 · 120 秒讲透三维度
Agent 评测我们分三个维度。// 第一·性能评测。/ 重点看首 Token 延迟、端到端响应时长、单 case Token 消耗。/ 风控场景对延迟敏感——/ 内容审核要求秒级，/ 这直接决定 Agent 能不能上线。// 第二·稳定性评测。/ 这是被很多人忽略的维度。/ 重点看同输入输出一致性、长尾鲁棒性、对抗 Prompt 抵抗率。/ 风控不能有"看心情判定"——/ 同一个 case 跑十次，结果必须高度一致。// 第三·效果评测——这是最难也最重要的。/ 我们用两种方法// 方法 A·黄金集。/ 请资深审核员标注一批高质量样本作为静态参照，/ 覆盖典型 case、长尾 case、对抗样本。/ 每次 Agent 迭代都跑一遍，/ 看召回率、误杀率、F1 是否进步。// 方法 B·教师模型。/ 用更强的模型当"教师"，/ 对线上 case 给参考判定，/ 和我们 Agent 的判定做对比。/ 这种方法可以规模化、不依赖人工标注，/ 但教师模型本身的偏见会传导进来。// 实际系统里两种方法互补使用——/ 黄金集作主评测，教师模型做日常监控。
三维框架两种方法互补使用
核心亮点
  • 先给三维框架——开口就把"三个维度"摆出来，立刻显得评测体系是有结构的，不是临时拼凑的。
  • 主动提"稳定性"维度——很多人只讲性能和效果，提稳定性能立刻让面试官认出你做过真正上线的系统。
  • "看心情判定"——用日常语言描述技术问题，比"输出一致性"更有画面感。
  • 两种效果评测的取舍——黄金集 vs 教师模型，主动暴露教师模型的偏见传导问题，体现成熟度。
  • "互补使用"——不二选一，主作监控副作监控，体现工程思维。
常见误区
  • "我们用准确率评测。" 风控场景必须分召回和误杀——准确率会把两者糊在一起，掩盖问题。
  • 只讲效果评测，不讲性能和稳定性。 三个维度都讲，证明你做过真正上线的 Agent，不是 demo。
  • "我们用 GPT-4 当教师模型对比。" 主动提教师模型的偏见传导问题——教师模型不是上帝，它的偏见会被你的 Agent 学走。
  • 把"评测集"讲成静态的——做一次就用很久。 评测集要版本化——黑产打法变了、业务场景变了，评测集也要跟着更新，否则失去意义。
  • 忽略"置信度校准"。 置信度校准 = Agent 说"90% 确定"时实际准确率是不是 90%。这是分层决策的基础，不能省。
Chapter Eight
viii · 操盘

操盘者的权衡与取舍

到这一章，已经讲清楚了"是什么、为什么不一样、怎么搭、怎么评"。 操盘者真正的护城河，是判断力——在两个都对的方案之间选一个， 在两个都错的现实里走一条。

三把尺，三个权衡。这一节不追求"全景画面"， 而是用具体的判断细节——尤其是那些"反 AI 神话"的判断——证明你不是被技术冲昏头的工程师。
三把尺口播 · 操盘者金句库
作为操盘者，我有三把判断尺。// 第一把尺·自治度与风险敏感度。/ 风控领域有一句反直觉的话：/ "越靠近自治，越需要保守。"/ 低风险场景可以全自治，/ 高风险场景必须 Agent 建议 + 人审兜底，/ 这是 Harness 设计的核心原则。// 第二把尺·自我进化与兜底。/ 自我进化主线我们专门留了一道不能完全闭环的设计——/ Agent 可以监控衰减、可以生成候选迭代，/ 但人审委员会是必经环节。/ 这不是技术能力问题，/ 这是合规和治理的问题。// 第三把尺·整体规划与分主线突破。/ 摊子要大、整体设计要完整，/ 但落地要尖——先打透两条主线，再扩展。/ 这是大项目能稳的关键。
越自治越保守进化不能闭环摊子大但落地尖
核心亮点
  • "反 AI 神话"的判断——主动说"自我进化不能完全闭环"，能立刻让面试官认出你是有判断力的操盘者，不是被技术冲昏头的人。
  • 合规与技术分离——"这不是技术能力问题，是合规和治理问题"——风控领域专属的成熟度表达。
  • "摊子大但落地尖"——这句话同时回应了"格局虚高"和"做得不深"两种质疑。
  • 主动暴露权衡而非全是优点——成熟的操盘者会主动讲取舍。
常见误区
  • "Agent 自治度越高越好，我们做到了 X% 自动化。" 风控领域反过来——"越自治越保守"。盲目追求自治率是不懂行业。
  • "我们的系统已经实现了完全自动迭代，不需要人介入。" 主动留人审环节，反而是成熟的设计。完全闭环 = 失控风险。
  • 被追问技术细节时硬撑，假装每个模块都自己写的。 承认"具体代码细节是 XX 同学主导，我从架构设计层面理解是 ABC"，反而显诚信。
  • 把"反思"讲成"做得不够好的地方"，自我贬低。 把反思讲成"已经迭代过的判断"——曾经选过 A，发现问题后改成了 B。
Appendix
ix · 问答弹药库

面试官的追问与你的回响

16 题分布在 8 章之外的"追问区"——前 4 题对应业务背景与演进、中 8 题对应方案与技术、后 4 题对应评测与操盘。 标记 陷阱 的题目考视野和成熟度，难 的考深度。

陷阱
Q · 01
传统风控不是有效率不高吗，为什么非要重构？加点 AI 提效不行吗？
考察意图：识别"重构必要性"的能力
传统风控不是效率问题，是结构性裂缝。三道—— 规则爆炸（几千条规则互相冲突没人敢删）、证据链碎片（六类系统散落、人工拼接半小时）、对抗节奏（黑产天级 vs 我们月级，差一个数量级）。 这三道都是范式极限，靠加人加规则加模型都补不上。所以我们做 AI Native——换范式，不是优化范式。
"换范式不是优化范式" 是开局表态
基础
Q · 02
你说的四段演进路径，行业大多数在哪一段？
考察意图：对行业现状的判断
行业大多数在ERA 02 → 03 的过渡期——已经在用 ML 模型给人当参谋，正在尝试让 LLM 进入决策路径。 少数头部团队进入 ERA 03 早期——做了"AI 驱动规则制定"，但仍然没解决人机协作问题。 我们的目标是直接做 ERA 04，用决策与沉淀两条主线先验证范式，再扩到其他主线。
主动定位行业位置，体现判断力
基础
Q · 03
"AI First"和"AI Native"区别到底是什么？
考察意图：概念辨析能力
AI First 是工具升级——AI 进入决策路径，但流程还是旧的，人还在做编排和监督。 AI Native 是范式重构——业务链路按 Agent 的能力重新设计，人变成 Agent 的协作者与监督者。 一句话总结——AI First 是"把 AI 加进来"，AI Native 是"围着 AI 重新建"。
"加进来 vs 重新建" 一句话辨析
陷阱
Q · 04
你说改造整个领域，会不会摊子铺得太大、什么都做不深？
考察意图：戳"格局虚高"
用"整体规划 + 分主线突破"的方式管摊子。全景设计必须有——五条主线之间强耦合，没有整体设计就会各自为战。 但落地绝不平铺，按"先建飞轮、再做规模"先打透「决策 + 沉淀」两条，跑通验证再扩。 这和单做一个模块的本质区别在于——架构、Harness、上下文层一开始就为五条主线设计，避免后期重构。
"整体规划 + 分主线突破" 是大项目操盘标签
基础
Q · 05
为什么选 Trigger-based Reasoning，不用 Workflow 编排？
考察意图：技术选型判断
三个理由。一·任务本质不允许——风控审核不是固定流程，"看到 X 就去想 Y"是审核员每天在做的事，Workflow 把动态过程固化，要么漏查要么过度查询。 二·对抗安全——固定流程被黑产摸透就能绕过，触发器驱动的推理路径每次不同，天然有对抗性。 三·经验工程化——审核员的经验本身就是触发器结构，触发器驱动才能真正复用专家经验。 Workflow 不是不好，是不适合风控审核——请假审批用它合适。
"任务本质 + 对抗 + 经验复用" 三层论证
难
Q · 06
触发器机制和 ReAct 有什么本质区别？
考察意图：Agent 架构理解深度
ReAct 是单 Agent 的串行循环——Thought → Action → Observation → Thought……一条推理链。 Trigger-based 是领域级架构——一个推理引擎绑定一组语义触发器，每个触发器对应一个调研动作。 关键差别三点——一·触发器是并行的，可以同时触发多个调研；二·触发器有领域语义（比如"账号特征→关联图谱"），不是模型自由发挥； 三·证据回流再触发，新证据可以激活新触发器，形成多轮自主寻证。可以理解为——ReAct 是"通用骨架"，Trigger-based 是"领域级特化"。
"通用骨架 vs 领域特化" 收尾
基础
Q · 07
这五条主线为什么是这五条？怎么拆出来的？
考察意图：拆解逻辑
按风控审核完整生命周期拆。任何风险事件都会经历——被发现（识别）→ 被判定（决策）→ 被处置（流转）→ 被沉淀（知识）→ 被进化（迭代），缺一不可。 原则三条——每条主线独立价值闭环、每条对应真实业务痛点、主线之间能复用底层能力，所以抽出三大支柱作为公共基础。
"生命周期"是体系化思维的锚点
基础
Q · 08
你的 Agent 和直接用 GPT-4 调 API 有什么本质区别？
考察意图：对 Agent 的理解深度
三个本质区别。一·自主规划——Agent 自己决定调研什么、调用哪些工具，不是被动应答。 二·工具调用闭环——能查黑词库、关联图谱、外部信息，证据强制引用。 三·领域 Harness——什么能做、什么必须人审，由我们写死的边界控制。 直接调 API 是"问答机"，Agent 是"会做事的同事"。
用 "问答机 vs 做事的同事" 比喻收口
难
Q · 09
Agent 出错了怎么兜底？误杀一个用户的代价怎么承受？
考察意图：风险意识与工程保障
四层兜底。一·分层决策——按风险等级走不同自治度，高风险必经人审。 二·置信度门限——Agent 自己判置信不足时主动转人审。 三·证据强制引用——每个判定必须挂靠工具返回结果，无证据则拒判。 四·灰度 + 回滚 + 影子模式——新策略先并行运行，效果验证后再切流。 设计原则——宁可漏放，不能误杀。
"宁可漏放，不能误杀" 是风控领域专业表态
难
Q · 10
幻觉怎么控？黑产用对抗性 Prompt 怎么防？
考察意图：安全工程能力
幻觉控制——证据强制引用（无证据则拒判）+ 关键决策双模型交叉验证 + 置信度校准。 对抗 Prompt——输入侧内容隔离（用户内容用 XML 包裹 + role 分离，让模型清楚这是被审核内容不是指令）、 输出侧结构化校验（必须符合预设 Schema 才接受）、 以及越狱检测前置链路（专门一道 Agent 识别可疑指令）。
XML 包裹 + 越狱检测前置 是工程深度信号
基础
Q · 11
为什么不用传统机器学习模型？
考察意图：技术选型判断
不是不用，是分工不同。传统模型在高频、稳定、明确场景仍最强（比如成熟黑词识别）。 Agent 的不可替代价值在低频、新型、长尾——冷启动场景、可解释要求、新型风险快速适配。 实际系统两者共存互补——规则引擎 + 传统模型预筛，Agent 处理复杂 case。
不贬低传统方案，体现工程成熟度
难
Q · 12
成本怎么控？LLM 调用 + 多 Agent 链路成本很高吧？
考察意图：工程成本意识
四个手段。一·模型分层——轻量模型预筛 80% 简单 case，大模型只处理 20% 复杂 case。 二·相似 case 缓存复用——历史判定结果做语义检索，命中则复用。 三·并行工具调用——独立工具不串行等待。 四·Token 预算控制——单 case 硬上限，超过则降级或转人审。 整体成本比纯人审还便宜——关键是不要把 Agent 当万能工具。
"不要把 Agent 当万能工具" 是反 AI 神话的表态
难
Q · 13
你们怎么评测 Agent？指标体系是什么？
考察意图：评测体系完整度
三个维度。一·性能——首 Token 延迟、端到端响应、Token 消耗、工具调用并发度。 二·稳定性——同输入输出一致性、长尾鲁棒性、对抗 Prompt 抵抗率、错误兜底触发率。 三·效果——召回率、误杀率、F1、置信度校准、与人审一致率。 效果评测两种方法互补——黄金集做主评测（专家标注、覆盖典型+长尾+对抗），教师模型做日常监控（规模化但要警惕偏见传导）。
三维框架 + 两种方法 = 完整评测答案
难
Q · 14
用大模型当教师模型评测，怎么保证教师本身是对的？
考察意图：评测方法论的深度反思
这是个非常好的问题，我们直接面对过。教师模型不是上帝，它有偏见传导风险——教师模型错误的偏好会被被评测的 Agent 学走。 所以我们的做法是——一·教师模型只用在有客观标准的子任务（语义一致性、Schema 合规、证据完备性），不用在主观判定上。 二·教师模型的判定要定期用黄金集校准，发现偏移就调整 Prompt 或换模型。 三·关键决策不让教师模型最终拍板——它只做"批量预筛 + 异常告警"，最终判定回到人审或黄金集。
主动暴露教师模型的局限，体现成熟度
陷阱
Q · 15
你个人主要负责哪条主线？其他主线了解多少？
考察意图：区分操盘者 vs 执行者
我的角色是整体操盘者。深度参与的是「审核决策 + 知识沉淀」，包括架构设计、技术选型、核心模块代码。 另外几条主线参与了整体设计和关键节点评审，具体执行由其他同学负责。 五条主线之间的协同设计、接口对齐、Harness 规范由我牵头。 所以问到任何一条主线，我都能讲清楚设计思路和关键决策；具体代码细节，可能不如负责的同学深入。
诚实分级比假装全懂更可信
难
Q · 16
自我进化主线最难在哪？你说"不能完全闭环"是因为做不到吗？
考察意图：技术 + 治理双重判断
不是技术做不到，是不应该。技术上 Agent 完全可以监控衰减、生成迭代、自动上线，但风控领域不能这么做。 三个原因。一·合规——策略变更涉及用户权益，监管要求人为可追溯。 二·对抗安全——完全闭环则黑产可通过构造样本"教坏"系统。 三·价值观对齐——什么算"违规"涉及社会价值判断，不能让 Agent 单方面定义。 所以专门留人审委员会作为必经环节——"越靠近自治，越需要保守。"
"不能做"讲成"主动选择不做"，体现治理思维
