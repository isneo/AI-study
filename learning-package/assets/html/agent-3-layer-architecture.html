0为什么需要"3 层"——能力与风险并存的根问题

这份文档想做什么
之前的 《Agent 生态认知地图》 把生态拆成 7 层 stack（L1-L7），那是"生态部件栈"视角。本文换一个抽象维度——直接对准 LLM 的根问题：能力与风险并存。

所有 Agent 框架做的事情，本质上都在回答同一组问题的三个不同侧面：模型凭什么知道该做什么（信息）→ 模型怎么把想法变成行动（运行）→ 模型行动后如何不过界、如何证明可交付（保证）。本文就用信息层 / 运行层 / 保证层这 3 个维度，把生态里 11 个最具代表的组件压进一张 11 × 3 矩阵，看每家做了什么真正的工程突破。

0.1根问题：能力与风险并存

大语言模型有两个同样硬的事实：

  • 能力：通用语言能力让 LLM 能完成大量不同形态的任务——分类、抽取、推理、规划、生成、调用工具。这是过去 60 年 AI 没有的"不用每个任务单独建模"的基础。
  • 风险：
    幻觉幻觉 = LLM 生成语气自信但事实错误的内容。本质原因是模型按"下一个 token 最可能是什么"工作，而不是"这个事实是不是真的"。幻觉不是边角 bug，是架构层面的硬约束——模型规模再大也只能减少不能消除。
    不是边角 bug，是架构硬约束。模型规模再大也只能减少不能消除。Agent 上生产，一个错的执行可能封错用户、扣错款、删错文件。

所以 Agent 框架的真正使命，不是"让模型更聪明"——那是模型公司的事。Agent 框架做的事情，可以用一句话概括：把"给信息、给能力、加边界"三件事工程化。

codex 推演定调（共审依据）
"Agent 框架本质是在释放能力和收敛风险之间做工程化折中。信息层解决模型凭什么知道，运行层解决模型怎么行动，保证层解决模型行动后如何不过界、可交付。这是同一个根问题的三种工程回答。"

1.1三层一句话定义（codex 推演定版）

第 1 层 上下文与记忆 · Information

模型不是"知道世界"，而是"在一次调用里看到什么就依赖什么"。这一层用 4 类注入器（静态指令 / 文件渐进披露 / 运行时按需注入 / 工作记忆）+ 5 类压缩器把对的东西塞进窗口、同时避免无关信息和旧脏数据。2025-Q4 范式：CLAUDE.md / AGENTS.md / Agent Skills 主导，RAG 退到子类。

第 2 层 运行时 · Execution

模型本身只会生成下一步，运行时把"下一步"变成可执行的流程。编排、工具、状态、重试、人工介入和 trace 都在这里——把语言能力接到真实任务链路上。

第 3 层 约束与验证 · Assurance

Agent 最大风险不是不会做，而是看起来会做却越权、编造或交付不可用。约束负责在关键动作前拦住，验证负责在阶段边界证明结果可靠。

第 1 层 · 上下文与记忆 · Information

模型不是"知道世界"，而是"在一次调用里看到什么就依赖什么"。这一层把必要事实/任务边界/历史经验按可控方式注入，同时避免把无关信息、脏记忆和旧判断塞进窗口。
本层失败会怎样
给得太少 → 模型缺事实，开始编（幻觉）；给得太多 → 上下文窗口爆 / 注意力被无关内容稀释 / 旧脏数据污染当前判断。两边都失败——所以这一层不是"塞更多"，而是"准确给"。

3.1调用时上下文：本次调用给什么 (Layer 1a)

每次调 LLM 时，输入是一个消息列表。这个列表里塞了什么、按什么顺序排、长度多少，直接决定模型这次能不能做对。和 codex 多轮推演后定版的分类：4 类注入器 + 1 类压缩器（5 子类）。

codex 推演定调（K2 round）
"RAG 不是必要事实的唯一代表。真正的注入分 4 类：静态指令 / 文件型渐进披露 / 运行时按需注入 / 工作记忆。RAG 只是'运行时按需注入'里的检索型子类，和 MCP 工具结果（执行型）平级。"
"压缩本身值得独立成 5 类：summary-replace / sliding-trim / tool-output pruning / multimedia compression / pointer-reference。"

① 静态指令 Static Instructions

system promptsystem prompt = 给 LLM 的"底层指令"。比如"你是一个风控助手，每次回答先列证据再下结论"。它在普通对话开始前注入，决定整个对话的"性格"。一个 Agent 最核心的设定就藏在系统 prompt 里。
+ 工具描述 + 输出格式约束。整个 session 里基本不变——是 Agent "人格"和"能力边界"的起点。代表：OpenAI Agents SDK 把它做成 Agent 类的构造参数；DSPy 用 signature（如 "question -> answer"）声明输入输出语义，编译器自动生成 prompt，开发者不直接写。

② 文件型渐进披露 File-backed Progressive Disclosure（2025-Q4 业界主流新范式）

这是当前最热的范式。Anthropic "Effective Context Engineering" + 2025-12-18 发布的 Agent Skills 开放标准（已被 OpenAI / Google / GitHub / Cursor 采用，400k+ skills），核心是"不要把所有上下文一次塞进窗口"——分层、按需、文件系统而非向量库。3 个典型形态：

  • Claude Code CLAUDE.md 多层级：User-level（~/.claude/CLAUDE.md，所有项目都加载）+ Project（仓库根 CLAUDE.md 或 .claude/CLAUDE.md）+ Local（CLAUDE.local.md，不入 git）+ Managed Policy（企业管理员部署）。session 启动时从 cwd 向上 traversal 文件系统根，所有命中的 CLAUDE.md 按层级顺序 concat 进上下文（不是优先级覆盖；同目录里 .local append 在 CLAUDE.md 后面，树上从 root 到 cwd 排序）；进入子目录读/写文件时按需加载那一层的 CLAUDE.md。来源：code.claude.com/docs/en/memory
  • OpenAI Codex CLI AGENTS.md 分层：仓库顶层 AGENTS.md + 子目录 AGENTS.md（如 codex-rs/tui/src/bottom_pane/AGENTS.md）+ child_agents_md feature flag 自动告诉 Agent "你正在某子目录的 scope 下"。来源：developers.openai.com/codex/guides/agents-md
  • Agent Skills 开放标准：每个 skill = 一个目录 + SKILL.md + YAML frontmatter（name + 1 行 description）+ 可选 references / scripts。startup 阶段只把所有 skill 的 name+description 注入 system prompt（约 80 tokens/skill），让 LLM 自己判断什么时候需要；需要时才 load full instructions；再需要时才 load supporting files。"一个 Agent 知道几十个 skills 的 metadata 成本 < 加载一个 full skill"。

③ 运行时按需注入 On-demand Injection（拆 2 个子类）

  • 检索型 (Retrieval)：
    RAGRAG = Retrieval-Augmented Generation，检索增强生成。先用语义检索从知识库取出相关片段，再把片段当 context 给 LLM 一起生成答案。是缓解 LLM 幻觉、补充时新知识的标准做法。
    / 向量召回 / BM25 / 实体加权 / reranker——从知识库取材料。代表：LangChain LCEL 的 retriever | prompt | model | parser。但 RAG 只是这一层的一种。
  • 执行型 (Execution)：Function Calling / MCP tool 结果 / Memory store query 结果——从运行时调用产生的。代表：OpenAI Agents SDK 的 Tool 一等公民，结果按 schema 回填进上下文。

关键差异：检索型是"主动找已存在的材料"，执行型是"主动产生新的事实"——两者对幻觉防御的语义完全不同。

④ 工作记忆 Working Memory

之前几轮对话 + 工具调用结果回填——让模型知道"我之前都做过什么"。包含：assistant 的 reasoning chain、tool calls 与 results、user 的追问。是上下文窗口里增长最快的部分，也是压缩器主要处理的对象。LangGraph 的 State 是这一层的一等公民（节点函数读/写 state）；OpenAI Agents SDK 的 Sessions 抽象短期会话。

上下文窗口有硬上限（GPT-5.5 / Claude 4.5 ~200K tokens；超过 ~30% 注意力会被稀释）。压缩不是"删信息"，是"把重要信息用更少 token 表达"。codex K2 round 给出 5 类（不是 4 类——遗漏的是 Pointer/Reference Compression）：

3.2闭环记忆模块：跨调用沉淀什么 (Layer 1b)

记忆和上下文的区别在于跨调用：上下文是"本次调用给什么"，记忆是"这次调用产生的事实，下次还要用"。注意：同一份 CLAUDE.md / AGENTS.md 既出现在 3.1（注入机制）也出现在 3.2（长期资产）——但角度不同：3.1 讲它如何被加载进本次调用，3.2 讲它如何作为可审计的长期资产被维护。codex K2 round 强调这一双重身份。

3.4不要误用

· RAG 不是上下文注入的全部——2025-Q4 业界主流已经升级到"文件型渐进披露 + 运行时按需注入"。还把 RAG 当成"必要事实唯一注入手段"的项目架构上落后了一代。
· CLAUDE.md / AGENTS.md 不是"普通文档"——它是session 启动时按文件系统 traversal 自动加载的指令记忆，写错一条整个 Agent 行为偏。要 git review 当代码改，不能当 README。
· Agent Skills 不等于 MCP——MCP 解决"工具协议跨厂商"，Skills 解决"工具/知识列表爆炸"（progressive disclosure），互补不冲突。
· 压缩 5 类不能互相替代——summary-replace（贵 + 有信息损失）、sliding trim（保命）、tool output pruning（最便宜）、multimedia（横切）、pointer/reference（不损失但需 fetch 能力）——生产 Agent 一定混合。只做 summary-replace 是"老 LangChain ConversationSummaryMemory" 时代的做法。
· mem0 不是权限系统——它管"记什么、怎么去重"，不管"谁能改"。tier ACL 要自己加。
· Letta 的 read_only block 不是审计日志——它防止 Agent 改，但不记录"谁试过改"。审计要另起。
· 文件型记忆 ≠ 向量型记忆的替代——文件型擅长规则规约（红线 / SOP），向量型擅长语义经验（历史 case）。混合用，不互相吃。
· LangChain 不是 Agent 框架——LCEL 适合线性 RAG，复杂分支 / 循环 / 人审中断应升级到 LangGraph。
· "塞更多到 context"≠ 模型表现更好——超过 ~30% 上下文窗后，注意力会被稀释，反而效果下降（experiment-verified）。这正是 Agent Skills "只在 startup 注 80 tokens/skill metadata"设计的根本原因。

4.4主贡献组件精读

LangGraph · 状态图 + checkpoint + interrupt（运行层主力）

突破：把 Agent 显式建模成状态机——节点 = 步骤，边 = 转换条件。checkpoint 让长任务挂了能从最近一步接着跑；interrupt 让 HITL（人在回路）变成图里的一个节点而不是事后补丁。

核心抽象：StateGraph(State) · add_node(name, fn) · add_edge(from, to) · add_conditional_edges(from, condition_fn) · Checkpointer · interrupt()

参考方式：Python 库不直接用，但"状态图 + checkpoint + interrupt"思想必须照搬到 Java——快手有 LangGraph4j 这个 JVM 端口可以直接用，或者自研 ~150 行受控最小状态机（v1 case 调查不复杂）。HITL 一等公民 这一条对风控审核场景是底层不可妥协的设计。

外部依据：LangGraph HITL 文档

OpenAI Agents SDK · agent + handoff + tools + tracing（运行层主力）

突破：把"构建一个 Agent"压成几个一等公民抽象——Agent · Runner · Tool · Handoff · Guardrail · Tracing · Sessions。handoff 是特殊工具，Agent 转交控制权就是调一个特殊 tool；tracing 内置，每一步自动出 span；guardrails 一等公民（详见第 3 层）。

参考方式：Python/TS 库不直接用（快手是 Java），但抽象集 = 你自研 Java Agent 框架的最小 vocab。把 Agent / Tool / Handoff / Guardrail / Trace 这套词在 Java 这边各起一个类，是最省脑的起点。

外部依据：openai-agents-python 文档 · guardrails 文档

Claude Code · 30 行 loop 哲学 + plan/permission 工具循环（运行层主力）

突破：Anthropic 的 "Building Effective Agents" 白皮书 主张优先选简单可组合的实现，必要时再上框架但要理解底层机制——只要把状态、工具、人审、审计这 4 件事写清楚，让 LLM 在受控 loop 里跑就够。Claude Code 是这套哲学的工程化版本。

参考方式：哲学照搬——快手 Java Agent v1 先用受控最小 loop（显式状态枚举 + 预算/超时/trace + 异常回滚 + 单文件 ~150 行）跑通核心；v2 case 数变多 + 流程复杂时再升级到 LangGraph4j 状态机。"先简单后复杂、必要时再上框架"，这是 Anthropic 的真实主张。

OpenAI Codex CLI · Rust + 近 90 个子 crate 的工程化 agent（运行层主力）

突破：把 Agent runtime 拆成极细粒度的 Rust crate 体系（codex-rs/ 下近 90 个子模块 / crate · 实测 ls codex-rs/*/ 共 93 项）：core（runtime）· execpolicy（执行策略 DSL · 见第 3 层）· sandboxing（OS 级隔离）· hooks（事件 + registry）· memories（read/write 分离）· rollout + rollout-trace（trace/rollback）· tools（工具注册）· apply-patch（diff 应用）· tui（终端 UI）。每个 crate 独立责任，工程纪律标杆。

codex-rs/core/ · codex-rs/execpolicy/ · codex-rs/sandboxing/ · codex-rs/hooks/{src,schema,bin} · codex-rs/memories/{read,write} · codex-rs/rollout/ · codex-rs/tools/

参考方式：Java 工程化要学这套包结构哲学——每个责任一个独立 maven module（runtime / execpolicy / sandbox / hooks / memories / rollout / tools / apply-patch / ui），不要把所有逻辑塞一个大 service。这是Rust 风格的极致 SRP 直接映射到 Java 多模块设计。

OpenClaw · Gateway-Product 分离 + plugin SDK 契约（运行层主力）

突破：Mono repo 拆 3 层 SDK 契约——plugin-sdk（插件接口）+ memory-host-sdk（"给插件用的读写记忆"抽象）+ plugin-package-contract（插件包清单 schema：声明权限 / 工具 / 记忆访问）。Gateway 进程负责所有通讯渠道接入，助手进程只关心理解 + 工具调用 + 记忆。

参考方式："Gateway = 控制平面，Product = 助手本身" 直接映射到快手 The Gate 进程职责分离。plugin-package-contract 的契约清单是自研 Agent 插件市场的最佳起点——每个插件声明"我用哪些 tool、读写哪些 memory tier、要什么权限"，The Gate 据此发 token。

Hermes Agent · curator + auxiliary client + ContextVar 写入隔离（运行层 + 第 3 层共用）

突破：4191 行 conversation_loop.py 主循环 + 1781 行 curator.py 背景维护 orchestrator。curator 用独立的 auxiliary client fork 出 review agent，不污染主 session 的 prompt cache。tools/skill_provenance.py 用 ContextVar 区分 "background_review" vs "foreground" 写入来源——curator 只动 agent 自己创建的技能。

参考方式：ContextVar 区分背景 / 前台写入 直接照搬到 Java：ThreadLocal<WriteOrigin> 在每次 LLM 调用前 set，memory_repository.write() 内部检查。auto-promote 只能动 agent 自创的，curator 永不越权改用户手写的内容。archive-not-delete + pinned 跳过自动转换 + 闲置 ≥7 天才跑 这套"自动化但不激进"组合是风控 auto-policy-update 的安全护栏模板。

4.5不要误用

· LangChain 不是 LangGraph——LangChain 适合线性 RAG，复杂状态/分支/人审恢复用 LangGraph。
· Multi-Agent Team（CrewAI / AutoGen）生产慎用——N 个 Agent 互相喊话 → token 暴增 / 对话失控 / 错误归因困难。研究 demo 漂亮，生产关键路径会出事故。
· Temporal 不是 Agent 框架——它解决"流程怎么活下来"，不解决"Agent 怎么思考"。建议 LangGraph + Temporal 叠加，不要单用 Temporal 当 Agent runtime。
· OpenAI Agents SDK 的 Handoff 不是万能——handoff 后老 Agent 的上下文要不要带过去？审计如何串起来？职责边界如何定义？大多数项目两个 Agent 之后就乱。
· trace / observability 不能替代 checkpoint / resume——Langfuse / OpenAI Tracing 让你看见失败，不会让长任务自动从中断恢复。要做长事务恢复，必须叠加 LangGraph Checkpointer 或 Temporal durable workflow。

第 3 层 · 约束与验证 · Assurance

Agent 最大风险不是不会做，而是看起来会做却越权、编造或交付不可用。约束负责在关键动作前拦住，验证负责在阶段边界证明结果可靠。
本层失败会怎样
无约束 → Agent 越权（封错用户 / 删错文件 / 扣错款）→ 上生产即事故。无验证 → 交付质量没保证（代码编译不过 / 计划逻辑矛盾 / 答案明显错）→ 用户/人审接到一堆烂活，最终关掉 AI 回到全人工，Agent 项目失败。这一层不是锦上添花，是 Agent 上生产的前提。
图 3 · 保证层机制图。左：约束（input/output guardrail · execpolicy · sandbox · hook · approval）作用在 Agent 动作前/中。右：验证（lint · DSPy assertion · LangGraph interrupt · Langfuse eval · plan-mode 规则校验）作用在 Agent 动作后/阶段边界。两端共同夹住"越权 + 烂活"两种失败。

5.2验证：事后 / 阶段证明 (Layer 3b)

验证关注"Agent 做完一件事后，怎么证明做对了"。形态：

  • 领域 lint（coding agent 专用）：Agent 改完代码，跑 ruff / eslint / mypy / tsc——编译不过 / 语法错直接打回。Aider / Codex / Claude Code 都内置。
  • Plan 模式规则校验：Agent 先输出计划（plan），系统按规则校验计划本身合不合理——比如"这个计划里有删除生产数据的步骤" → 阻断。
  • DSPy assertion + metric：声明 "输出必须满足 X 条件"，不满足时自动重试 / 自我修正——见 DSPy Assertions 论文。
  • LangGraph Interrupt 人审节点：跑到关键边界自动暂停，把当前结果交给人审通过/拒绝。HITL 标准实现。
  • Langfuse trace + eval 闭环：每个 trace 都可被 LLM-as-judge 或人工标注打分，差的 trace 进 dataset 用于 prompt 迭代或微调。把运行过程变可复盘资产。

    decision = "prompt",
    justification = "git push 影响远端，需用户确认",
    match = [["git", "push"], "git force-push"],   # 加载时自校验
    not_match = [["git", "status"]],

外部依据：DSPy Assertions 论文 · dspy.ai

Claude Code · plan/permission/hook 三件套（保证层标杆）

突破：Anthropic 把"先 plan 后 execute" 做成 Claude Code 的核心心智——Agent 接到任务后先输出计划（plan mode），人审通过后再执行。permission 机制让用户选择"这次允许 / 这次拒绝 / 总是允许 / 总是拒绝"，每个权限决定持久化。hook 机制让用户在 TaskStart / BeforeToolCall / AfterToolCall / TaskCancel 等关键事件插自定义代码。

参考方式：plan-then-execute 思想直接照搬到快手风控 case——Agent 先输出"我打算这样调查"，分析员人审通过后再执行真实查询/封禁。permission 4 档是 The Gate 上的标准接口设计。hook 机制和 Codex CLI 的 hooks 是同思路，可以混合借鉴。

外部依据：Claude Code 文档 · Building Effective Agents 白皮书

Langfuse · trace + eval + annotation 闭环（保证层标杆）

突破：开源 LLM 应用观测平台。每个 Agent 任务都是一条 trace，trace 下挂多个 span（每次 LLM 调 / tool 调）。每个 trace / span 都可以被评分——LLM-as-judge 自动评 + 人工标注。差的 trace 进 dataset，用于 prompt 迭代或微调数据。把 Agent 运行过程变成可复盘资产。

参考方式：Langfuse 可以直接部署在快手内网（开源 + 自部署 + Java SDK）作为风控 Agent 的统一观测面。Trace + Eval + Annotation 闭环是 AI Native 风控的数据飞轮源头——每个 case 的 trace 进 dataset，差的 case 改进 prompt，好的 case 沉淀 Memory tier。这是直接用 + 思路借鉴的混合。

外部依据：langfuse.com/docs

LangGraph Interrupt · HITL 标准实现（保证层 + 运行层共用）

突破：interrupt() 在状态图任意节点调用 → Agent 暂停 + 写盘 + 退出 → 外部把"人审通过/拒绝"喂回 → 从中断点继续。HITL 不是事后补丁，是图里的一等公民。

参考方式：思想必须照搬——任何风控高影响动作（封禁 / 资金 / 公开声明）必须用类似机制走 HITL，不允许 Agent 自动 commit。Java 实现可以用 LangGraph4j 的 interrupt 或自研最小状态机时加 WAITING_HUMAN 状态。

外部依据：LangGraph HITL 文档

5.4不要误用

· Guardrail 不能替代 sandbox——Guardrail 在语义层拦（输出含违规词），Sandbox 在系统层拦（不让 Agent 真的 rm -rf /）。两层都要。
· execpolicy 不是 IAM——它管命令能不能跑，不管用户能不能调 Agent。组合用：IAM 决定谁能开 session，execpolicy 决定 session 里能做什么。
· Langfuse 不是自动修复系统——它记录 + 分析，但发现差 trace 后要不要改 prompt / 重新微调，是人决定的。
· DSPy assertion 不是 lint 替代——assertion 关注"语义满足"（输出包含必要字段、不矛盾），lint 关注"语法 / 编译正确"。Coding agent 要叠加用。
· plan mode 不是 100% 保险——plan 写得对不一定 execute 就对（plan→execute 之间可能因状态变化失效）。plan + 阶段验证 + 人审 + sandbox 才是完整链路。

6.2三层落地优先级（codex 共同推演）

优先级建议
若你的 Agent 是 PoC：先信息层（RAG + 简单 system prompt），让模型先能用起来。
若你的 Agent 进了内部 beta：补运行层（state graph + trace），让你能调试。
若你的 Agent 准备上生产：必须补保证层（guardrail + execpolicy + sandbox + interrupt），缺一不可。顺序反了会有事故——见过很多团队上来就建复杂 multi-agent，但没有 sandbox 和 trace，上线第一天就被 prompt injection 攻陷。
