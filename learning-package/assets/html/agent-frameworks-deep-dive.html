3.1

Spring AI · Java AI 应用标准适配层

Java GA 1.1.6 Spring 官方 8/10

是什么

Spring AI 是 Spring 官方出的 AI 应用集成层，目标是把大模型调用、Tool Calling、RAG、向量库、Chat Memory、Advisor 等能力放进 Spring Boot 工程习惯里。它不是重型 Agent 框架，更像 Java AI 应用的"标准适配层"。当前已经是 Java 企业应用里最值得优先看的底座之一。

当前状态：1.1.6 GA（2026 年 5 月），1.0 在 2025 年 5 月 20 日 GA。生产采用度主要来自 Spring 生态用户。

1.1 重要变化提醒（影响落地）
1.1 起 ChatMemory Advisor 不再隐式从请求上下文里取 conversationId，必须在调用处显式传入 conversationId（例如通过 ChatClient.prompt().advisors(a -> a.param(CONVERSATION_ID, sessionId))）。否则会话记忆会落到默认 scope 上，多用户混用时出现串记忆。Memory v1 升 v1.1 的工程要做 grep + 改造。

核心抽象

核心是 ChatClient · ChatModel · Tool · Advisor · ChatMemory · VectorStore。

  • ChatClient：流畅 API 调模型，类似 RestClient / WebClient 风格
  • ChatModel：底层模型抽象，30+ 提供商（Anthropic / OpenAI / Qwen / vLLM / Bedrock / Vertex）
  • @Tool：Java 方法暴露给模型，类似 Spring MVC 的 @RequestMapping
  • Advisor：像 Spring 拦截器，可做 RAG / Memory / 审计 / 上下文增强
  • ChatMemory：管对话历史（短期 + Conversation Memory）
  • VectorStore：向量库抽象（Milvus / Pinecone / Qdrant / PGVector / Redis 等 20+）

抽象偏薄——它不替你设计复杂 Agent 状态机。

代表代码

@Service
class RiskTools {
    @Tool(description = "查询用户最近30天风险档案")
    public String queryUserArchive(String userId) {
        // 实际调你现有 Java 服务（Dubbo / 内部 API）
        return archiveService.query(userId, "30d");
    }
}

@RestController
class AgentController {
    private final ChatClient chatClient;

    AgentController(ChatClient.Builder builder, RiskTools tools) {
        this.chatClient = builder
            .defaultTools(tools)        // 把 Java 工具注册给模型
            .build();
    }

    @GetMapping("/agent")
    String investigate(String userId) {
        return chatClient.prompt()
            .user("判断用户 " + userId + " 是否疑似营销号，并说明证据")
            .call()                    // 触发模型，模型可选择调 tool
            .content();                // 返回最终文本
    }
}

设计哲学

Spring AI 解决的是"Java 工程怎么稳定接模型、工具、RAG、记忆"，不解决"复杂 Agent 怎么编排、怎么仲裁、怎么长期治理"。它的优势是贴近 Spring Boot、易接入现有 Java 服务；缺点是 Agent Society、The Gate、状态机、长流程要自己建。

风控适用度评估

可直接借鉴
Tool 管理（@Tool 注解风格）· Advisor 拦截链（做 The Gate 部分检查很合适）· ChatMemory · VectorStore 接口 · 模型厂商抽象（自然支持多 provider 路由）
不适合直接承担
多 Agent 协作 · 强状态流转 · 跨域仲裁 · 生产写动作治理（这些 Spring AI 都不管）

结论：Spring AI 适合做LLM Gateway SDK + Tool 接入层 + RAG/Memory 适配层，不是终态 Agent 架构本身。

3.2

LangChain4j + LangGraph4j · Java Agent 编排

Java 17+ GA + Spring AI 集成 8.5/10

是什么

LangChain4j 是 LangChain 思路在 Java 生态的实现，核心目标是让 Java 应用方便接入 LLM、Tools、Memory、RAG、AiService。

LangGraph4j 补上"状态图 / 多步流程 / Agent 编排"的部分，借鉴 LangGraph 的 StateGraph 思路。最重要的是——LangGraph4j 已经跟 Spring AI 原生集成（提供 AgentExecutor.builder() API）。

二者组合对快手全 Java 路线非常关键。

核心抽象

LangChain4j：ChatLanguageModel · AiServices · @Tool · ChatMemory · EmbeddingStore · ContentRetriever。

AiServices 很关键：用 Java interface 把 LLM 包装成强类型服务——这跟 Spring Data 的 Repository 接口式定义同样优雅。

LangGraph4j：StateGraph · Node · Edge · AgentState · Checkpoint。把 Agent 从"while 循环"推进到"显式状态机"。

代表代码（LangChain4j AiService 风格）

// 1. 定义 AI service（像写 Spring Repository）
interface RiskAssistant {
    String analyze(String caseText);   // 像调普通 Java service 一样调 LLM
}

// 2. 定义工具
class RiskTools {
    @Tool("查询相似历史 case")
    String similarCases(String text) {
        return "相似 case: case_123, case_456";
    }
}

// 3. 装配 + 调用
ChatLanguageModel model = OpenAiChatModel.builder()
    .apiKey(System.getenv("OPENAI_API_KEY"))
    .modelName("gpt-4o-mini")
    .build();

RiskAssistant assistant = AiServices.builder(RiskAssistant.class)
    .chatLanguageModel(model)
    .tools(new RiskTools())
    .chatMemory(MessageWindowChatMemory.withMaxMessages(10))
    .build();

String result = assistant.analyze("这个视频疑似营销号，请给证据链");

代表代码（LangGraph4j StateGraph 风格）

// 1. 定义状态
class SimpleState extends AgentState {
    public static final String MESSAGES_KEY = "messages";
    public static final Map<String, Channel<?>> SCHEMA = Map.of(
        MESSAGES_KEY, Channels.appender(ArrayList::new)
    );
    public SimpleState(Map<String, Object> initData) { super(initData); }
}

// 2. 定义节点
class GreeterNode implements NodeAction<SimpleState> {
    public Map<String, Object> apply(SimpleState state) {
        return Map.of(SimpleState.MESSAGES_KEY, "Hello!");
    }
}

// 3. 构图 + 运行
var graph = new StateGraph<>(SimpleState.SCHEMA, SimpleState::new)
    .addNode("greeter", node_async(new GreeterNode()))
    .addEdge(START, "greeter")
    .addEdge("greeter", END)
    .compile();

for (var item : graph.stream(Map.of(SimpleState.MESSAGES_KEY, ""))) {
    System.out.println(item);
}

跟 Spring AI 集成（关键）

// LangGraph4j 自带 Spring AI 集成（spring-ai-agent 模块）
var agent = AgentExecutor.builder()
    .chatModel(springAiChatModel)           // 直接用 Spring AI ChatModel
    .toolsFromObject(new RiskTools())     // 注册工具
    .build()
    .compile();

for (var item : agent.stream(Map.of("messages", "分析这个 case"))) {
    System.out.println(item);
}

设计哲学

LangChain4j 追求"Java 开发者自然使用 LLM"，比 Spring AI 更 Agent 化；LangGraph4j 追求"流程显式化 + 状态化"。它们的核心价值是把 Agent 从 prompt 拼接拉回 Java 类型系统和状态机。

风控适用度评估

可直接借鉴
AiService 强类型接口（让 Agent 输出是 EvidenceBundle / Recommendation DTO 而不是字符串）· @Tool 注解 · Memory / RAG 抽象 · StateGraph 编排范式 · 跟 Spring AI 的集成方式
不适合直接照搬
权限治理 · The Gate · 审计 · 跨域冲突协议 · 超大规模风控对象模型

结论：它最适合参考 Agent Contract、Tool Contract、StateGraph DSL，生产控制面仍要自研。

3.3

Embabel · Rod Johnson 的 JVM Agent 框架（GOAP）

Kotlin/Java 早期 Spring 创始人 7.5/10

是什么

Embabel 是 Spring 创始人 Rod Johnson 创立的、JVM/Spring 风格的 Agent 框架。定位比 Spring AI 高一层——Spring AI 更像 Servlet API，Embabel 更像 Spring MVC。

它试图把 Agent 拆成目标、动作、条件、规划和执行，更贴近企业 Java/Kotlin 工程。

关键特性：借鉴游戏 AI 的 GOAP（Goal-Oriented Action Planning） 算法做规划——这是它跟所有其他 Agent 框架的核心差异。

当前状态：生态还新（2025 年首次公开），生产采用度不如 Spring AI / LangChain4j，但设计哲学值得认真读。

核心抽象

核心是 Agent · Action · Goal · Tool · Planning。

关键区别：不是让你手写固定链路，而是定义"有哪些动作、动作需要什么输入、产出什么结果、能达成什么目标"，再由框架围绕目标组织执行。

抽象比 Spring AI 重，比 LangChain4j 更偏 agent runtime。

代表代码

@Agent(description = "营销号风险调查 Agent")
class MarketingRiskAgent {

    @Action
    public Evidence collectEvidence(CaseInput input) {
        // 普通 Java 方法，描述一个可执行动作
        return new Evidence("近30天高频发布营销内容，关联商家3个");
    }

    @Action
    public Recommendation recommend(Evidence evidence) {
        // 下游动作消费上游动作产物，框架自动做依赖编排
        return new Recommendation("ESCALATE", "证据不足，建议人审复核");
    }
}

这段代码表达 Embabel 的关键思想：不是写"第 1 步、第 2 步"的脚本，而是声明动作和产物，让框架围绕目标组织动作。框架基于 GOAP 算法自动决定调用顺序。

设计哲学

Embabel 想解决"企业级 Agent 不该只是 prompt + while loop"，而应该像 Spring 应用一样由动作、目标、依赖和工具组成。Rod Johnson 的论点："JVM 是大部分企业业务逻辑的地方，AI agent 应该贴近 JVM 而不是被 Python 抢走"。

风控适用度评估

可借鉴（思想，不是 API）
Action-driven 设计 · Action Catalog 概念 · Spring 生态整合方式 · 动作依赖建模 · GOAP 规划思想（可借鉴用来做风控调查的动作选择）
风险
生态年轻 · API 稳定性 · 生产案例少 · 社区规模都要验证

结论：它适合启发 Agent 内部动作模型，但不建议直接作为主框架押注。借鉴 Action / Goal 思想去做自研的 Action Catalog 是关键。

3.4

LangGraph · Python 状态机 Agent 的事实标准

Python GA 大量生产采用 9 思想 / 6 落地

是什么

LangGraph 是 LangChain 生态里专门做有状态 Agent 编排的框架，Python 为主。它把 Agent 建模成图：节点是步骤，边是流转，状态贯穿全程，Checkpointer 负责恢复，Interrupt 支持人类介入。

它是当前最值得研究的 state machine agent 框架——LangGraph4j 是它的 Java 实现。

核心抽象

核心是 StateGraph · State · Node · Edge · Conditional Edge · Checkpointer · Interrupt · Subgraph。

  • State：共享上下文（用 TypedDict 定义 schema）
  • Node：一步计算（函数）
  • Edge：决定下一步
  • Conditional Edge：基于 state 动态决定走哪个 node
  • Checkpointer：保存状态（支持 Postgres / Redis / In-Memory），跨进程可恢复
  • Interrupt：让流程停下来等人（human-in-loop 原生支持）
  • Subgraph：图嵌套（复杂场景）

它是重编排框架，不只是模型调用封装。

代表代码

from typing import TypedDict
from langgraph.graph import StateGraph, END
from langgraph.checkpoint.memory import MemorySaver

# 1. 定义状态 schema
class RiskState(TypedDict):
    case: str
    evidence: str
    recommendation: str

# 2. 定义节点函数
def load_context(state: RiskState):
    return {"evidence": "近30天发布营销视频12条，历史处罚2次"}

def synthesize(state: RiskState):
    return {"recommendation": f"建议人审复核，证据：{state['evidence']}"}

# 3. 构图
graph = StateGraph(RiskState)
graph.add_node("load_context", load_context)
graph.add_node("synthesize", synthesize)
graph.set_entry_point("load_context")
graph.add_edge("load_context", "synthesize")
graph.add_edge("synthesize", END)

# 4. 编译时挂上 checkpointer（支持状态保存 / 恢复）
app = graph.compile(checkpointer=MemorySaver())
result = app.invoke({"case": "疑似营销号视频"})

设计哲学

LangGraph 解决的是"Agent 不能只是不可控循环，必须显式状态化、可恢复、可插人、可观测"。它不关心 Java 生态，也不替你做企业权限和生产写动作治理。

风控适用度评估

可直接借鉴（思想 9/10）
StateGraph · Checkpointer · Interrupt · Human-in-loop · 条件边 · 可恢复执行 · Subgraph 嵌套
直接落地 6/10
Python 技术栈 · 与快手 Java 服务体系距离大 · 生产权限治理不足

3.5

Temporal / Restate / Conductor · Durable Workflow 底座

Java / 多语言 大厂生产 非 Agent 框架 9/10

是什么

它们不是 Agent 框架，而是 durable workflow engine。Temporal 最成熟，强在长流程、重试、超时、状态恢复、补偿；Conductor 源自 Netflix，更偏微服务编排和可视化流程；Restate 更轻，强调 durable execution + event-driven service。

快手 Case 调查 Agent 这种 5-30s、可能失败重试、要恢复现场的流程，Temporal 是关键底座。

核心抽象（Temporal）

Workflow · Activity · RetryPolicy · Signal · Query · Timer · Workflow State。

  • Workflow：管流程状态（durable）
  • Activity：执行外部副作用（调 LLM / Tool）
  • Worker：执行 Workflow + Activity 的进程
  • RetryPolicy：失败按策略重试
  • Worker 挂了可恢复（核心卖点）

Temporal 不懂 LLM，但很适合托管 Agent 外层生命周期。

代表代码（Java SDK）

@WorkflowInterface
public interface CaseInvestigationWorkflow {
    @WorkflowMethod
    InvestigationResult run(CaseRequest request);
}

public class CaseInvestigationWorkflowImpl implements CaseInvestigationWorkflow {
    private final AgentActivities activities =
        Workflow.newActivityStub(AgentActivities.class,
            ActivityOptions.newBuilder()
                .setStartToCloseTimeout(Duration.ofSeconds(30))
                .setRetryOptions(RetryOptions.newBuilder()
                    .setMaximumAttempts(3)
                    .build())
                .build());

    public InvestigationResult run(CaseRequest request) {
        Context ctx = activities.loadContext(request);       // 查档案/证据
        Evidence ev = activities.runAgentLoop(ctx);          // 调 Agent loop
        return activities.persistResult(request.caseId(), ev); // 落库/发消息
    }
}

设计哲学

Temporal 解决"流程一定要跑完，失败可恢复"；不解决"Agent 怎么推理"。它适合做 Java Agent Service Host 的外壳，让 Agent loop 不直接裸跑在线程里——挂了能在另一个 worker 续上。

风控适用度评估

直接借鉴 / 直接采用
Case 调查 / 判罚 review / 策略治理巡检 / 补偿 / 超时 / 幂等 都需要它。Temporal 是快手 Agent 长流程底座的优先选项。
不适合
1500 万 QPS 实时机审；毫秒级链路不能进 workflow（开销太大）

结论：Temporal = Agent 长流程外壳。Restate / Conductor 可作为备选对比，Temporal 是首选。

3.6

MCP (Model Context Protocol) · Tool 协议的 USB-C

协议 (非框架) 事实标准 Anthropic 推出 8/10

是什么

MCP 是 Model Context Protocol，Anthropic 推出的工具/上下文接入协议（2024.11），后来被多家生态采用：Claude / ChatGPT / Cursor / VSCode / Zed / 各种 IDE 都支持。

它不是 Agent 框架，而是让模型应用以标准方式发现和调用外部工具、资源、提示模板。可以理解为"Agent 调工具的 USB-C"。

核心抽象

核心是 Server · Tool · Resource · Prompt · Client · Transport。

  • Tool：可调用动作（带 JSON Schema 参数）
  • Resource：可读上下文（如文档、数据）
  • Prompt：模板
  • Client：模型应用（Claude / ChatGPT 等）
  • Transport：stdio / HTTP / SSE 等

MCP 的边界是"连接工具"，不是"治理工具"。

代表代码（TypeScript SDK）

import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { z } from "zod";

const server = new McpServer({
  name: "risk-tools",
  version: "1.0.0"
});

server.tool(
  "query_user_archive",
  { userId: z.string(), days: z.number().max(90) },
  async ({ userId, days }) => {
    const result = await queryArchive(userId, days); // 调内部档案服务
    return {
      content: [{ type: "text", text: JSON.stringify(result) }]
    };
  }
);

await server.connect(transport);

设计哲学

MCP 解决工具生态碎片化：每个模型应用不用为每个工具单独写适配。但它默认不是风控级安全网关——快手不能把 MCP server 直接接生产服务，必须外包一层 The Gate / Action Adapter。

风控适用度评估

可借鉴
Tool schema · tool discovery · resource/prompt 标准化 · 生态互通（如果未来要对外暴露 agent 能力）
不适合直接照搬
权限 · 审计 · T3 动作隔离 · 跨业务线 ACL · 工具调用预算
MCP 没有这些——它假设 host 可信。但在风控里 Agent 不可信。

结论：MCP 适合作为内部工具协议参考，不应替代 The Gate。可以借鉴 schema 设计但要加风控元数据（domain_scope / required_token / audit_level / rollback_capability）。

3.8

OpenAI Agents SDK · 轻量 Agent Loop + Handoff

Python / TS GA 2025.03.11 Handoff 范式 7.5/10

是什么

OpenAI Agents SDK 是 OpenAI 的生产级 Agent SDK，2025 年 3 月 11 日 GA，取代了早期的 Swarm。它强调轻量 Agent loop + tools + handoff + guardrails + tracing。

它不是 LangGraph 那种显式图范式，更像"Agent 之间通过 handoff 切换控制权"——这是它的核心设计差异。

核心抽象

核心是 Agent · Runner · Tool · Handoff · Guardrail · Tracing · Sessions。

  • Agent：定义指令（instructions）和工具
  • Runner：执行循环
  • Tool：函数 / MCP server
  • Handoff：把任务交给另一个 Agent
  • Guardrail：输入/输出约束（前置 / 后置检查）
  • Tracing：内置可视化（debug 友好）
  • Sessions：会话持久化 / 自动维护 conversation history（跨多轮 Run 共享对话和工具结果）

代表代码（Python）

from agents import Agent, Runner, function_tool

@function_tool
def query_similar_cases(case_text: str) -> str:
    return "similar_cases: case_123, case_456"

content_agent = Agent(
    name="Content Risk Agent",
    instructions="判断内容是否疑似营销号，必须引用证据。",
    tools=[query_similar_cases],
)

compliance_agent = Agent(
    name="Compliance Agent",
    instructions="检查是否涉及合规高风险，需要升级则说明原因。",
)

triage_agent = Agent(
    name="Risk Triage Agent",
    instructions="先判断该交给哪个 Agent。",
    handoffs=[content_agent, compliance_agent],  # handoff 是核心抽象
)

result = Runner.run_sync(triage_agent, "分析这个疑似营销号 case")
print(result.final_output)

设计哲学

它解决"少写 glue code，把 Agent、Tool、Handoff、Trace 组织起来"。核心差异是 handoff，而不是 graph——通过让 Agent A 把控制权 handoff 给 Agent B 实现协作。适合客服、办公、代码、工作流助手；复杂状态机要自己补。

风控适用度评估

可借鉴
Handoff（Agent 间任务移交）· Guardrail（Policy-as-Code 前置检查）· Tracing（内置实现参考）· 轻量 Agent loop · 工具注册体验
不适合直接落地
Python/JS 生态 · 快手全 Java · 强状态长流程 · The Gate · 跨域仲裁

结论：它适合参考"多 Agent 任务切换" 和 guardrail API，不适合作为快手主 runtime。Handoff 思想可以借鉴用到 Risk Agent Society 设计里——但 handoff 必须受 The Gate 和 Policy 控制。

6引出自研必要性

TL;DR
没有任何现成 Agent 框架能直接承载快手风控终态。原因不是它们不成熟，是它们普遍缺少快手需要的四件事：全 Java 工程一致性 / 生产写动作统一 Gate / 风控级权限审计 / 跨域风险对象模型 URG。合理路线 = 自研控制面 + 借鉴成熟抽象 + Java 生态落地。

6.1现成框架直接用为什么不行（5 个原因）

1. 它们大多解决"Agent 怎么跑"，不解决"风控动作怎么被允许"

快手终态最核心不是 Agent loop，而是：

  • 哪些动作能写生产
  • 哪些必须人审
  • 哪些必须多签
  • 哪些永远禁止
  • 谁能签 token
  • 出事怎么回滚
  • trace 如何审计
  • memory 如何撤回

LangGraph / OpenAI Agents SDK / CrewAI / AutoGen / Spring AI 都不会替你解决这些。

所以快手必须自研 The Gate / Action Adapter / Policy-as-Code / Permission Token / Rollback / Kill Switch。

结论 1
框架可以管"推理"，不能管"生产责任"。

2. 现成框架的 Memory 太轻，承载不了风控数据飞轮

风控 Memory 不是聊天历史，也不是普通 RAG 文档。它至少要有：

{
  source_type,
  domain_scope,
  human_verified,
  policy_version,
  trace_lineage,
  tier,                  // A/B/C/D 分级
  expires_at,
  revoked_at,
  applicability_contract,
  cross_domain_allowed
}

LlamaIndex / LangChain / Spring AI 都能做检索，但都没有天然的风控 Memory 状态机。

快手需要的是：

DRAFT -> QUARANTINED -> VERIFIED -> PROMOTED -> DEPRECATED -> REVOKED
结论 2
RAG 框架只能提供检索能力，不能提供风控 Memory 治理。

3. 多 Agent 框架缺少冲突仲裁协议

CrewAI / AutoGen / A2A / OpenAI Handoff 都能让多个 Agent 协作，但快手需要的是：

  • 内容安全 Agent 说高风险
  • 电商 Agent 说正常带货
  • 未成年保护 Agent 说升级
  • 合规 Agent 说证据不足

这时不能靠"聊天继续讨论"。必须有：

  • 硬优先级：合规 > 用户安全 > 反欺诈 > 业务增长
  • Arbiter Agent（仲裁角色）
  • Policy-as-Code 优先级
  • escalation rule
  • conflict trace
  • human final authority
结论 3
现成多 Agent 框架有协作，没有风控仲裁。

4. Python / TS 生态不匹配快手核心生产体系

OpenAI Agents SDK / LangGraph / CrewAI / LlamaIndex / DSPy / Pydantic-AI 都很强，但快手风控主链路是 Java：

  • 机审 Java 实时引擎
  • Dubbo / gRPC 服务
  • Spring Boot 体系
  • Java 策略平台
  • Java 权限 / 配置 / 监控体系
  • 内部研发主力也是 Java 后端

如果核心 Agent runtime 用 Python，会引入：

  • 服务治理割裂
  • SDK / RPC 边界复杂
  • 线上排障成本高
  • 权限体系复用差
  • 组织学习成本高
结论 4
Python 框架适合做参照物，不适合做快手风控核心 runtime。

5. 云厂商 Agent 产品不能绑定快手终态

Bedrock AgentCore / Agentforce / ServiceNow AI Agents 很有参考价值，但快手不能把风控核心控制面托给外部平台：

  • 数据出域
  • 合规不可控
  • 模型路由不可控
  • trace / memory 生命周期不可控
  • The Gate 无法深度内嵌
  • 成本和供应商绑定不可接受
结论 5
云厂商产品适合反推生产能力清单，不适合成为快手风控底座。

TL;DR

层类型允许范围边界

9.3Memory 6 状态机

DRAFT  →  QUARANTINED  →  VERIFIED  →  PROMOTED  →  DEPRECATED
                                ↓                       ↑
                              事故 / 撤回 → REVOKED  ←  长时间未引用

每个状态明确：谁能写 / 谁能读 / 哪些 Agent 可引用 / 过期时间 / 撤回后如何影响历史 trace。详细见 ai-native-risk-case.html Part 8.6.4。

9.5飞轮转速指标

  • Memory 新增 + 升级数 / 月（飞轮"产出"）
  • Eval gold set 增长率（评估能力 vs 业务复杂度赛跑）
  • 新对抗发现 → 入对抗样本库的中位时间（对抗响应速度）
  • Cross-domain memory reuse 率（受 applicability_contract 约束）
  • Agent 版本回滚率（L2 自动 promote 健康度，>10% = guardrail 不严）
  • Failure Taxonomy 覆盖率（覆盖率低 = 飞轮在堆数据不在进化）
  • 人审采纳率随时间变化（Agent 信任度演化）
核心原则
没有 eval 的自进化就是自我污染。飞轮的关键不是"自动越来越聪明"，是"每一次升级都被 eval / 人 / 治理验证过"。

B术语速查

文档全篇出现的内部术语 + 行业术语集中解释。任何卡壳的词都能在这里查到。

Agent 是什么

一句话：一段会调用大模型的代码，能根据大模型的回复去调用工具、把结果再喂回去、直到大模型说"我答完了"。

本质 = LLM 调用循环 + 工具执行。不是新概念。Claude / Codex / ChatGPT 都是这个套路。

StateGraph / State Machine Agent

把 Agent 流程建模成状态图：节点是步骤（函数），边是流转条件，状态贯穿全程。代表是 LangGraph。

相比"while loop 让模型决定下一步"，状态图的好处：每一步显式 / 可追踪 / 可恢复 / 可中断给人。

GOAP (Goal-Oriented Action Planning)

来自游戏 AI 的规划算法。Embabel 借鉴这个思想：给定目标 + 可用动作集 + 前置条件，框架自动选择动作组合。

风控场景：定义"查档案 / 查相似 case / 查策略 / 升级人审"等动作，每个动作带 precondition / output / cost / side_effect_level，框架围绕"判定风险等级"目标自动选择调用顺序。

Handoff

OpenAI Agents SDK 的核心抽象。Agent A 把控制权移交给 Agent B 处理一个任务。

跟 graph 的区别：graph 是中央调度（外层决定下一个 node），handoff 是去中心化（Agent A 自己决定移交给谁）。

风控扩展：handoff 必须受 Policy 约束（不能自由跳转），只能在 Risk Agent Card 声明的 allowed_handoffs 范围内。

MCP (Model Context Protocol)

Anthropic 推出的工具 / 上下文接入协议。让模型应用以标准方式发现和调用外部工具。

核心抽象：Server / Tool / Resource / Prompt / Client / Transport。

类比："Agent 调工具的 USB-C"。已被 Claude / ChatGPT / Cursor / VSCode 等支持。

A2A (Agent-to-Agent Protocol)

Google 推出的多 Agent 通信协议。不同 Agent / 不同系统之间如何发现、通信、委派任务。

核心抽象：Agent Card（描述 Agent 能力）/ Task（委派任务）/ Message / Artifact / Capability。

跟 MCP 的关系："MCP 接 tool，A2A 接 agent"。

Durable Workflow / Durable Execution

Temporal 的核心概念：workflow 状态被持久化，进程崩了能在另一个 worker 续上。

实现方式：Event History（事件历史）记录每一步状态变更，恢复时 replay event history 重建状态。

风控应用：Case 调查 Agent 跑到一半挂了，能从最近 checkpoint 续上，不重头跑。

The Gate（控制平面）

快手自研架构核心概念。所有"写到生产"的动作必须经过的唯一入口。包含 4 个组件：

① Action Adapter（唯一写动作入口）② Policy-as-Code（红线代码化）③ Trace / Replay / Audit（全链路可追溯）④ Rollback（git-style 可回滚）

类比：机场安检——每个想登机的人必须过同一道门 / 同一套检查。

URG (Unified Risk Graph)

快手自研架构核心概念。风控所有事实的统一数据底座。所有人 / Agent / 系统读同一份事实。

MVP 期 4 个核心实体：Case / Action / EvidenceRef / TraceRef / Memory。其他实体逐步并入。

关键约束：Agent 永远不直接写 URG，必须经 Action Adapter 过门。

Risk Agent Society

多个域内 Agent 协同跑的运行时层，跑在 The Gate + URG 之上。

包含：内容安全 Agent 群 / 反欺诈 Agent 群 / 商家电商 Agent 群 / 合规审查 Agent 群 / 跨域 Arbiter Agent。

"Society" 这个词刻意选用——多 Agent 不是"线性流水线"，是"协作社会"：有冲突、有仲裁、有专长分工，但仍受控制平面 + 治理层约束。

Memory Tier (A/B/C/D)

每条 Memory 按"可信度"分 4 级：

  • A 最高 · 事故复盘 / Curator 显式标注 / 监管确认
  • B 较高 · 判罚生效 + outcome 回填后
  • C 中 · 人审采纳 + 复核通过
  • D 最低（DRAFT）· Agent 自己写的 / 未验证 · 不进检索

Agent 写的 memory 永远 tier=D。必须经升级路径才能进检索。防"Agent 错 → memory → 再被引用"。

Failure Taxonomy（失败原因结构化标签）

codex CTO 关键新增。把 Agent 失败拆成可训练的结构化根因标签：

  • EVIDENCE_MISSING（证据缺失）
  • TOOL_ERROR（工具错）
  • PROMPT_ERROR（Prompt 错）
  • MODEL_ERROR（模型错）
  • MEMORY_POISONED（Memory 污染）
  • HUMAN_OVERRIDE_WRONG（人审错被申诉推翻）
  • POLICY_DRIFTED（策略口径变了）

用途：让飞轮知道改 Agent / 改工具 / 改策略 / 改审核培训——没这个飞轮只是堆数据不进化。

applicability_contract（跨域迁移契约）

跨域 Memory 迁移默认禁止，需要显式 contract 才允许：

domain_scope · subject_type · evidence_prerequisites
forbidden_scenarios · migration_audit · holdout_test

例：内容营销号模式不能直接迁移到电商商家。反欺诈设备关联不能直接迁移到未成年保护。

Checkpointer / Checkpoint

LangGraph / LangGraph4j 概念。保存 Agent 在某个时点的完整状态，进程崩了能从 checkpoint 恢复。

实现：序列化 state（JSON / Protobuf）+ 存储到 Postgres / Redis / 文件系统。

HITL (Human-in-the-Loop)

人在环。Agent 跑到某个点停下来等人决策 / 审批 / 修正，然后继续。

LangGraph 原生支持（Interrupt + Checkpointer）。OpenAI Agents SDK 通过 HITL interruptions / RunState 支持（Sessions 用于会话历史，与 HITL 兼容）。

风控应用：高影响判罚 Review、新对抗模式确认、Memory 升级签发等都是 HITL 点。

Tracing / Trace

Agent 跑的完整过程录像：调了哪个 LLM、prompt 是什么、调了哪些工具、参数是什么、返回了什么、最终决策。

用途：监管解释 / 事故复盘 / Agent 回归测试 / 数据飞轮。

跟普通日志区别：结构化 / 可关联（traceId）/ 可原样回放。

Guardrail（护栏）

OpenAI Agents SDK 概念。Agent 输入 / 输出的前置 / 后置检查。

例：用户输入是否含 PII？Agent 输出是否引用了禁用词？

对应快手：Policy-as-Code 的运行时检查。

Eval / Gold Set

Eval：评估 Agent 质量的流程。给定一组测试 case，跑 Agent，对比实际输出 vs 期望输出。

Gold Set：人工精标的"标准答案集"。1000-5000 条人工标注的 (case, 正确判罚) 对。

用途：每日跑 eval / Prompt 变更前 A/B / 新版本上线门槛 / 数据飞轮。

Advisor（Spring AI）

Spring AI 的拦截器链，类似 Spring MVC 的 HandlerInterceptor / Servlet Filter。

每次 ChatClient 调模型前后，按顺序经过多个 Advisor：RAG（注入检索片段）/ Memory（注入历史）/ 审计 / 内容审查 / 重写。可叠加可关闭。

风控场景：The Gate 的"前置 Policy 检查 + 后置审计落库"就可以做成 Advisor 链。

ToolCallback（Spring AI）

Spring AI 把"模型要调一个工具"这件事抽象成 ToolCallback：包含 name / description / inputSchema / 真正执行的 lambda。

@Tool 注解方法在运行时会被包装 / 转换成 ToolCallback；也可以手工 new 一个 ToolCallback 把任意 Java 方法暴露给模型。

对应概念：OpenAI 的 function tool / Anthropic 的 tool / MCP 的 tool。

Handoff（OpenAI Agents SDK）

Agent A 把控制权连同上下文整体交给 Agent B 继续处理（不是 A 调 B 然后 B 返回，是直接让 B 接管）。

实现：Handoff 在 OpenAI SDK 里是一种特殊 tool——模型决定"transfer to specialist agent"。

风控类比：内容安全 Agent 处理途中发现这是涉政 case，handoff 给涉政专项 Agent。但快手要做的是受 The Gate 控制的 handoff，不允许 Agent 自由 handoff。

Sessions（OpenAI Agents SDK）

OpenAI Agents SDK 的会话历史持久化机制：把 user/assistant/tool 消息按 session_id 自动存起来，下一次 Run 自动加载。

用途：跨多轮对话保持上下文，省得每次都手工拼 history 数组。

注意：Sessions 是会话内存，不是 HITL 机制；HITL 由 interruptions + RunState 负责，Sandbox 又是另一套"工具沙箱"能力，三者不要混淆。

RunState / Interrupt（OpenAI Agents SDK · LangGraph）

RunState：OpenAI Agents SDK 在 Agent 执行过程中序列化的中间状态，可挂起、可恢复，是 HITL 的载体——比如 Agent 调到一个需要人审的 tool 时抛 interruption，把当前 RunState 存起来，等人审完接着跑。

Interrupt：LangGraph 里类似的概念，在 graph 某节点 interrupt(payload)，等外部输入回来再 resume。

风控应用：高影响判罚的"机审建议 → 人审决定 → Agent 继续"流程，正好是 RunState/Interrupt 的典型场景。

Agent Card（A2A 协议）

Google A2A 协议里描述一个 Agent 能力的JSON 名片：name / description / url / capabilities / skills / securitySchemes 等字段（详见 A2A 规范），让其他 Agent 自动发现并协作。

快手风控扩展成 Risk Agent Card：加 domain_scope / policy_scope / trust_level / allowed_tools / forbidden_actions / escalation_rule / sla / owner / audit_level。这样平台才能审计"谁可以做什么"。

Policy-as-Code（治理）

把"哪些动作允许 / 哪些禁止 / 触发什么红线"用代码（DSL / OPA Rego / JSON Schema）写死，而不是写在文档里靠人记。

The Gate 在每次工具调用前查 Policy 决定放行 / 拒绝 / 升级人审。例：「单次封禁 ≥1 万账号需双人 + COO 审批」这种规则。

类比 K8s 的 OPA Gatekeeper / 云的 IAM Policy。

Permission Token（权限令牌）

Agent 调任何写操作前必须先去 The Gate 申请一个一次性 token：声明 who / why / scope / amount / ttl，Gate 校验 Policy 后签发。

token 跟着 tool 调用一起带上，事后审计能 100% 还原"谁授权的 / 凭什么授权 / 实际做了什么"。

类比 AWS STS 临时凭证、Kerberos ticket。

Kill Switch（红线开关）

不可绕过的全局熔断：一旦触发，Agent 立刻停止所有写操作，回到只读 / 只建议人审状态。

触发条件示例：误判率突增 / 异常封禁速率 / 错杀大 V / Memory 状态机违例 / 监管事件。

设计要点：① 必须独立于 Agent 自身；② 物理上由 The Gate 强制；③ 触发后只能由人解除。
