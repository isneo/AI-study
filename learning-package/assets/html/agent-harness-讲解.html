01

What · 什么是 Harness？

一个 AI agent 可以被解耦成两部分：Agent = Model + Harness

Model · 模型
负责"思考"：推理、规划、决定下一步做什么。
Harness · 支架 / 框架
模型之外的全部基础设施，负责"行动与约束"。
harness就是个外壳吗，重要吗？模型变强了它不就没用了？它和模型谁主谁次？
  • 模型 ≠ 产品，光换大模型解决不了问题。Anthropic 的硬证据：即便是前沿编程模型 Opus 4.5 跑在 Claude Agent SDK 的循环里、跨多个上下文窗口，如果只给一句高层提示（"克隆一个 claude.ai"），它仍然造不出生产级 Web 应用。问题不在模型聪不聪明，而在 harness 设计得好不好。
  • 评估对象是"模型 + harness 整体"，不能只看模型跑分。评估一个 agent 必须把模型和 harness 放在一起，而非孤立看模型——生产环境里大部分表现差异来自 harness 那一层。
  • model 与 harness 解耦 → 模型可插拔。解耦后可以换更新更好的模型而不重写整个系统。这也是"模型在变成商品、harness 才是壁垒"这类说法的由来——不过要保留：很多任务上模型能力天花板仍决定上限，harness 救不回一个根本不行的模型。

    Why · 为什么要有 Harness？

    Harness 不是某天被"发明"出来的，而是当模型能力越过某个门槛、人们想让它"自主跑很久"时，被现实问题逼出来的一层基础设施。

    2.1演进时间线（为什么是现在）

    Harrison Chase 把这段历史明确分成几个时代：从纯文本进出的原始 LLM，到流程被人写死的 RAG / workflow，再到把 LLM 放进控制流中心的 agent 雏形，最后到模型够强、能长时间自主工作后正式成型的 long-running agent harness。

    纠偏：警惕"把 harness 说成 2026 年初某人一拍脑袋发明的新概念"这种叙事。它的实质（循环、工具、记忆、状态管理）在 2022–2024 一直在被构建，只是当时没有统一名字——先有现象和工程实践，后有"harness"这个标签，不是反过来。

    2.1.1演进的两个方向：既"长出新能力"，也要"砍掉死重"

    "演进"不只是往外长。Anthropic 在更晚的《Scaling Managed Agents》(2026-04) 里点出了关键的另一半：harness 编码的，是"模型现在还做不到什么"的假设；而这些假设会随模型变强而过期，需要被频繁质疑。一个很具体的例子：Claude Sonnet 4.5 会在感到接近上下文上限时过早收尾（"上下文焦虑"），于是他们给 harness 加了 context resets；但换到更强的 Opus 4.5 上，这个行为消失了，那些 resets 就成了死重。

    长出新能力 · Chase
    "脚手架被改造、往外长"——模型能力解锁新场景，harness 随之扩展。
    砍掉过期假设 · Anthropic
    "做减法"——某些为旧模型加的脚手架，在新模型上变成多余甚至有害的死重，应当删除。

    Anthropic 自己的定调是中间立场："我们预期 harness 会持续演进"——既不是无限堆厚，也不是单纯瘦身。这也把第 1 章"模型 / harness 可插拔"推进了一步：可插拔的动态版，就是每次模型升级都要回头质疑"哪些脚手架已经变成死重"。

    一个常被误传的说法："模型决定智能上限，harness 决定可靠性下限"——只对了一半。过度约束的 harness 会反过来压低一个聪明模型的上限（那些 resets 就是例子）。模型与 harness 是动态关系，不是静态分工。

    2.2模型的先天缺陷（为什么必须有）

    一个 LLM，本质就是一个函数——输入一段文本（上下文），输出一段文本。从这个朴素定义出发，可以推出六条先天缺陷。它们性质各不相同（能力、留存、信息、质量、方向、安全），每一条都只靠"把模型做得更聪明"填不上，所以必须由外部系统来补。

    表 2.2 · 模型的六条先天缺陷
    模型固有特性由此而来的缺口
    ①只能吐文本，文本本身不改变世界没手脚：不能执行动作、影响外部环境
    ②无状态，调用之间不保留任何记忆记不住：每次调用都从零开始
    ③输入长度有上限（上下文窗口有限）装不下：复杂任务塞不进单个窗口
    ④输出是概率采样的会出错：结果不稳定、可能自信地犯错
    ⑤无内在目标，不会自己停没方向：不知朝什么目标走、何时算完
    ⑥不区分"指令"与"数据"没边界：读到的内容可能被当成命令执行
    • ②③ 有因果，不是孤立两条。长程 agent 的核心难题是它必须分段（discrete sessions）工作，每个新会话开始都没有之前记忆；就像一个项目由轮班工程师接力、每个新人都不知上一班干了啥。"装不下"（③）正是"为什么必须把任务切成多段"的原因，而一旦切段，"记不住"（②）就成了致命伤。
    • ④⑤ 的佐证来自可复现的翻车。裸模型跑长任务时会贪多（想一击做完整个 app）、虚报完成（看到进展就宣布结束）、跳过验证（没测就标记完成）——很大程度正是 ⑤ 的体现：它对"目标到哪了、该不该停"没有内在感知。
    • ⑥ 是注意力机制的固有结果。模型对上下文里所有 token 一视同仁，天生分不清"主人的命令"和"网页里夹带的命令"。这是"提示注入"成为结构性问题的根源。

    这六条合起来，决定了 harness 存在的全部理由：模型聪明，却被困在一个只能读写自身上下文的盒子里。

    边界说明：这"六条"是本文从"LLM 是无状态、概率性、文本进文本出的函数"做的推演框架，非某篇官方现成清单；但每条背后的具体事实（无状态、上下文有限、会虚报完成、提示注入等）均出自 Anthropic 等一手资料。本节只讨论模型固有特性导致的缺陷——"可观测性、成本、合规审计"这类推不出自模型本身的需求，属生产环境/组织的外部要求，留待后文。

    Solve · 能力与机制

    Harness 到底具备什么能力？

    承接第 2 章六条先天特性，本章做两件事：一是从每条特性推出 harness 必须具备的能力；二是讲清这些能力怎么靠"循环"组织、运转。本章只谈 harness 本身的抽象机制，不绑定具体产品。

    3.1从六条特性，逐条推出六类能力

    "装外设"还是"不信任产出"。

    给模型加外部部件（补 ①②③）—— 恰好对应 Anthropic 的 augmented LLM

    先天特性（缺）推出的能力（补）一句话职责
    ① 没手脚A. 动作执行（工具）让模型发出结构化动作请求，由 harness 真正执行，并把结果作为"观察"喂回
    ② 记不住B. 外部状态 / 记忆把跨轮要保留的信息存到模型之外、需要时读回。记忆（跨会话的持久知识，现已是 Anthropic 的 /mnt/memory/ 功能）是它最显著的一种；外部状态是更宽的伞，还含任务内进度（成熟形态是只追加事件日志，活在窗口外、按需切片回读）
    ③ 装不下C. 上下文管理决定每一步把"哪些"塞进有限窗口（检索 / 取舍 / 压缩）

    检查 / 约束模型的输出（补 ④⑤⑥）—— harness 的"纪律层"

    先天特性（缺）推出的能力（补）一句话职责
    ④ 会出错V. 验证与反馈用外部客观信号检查模型产出，错了把反馈喂回去重做
    ⑤ 没方向、不停L. 编排与目标管理把大目标分解成小步；决定下一步、是否继续、何时算完（终止条件）
    ⑥ 没边界G. 安全边界执行动作前做权限 / 风险闸门，隔离不可信内容，高风险动作交人；沙箱化执行
    // 六类能力一览（全文统一字母） 加外部部件 → A 动作执行 B 外部状态/记忆 C 上下文管理 (补 ①②③) 约束其输出 → V 验证反馈 L 编排+目标 G 安全边界 (补 ④⑤⑥)
    一个反直觉点：很多人以为"上下文管理（C）"足够补"记不住（②）"，其实不然——窗口里塞得再聪明，会话一结束还是清零。真正补 ② 的是 B（外部状态 / 记忆）；C 只在"装不下"（③）上帮忙。session（会话）≠ 上下文窗口：压缩（compaction，属 C）是有损且不可逆的，稳健做法是把上下文当成活在窗口之外的对象持久化进 session（只追加事件日志），用类似 getEvents() 的接口按位置切片回读。所以"真正补 ② 的是 B 而非 C"不只是逻辑推论，也是官方实践结论。

    3.2本质：六类能力靠"循环"跑起来

    3.1 推出了六个能力，但它们不是六个并排的零件——把它们串起来运转的是一个循环。agent 的业界共识定义就是循环：一个 LLM 在循环里调用工具、把结果喂回、直到任务完成或满足停止条件。其中 L（编排）地位特殊——它不是并列的一项，而是驱动其余五类的"运行时"。而循环本身很"笨"（业界常形容为 dumb loop），所有智能都在模型里——智能在模型，组织在循环。一圈之内各能力怎么咬合、跨轮怎么接力，第 4 章会画出来。

    04

    Harness 架构 · 两个垂直：编码 & 内容风险

    前三章讲清了 harness 要解决什么、由哪些能力构成、怎么靠循环运转。本章分四步：4.1 静态分层（有哪些层、各补哪条缺陷）→ 4.2 动态链路（一次任务怎么在组件间流动）→ 4.3 编码落地（Claude Code 与 Codex 各选什么）→ 4.4 内容风险落地（同一套层在"强对抗、主观弱反馈、强监管"的内容安全场景下被压成什么形状）。

    4.1Harness 框架图

    第 3 章逐条推出了六类能力（A·B·C·V·L·G）。把它们按"在模型的哪一侧、起什么作用"归拢，就得到三条模块拆分原则——每条对应下面一个模块：

    原则一（喂得准）：模型无记忆、且窗口有"容量"上限，不能全塞——所以要在它前面管好"喂什么"。（→ 模块一 C·B，补缺陷 ②记不住 ③装不下）
    原则二（管得住）：模型的产出不受控、不可信，要在它后面加约束与验证。（→ 模块二 G·V，补缺陷 ④会出错 ⑥没边界）
    原则三（跑得动）：模型只会输出文本、既没手脚也不会自己持续推进——所以要在底层架一层运行时：既替它执行动作、把结果喂回（让它"够得着"外部世界），又编排循环、决定下一步与何时停。（→ 模块三 运行时，补缺陷 ①没手脚 ⑤没方向）

    Harness 骨架总览（顶部用户层入口/交付，中间模型被左右环绕，底部运行时托底）：

    模块一　上下文与记忆 C·B：在模型前，给它"看得准、够得着"的信息——C 决定每轮塞哪些进有限窗口，B 把跨轮要留的状态/记忆存到窗口之外、按需读回。
    模块二　约束与验证 G·V：在模型后，对产出做闸门与裁判。其中 G 的权限闸是分级信任而非一刀切：低风险动作放行、中风险靠沙箱隔离内自主执行（减少打断）、高风险（不可逆/越权）才升级到人工审批。
    模块三　运行时：在底部横向托底，让 agent 稳定地跑——含编排 L（驱动循环）、工具执行 A·E（在沙箱内真正动手）、可观测 O（全程留痕给人看）三层。

    关于 L 的归属：§3.1 按"约束输出"把 L 和 G·V 列在一组（它管目标与终止条件，属纪律层）；这里按它"驱动整个循环"的运行时本质，把它落到底层。同一个 L，两种看法都对——§3.2 说过 L 地位特殊，它既是纪律、也是运行时。

    关于多 agent 编排：它不是新机制，而是把"模型+循环"这套 harness 当成一个零件、再套一层编排，属 L 层在"单 agent 不够用"时的扩展。两种主流骨架：主管-工人（orchestrator-worker），一个主 agent 拆任务、派给子 agent、再汇总（Anthropic 研究系统即用此式）；去中心化交接（handoff），agent 之间直接递交控制权（OpenAI SDK 提供）。派生子 agent 最核心的理由是上下文隔离——每个子 agent 拿独立窗口去啃一个子问题，本质是用"多开窗口"缓解特性 ③（装不下）。

    4.2动态链路：一次任务怎么在组件间流动

    4.1 讲了有哪些层，但没画"血怎么流"；§3.2 说六类能力靠"循环"跑起来——本节就把那个循环展开成一条具体链路：一次任务从入口到交付怎么流动，每个组件职责单一、各司其职，串成一条可重复的环。

    注：本节组件名沿用通用/业界口径（Task Manager、Event Store、Verifier 等），非任何产品的官方组件清单；各组件落到 CC / Codex 的具体取法见 §4.3 对照表。
    另：图 4.1 把工具执行 A·E 归在底部"运行时"（强调静态归属——执行实际发生在运行时环境内），本图把 A·E 画在循环中段（强调它在一轮里的动态时序：模型提动作之后、验证之前）。同一组件、两种视角，不矛盾。

    4.3落地对比：Claude Code / Codex 各选什么

    前两节讲的是不绑产品的通用层与链路。本节落到两个参考实现 Claude Code（CC）与 OpenAI Codex：同样的层，每个各选什么。通用架构是"设计空间"，CC / Codex 各是其中一个具体取点——有几层 CC 刻意做得很薄甚至不做，是它问题范围更窄（单机、可信环境）的合理省略，不是短板。

    L 编排+多agent 决定下一步/是否继续/何时停（＝补⑤：没方向、不会自停）；必要时把任务拆给子 agent（顺带缓解③：各子 agent 独立窗口） 单线程主循环；多 agent 受控派生(各自上下文窗口)、非蜂群，重可调试/透明 单 agent 循环为主，但原生支持多 agent——可派生 agent(角色/嵌套深度/并发默认上限 6)，transfer_to_X_agent 做 handoff
    G 治理/安全 动作落地前做权限/风险把关、隔离不可信输入、高风险交人；安全靠架构而非策略：凭证从不进沙箱（放 vault、由 proxy 代发）。＝补⑥ 权限模式 + hooks + 沙箱；改动须人批准(human-in-the-loop 是结构卡点) 两层：sandbox mode 定技术边界 + approval policy 定何时停下问你；项目分 trusted/untrusted；网络做 DNS/IP 分类
    O 可观测 把过程(轨迹/成本/失败)暴露给人，供监督、调试、审计。✗ 不对应六条——服务于人、非模型固有 终端实时可见 / transcript 会话日志 session-*.jsonl、可审计加载了哪些指令文件；diff 面板复查
    车队级治理 组织/车队层统一管理多 agent 的身份、计费、审计与跨 agent 信任。✗ 不对应六条——组织/车队层需求 基本不做 / 极薄（面向单机可信环境） 相对更全：面向管理员的组织级沙箱与审批约束、企业安全白皮书
    关键节点的 CC / Codex 取法　·　上下文/状态：CC=CLAUDE.md+autocompact；Codex=AGENTS.md 指令链每轮重建　·　工具+沙箱：CC=内建工具+MCP+OS沙箱；Codex=CLI工具+Responses API工具+MCP，原生沙箱默认断网　·　治理闸：CC=权限模式+hooks+改动须人批准；Codex=sandbox mode + approval policy 两层　·　验证：CC=跑测试迭代 + /goal 独立评估器；Codex=建测试/跑检查/复查 + Auto-review,/review　·　多 agent：CC=受控派生(各自上下文窗口)；Codex=spawn agents + transfer_to_X_agent handoff
    1. G 层两家不约而同走"沙箱 + 审批"两层——沙箱定"能做什么"、审批定"何时停下问人"。Anthropic《Managed Agents》进一步指出，光"别把 secret 给模型"不够：若不可信代码与凭证跑在同一沙箱，prompt injection 只需说服模型读自己的环境变量就能拿到 token。结构性修复是让凭证对执行沙箱根本不可达（放 vault、经 proxy 注入）——安全靠架构，不靠策略。
    2. 取舍主要差异：骨架几乎一样（单 agent 循环 + OS 沙箱 + 明文指令文件 + MCP + 沙箱/审批双保险），但 Codex 把"多 agent 派生/handoff""组织级治理"做得更显式、更原生，CC 更克制（多 agent 受控、强调"先做最简单的"）。
    3. 一个注脚：CC 的内核引擎（循环/工具/子 agent/MCP/compaction）来自从它抽取的 Claude Agent SDK，而权限模式/hooks/UI/人批准才是 CC 的产品层取舍——印证"内核可复用、产品层做取舍"。

    4.4换一个非编码垂直：内容风险 作为通用框架的"压力测试"

    CC 与 Codex 都是编码垂直，它们能那样设计，靠三个隐含前提：环境基本合作、可以慢慢跑（长循环）、验证即时（测试通过/失败立刻知道）。把同一套八层框架（A·B·C·V·L·G·E·O）搬到内容风险（UGC 平台的内容安全 / 审核——违规识别、有害与虚假信息、未成年保护、垃圾与诈骗内容）上时，这三个前提同时被打破——所以内容风险不是"换了工具的 CC"，而是检验框架边界的好样本：每层都还在，但有几层被业务特殊性逼着做结构性改造，而非只换选型。

    内容风险不是给通用 harness 多加一个垂直，而是它最严苛的考官之一——把"长循环 / 合作环境 / 即时验证"三条舒适假设逐一拿掉后，框架里真正刚性的是 G 与 O，最先变形的是 V 与 L。

    三条被打破的前提（＝业务特殊性的根）

    1. 环境从"合作"变"对抗"。编码场景里编译器/测试不会骗你；内容风险的输入主动伪装——变体字、谐音、夹带、图文混排、跨模态规避、多轮拆解；而且发布方现在也有 agent：研究显示 agent 能用提示混淆、角色扮演、子目标拆解逐步绕过审核护栏，机器还能批量生成"千人千面"的对抗内容，使审核"既增强了防守、也降低了攻击门槛"。后果：C 层必须默认"输入不可信"，G/V 要按对抗博弈来设计，而非一次性过滤。
    2. 决策有硬实时延迟。发帖/评论/直播/上传要在 秒级甚至毫秒级 完成先审或边审，延迟预算根本不允许跑多轮 LLM 长循环。后果：L 层被拆成"双时钟"——联机轻量先审/边审（小型 guard 模型 + 规则，agent 不在阻塞路径） + 离线异步 agentic 深度研判（复杂案例、团伙关联、申诉复核、政策研判）。这是对"单 agent 长循环"骨架最直接的结构性偏离。
    3. Ground truth 主观、带噪、且随政策漂移。这一点与金融风控不同：金融是"等数周拒付"的纯延迟，内容风险则是边界本身就模糊——违规与否随政策口径、文化语境、平台调性变化，隐式辱骂/暗语/反讽极难判，RLHF 分类器又常过度拦截（over-refusal）误伤正常内容。真标签来自人工复核 + 用户举报 + 申诉结果 + 政策迭代，滞后且彼此不一致。后果：文档最推崇的 V 层"模型之外的客观裁判"在决策时刻基本缺位——只能退而用代理信号（政策匹配、分类器置信度、人审抽样、红队模拟）兜底，真闭环是一条慢回路。这是内容风险对本框架最深的张力所在。
    叠加一层监管刚性：内容处置受欧盟《数字服务法》（DSA）约束——须提供理由说明（statement of reasons）、通知-行动（notice & action）与申诉/庭外争议解决机制、定期透明度报告，超大平台（VLOP）还要做系统性风险评估与独立审计、披露自动化审核的使用；中国侧则有《网络信息内容生态治理规定》（区分违法/不良信息，明确生产者—平台—使用者三类责任）、《生成式人工智能服务管理暂行办法》《AI 生成合成内容标识办法》（强制标识/水印）与算法备案；再叠加"三道防线"治理。于是 human-in-the-loop 在内容风险里是合规要求、不是审慎选项，O（可观测/审计）也从"给工程师看"升级为"给监管 / 申诉用户 / 公众透明度报告看"。

    通用层业务特殊性（为何要变）编码垂直（CC / Codex）内容风险垂直
    A 动作执行 动作影响言论与可见性、部分受监管，误伤有舆论与合规代价 读写文件、跑命令、检索、提 PR 删除 / 限流 / 降权 / 打标 / 年龄限制 / 封禁、加水印或合成标识、向监管或举报方出具理由、上报违法内容
    E 执行环境 样本含高敏 PII 与有害内容，处理本身要合规、要保护审核人员 OS 沙箱、默认断网 沙箱 + 数据脱敏 + 最小权限；敏感样本访问受控、审核员心理保护、模型/政策版本隔离
    B 外部状态 / 记忆 记忆受数据留存与隐私法约束，且要可追溯、可举证 纯文本 CLAUDE.md / AGENTS.md 政策库 / 判例库 / 账号信誉 / 历史处置 / 黑灰样本——落特征平台 + 向量库 + 图数据库，留存与申诉举证期受合规约束
    C 上下文管理 输入可能被对手污染，"喂什么"本身是攻防面 把相关代码/文件塞进窗口 检索适用政策/规则卡（违规定义·判断信号·例外·处置档位）作判定标尺 + 聚合该主体多模态信号（文/图/音/视/行为/关系）；默认"输入不可信"，防变体、夹带、prompt injection 经 UGC 渗入
    V 验证与反馈 真标签主观、带噪、随政策漂移，决策时拿不到客观裁判 跑测试 / lint，即时对错信号 ▲ 结构变形 决策时用代理信号（政策匹配 / 分类器置信 / 人审抽样 / 红队模拟）；真闭环是人审+举报+申诉+政策迭代的慢回路，边界还会随政策漂移
    L 编排 + 多 agent 联机发布/直播有秒级延迟预算，跑不动长循环 单线程主循环（必要时受控派生） ▲ 结构变形 双时钟：联机轻量先审/边审（无 agent 阻塞） + 离线异步多 agent 深度研判
    G 治理 / 安全 受监管，人审是法定卡点（误删须可申诉），非可选 权限模式 + 审批 + 沙箱 ▲ 显著加重 DSA（理由说明 + 申诉 + 透明度报告 + VLOP 审计）、中国生态治理规定 + 标识办法 + 算法备案、三道防线；嵌入式护栏 + 结构化 handover + 强制人监督
    O 可观测 审计与透明度是法定义务，须可解释给非工程读者与被处置用户 终端 transcript / 会话日志，给工程师看 ▲ 显著加重 每个处置自带"理由链 + 证据指针 + 命中政策条款"，供监管/申诉/用户查看；全程留痕 + 透明度报告统计
    怎么读这张表：A·E·B·C 主要是换选型（同一层、换工具/换约束强度），骨架没变；真正发生结构变形的是 V（验证从即时变"主观慢回路"）和 L（单循环裂成双时钟）；而 G 与 O 不是变形、是被监管显著加重并刚性化。这恰好印证 §4.3 的判断——"区别在每层的取舍"，只是内容风险把取舍幅度推到了"改结构"的程度。
    与第 5 章的衔接：内容风险把"对抗输入、主观/延迟验证、强监管"三件事推到极端，正是第 5 章"技术选型与稳定性"里可靠性 / 可观测 / 人机协同等重难点最尖锐的现实样本。
    来源诚实性：本节关于双时钟、慢回路、监管约束、多 agent 研判的判断，基于 DSA 官方文本与中国《网络信息内容生态治理规定》等监管资料、以及 LLM 内容审核与 agent 规避相关论文（见文末来源）；具体落地形态仍会随监管口径与模型能力变化，请勿当作业界定论。
