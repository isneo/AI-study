AI Native 风控审核系统：Artifact-first、Memory 边界与评估闭环

这份文档把今天的讨论重新整理成一套可落地的架构认知：先解释 artifact-first，再落到风控审核的产品与系统原则，最后用一个短视频审核案例串起 runtime、旁路 Agent、memory、artifact、RAG、tool、trace、评估与自进化闭环。

风格：recent-agenda-research 式目录文档 日期：2026-05-27 核心：正式资产是事实源，评估是进化闸门

0. 结论先行

Artifact-first：AI Native 产品不应以“聊天消息”为核心，而应以可保存、可编辑、可验证、可复用、可追责的业务产物为核心。对风控审核而言，这些产物不是漂亮页面，而是审核案卷、证据链、规则集、策略变更包、评估报告、发布记录和申诉记录。

风控指导意义：审核执行 Agent 的核心不是“给一个风险判断”，而是生成可复核的 RiskCase；策略运营 Agent 的核心不是“提一个建议”，而是生成可回测、可审批、可灰度、可回滚的 StrategyChange。两者通过评估和发布闭环连接。

Memory 边界：长期记忆系统仍然是 Agent 变强的核心，但它解决的是学习、适应和经验复用，不解决正式追责。昨天讨论的“runtime 生产情景记忆、消费语义记忆；旁路 Agent 负责情景到语义转化”大方向成立，但要修正：风控里的情景记录大多应落成 case-level artifact；正式语义知识如果会影响生产决策，也必须 artifact 化。Memory 负责沉淀偏好、经验摘要、候选洞察、工具使用经验和工作上下文。

自进化闭环：自进化不能是 memory 自动改规则，而应是：case artifacts → observations → candidate strategy artifacts → evaluation artifacts → acceptance/release artifacts → runtime consumption。评估层是进化闸门，不通过评估不能影响下一次生产审核。

Artifact

正式资产

有 owner、状态、版本、验收、审计、追责价值。例：RiskCase、RuleSet。

Memory

能力飞轮

帮助 Agent 复用经验、适应业务、减少重复探索，但不能直接作为封禁、限流、策略发布依据。

Runtime

处理当前案子

读取正式策略、调用工具、生成案卷和证据，不直接改生产规则。

Eval

进化闸门

所有候选策略必须经过黄金集、困难样本、回测、灰度和线上监控。

第一部分 · 概念与原则：为什么风控系统要 artifact-first

1. 什么是 Artifact-first

1.1 定义与类比

一句话定义：Artifact-first 是一种 AI Native 产品和系统设计原则：把“AI 与用户共同维护的正式产物”放在中心，而不是把聊天记录放在中心。

传统软件是 UI-first：用户通过按钮、表单、流程操作软件。早期 AI 产品多是 chat-first：用户问一句，AI 回一句。Artifact-first 则是：对话只是控制层，真正的价值沉淀在能持续演化的产物里。

形态核心对象用户感受风控审核里的对应物
UI-first页面、表单、按钮我操作系统传统审核后台、规则配置台
Chat-first对话消息我向 AI 问问题问“这个内容违规吗”，AI 回复一段解释
Artifact-first可演化业务产物我和 AI 一起推进一个可交付对象审核案卷、证据链、策略变更包、评估报告
类比：chat-first 像请教顾问，顾问答得再好，价值也散落在谈话里；artifact-first 像和同事共用一份卷宗、一张白板、一个代码仓库，最终沉淀的是可继续工作、可被别人接手、可被审计的东西。

1.2 不是所有产物都是 artifact

Artifact 概念必须收窄，否则“所有被存下来的东西”都会被叫作 artifact，边界就失效。一个对象适合升级为 artifact，通常要满足多数条件：会被正式消费、有生命周期、需要验收、需要版本/追溯、会影响业务决策。

对象是否建议作为 artifact原因例子
审核案卷是要复核、申诉、质检、追责、沉淀。RiskCase@case_10086
规则集 / 策略包是影响生产决策，必须版本化、审批、回滚。RuleSet.sexual@v18、StrategyPack@v32
评估报告是决定候选策略能否上线，是发布依据。EvalReport@2026_0527_01
工具原始返回通常不是只是过程材料；被采纳进证据链后才成为案卷的一部分。OCR raw output、模型 raw score
Trace / Log通常不是服务复现、排查、性能分析，不是业务判罚理由。run_abc.tool_calls
Memory insight通常不是是候选洞察；要影响策略必须先转成候选 artifact 并评估。“政治背景 OCR 误伤偏高”
RAG index不是是可重建的检索索引，不是事实源。规则向量库、案例向量库

2. 在风控审核领域的指导意义

风控审核有一个天然特点：每次判断都可能影响用户权益、业务损失、平台合规和舆情风险。因此它不能只追求“AI 判断得快”，还必须能回答：为什么这么判、依据哪版策略、证据是什么、谁复核过、后来申诉是否成立、是否能复现当时决策。

2.1 两类 Agent 与治理层

执行面

审核执行 Agent

像一线审查员 + 办案卷宗。它处理当前内容，收集证据，引用规则，形成可复核决策草案。

控制面

策略运营 Agent

像策略分析师 + 作战地图。它看大量案卷，发现模式，提出候选策略，推动评估和发布。

治理面

评估与验收体系

像质检、法务、发布委员会。它决定候选变化能否进入生产，并保留回滚和审计依据。

2.2 设计原则

原则 1：不要 agent-first，要 artifact-first

不要先问“我要几个 Agent”，先问“哪些业务产物必须被复核、追责、复用”。风控里通常是案卷、证据链、规则集、策略变更包、评估报告、发布记录。

原则 2：AI 给结论必须给证据结构

“高风险，建议封禁”不是合格输出。合格输出必须有风险类型、证据、规则版本、模型版本、处置矩阵、人工复核状态。

原则 3：建议与执行分离

AI 可以建议拦截、限流或复核，但高损害动作应经过策略引擎、权限系统、审批流或人工复核。

原则 4：所有生产依据都要版本化

每个判断都要能追溯：当时用了哪个规则、模型、阈值、prompt、策略包、评估报告。

原则 5：策略上线前必须评估

没有 backtest、golden set、slice analysis、灰度监控的策略建议只是观点，不是生产资产。

原则 6：人工反馈是一等信号

人工改判、申诉成功、质检失败、投诉、真实损失都不是附属备注，而是旁路 Agent 发现模式和构建评估集的关键输入。

3. 为什么这么设计：相关概念的区别、关系、边界

行业里常把“知识库、RAG、memory、案例库、规则库”混着说，是因为它们可以重叠在同一份资料上，但它们描述的是不同维度：对象是什么、怎么检索、是否记住、能否执行、是否正式依据。

3.1 概念边界表：带具体例子

概念它回答的问题风控例子和其他概念的关系边界判断
Artifact 这个东西是否要被业务正式消费、验收、版本化、追责？ RiskCase、RuleSet、StrategyChange、EvalReport。 是事实源；RAG 可以检索它，tool 可以读写它，memory 可以从它派生摘要。 正式依据必须 artifact 化。
Memory 系统长期记住什么以便未来更懂上下文？ “社区业务线更重视召回率”“最近软色情+私域引流上升”“上次评审要求政治背景误伤单独看”。 是派生层。可以由 artifact/trace/反馈总结而来，也可提示 Agent 关注重点。 不能单独作为处罚或发布依据。
RAG 运行时怎么把相关资料找出来？ 相似案例检索、规则片段检索、政策说明检索、历史申诉检索。 是方法，不是资产类型。RAG corpus 里的对象可能是 artifact，也可能是 memory。 检索结果必须回指 source artifact / version。
Tool 系统能查询、计算、判断、执行什么？ OCR、ASR、安全分类模型、规则引擎、画像查询、回测引擎、策略发布工具。 工具输出是过程材料；写入案卷证据链后，才成为 artifact 的一部分。 会产生副作用的 tool 必须受权限和 gate 控制。
Trace 系统当时怎么跑的？ 调用了哪个模型、上下文灌入了哪些片段、耗时、重试、原始返回。 服务可观测和复现；可被旁路 Agent 分析，但默认不是业务理由。 trace 支持审计，不等于案卷证据。
Observation 系统观察到什么候选信号？ “政治敏感 OCR 背景误伤偏高”“某规则命中后人工驳回率上升”。 可能从 case、trace、quality review 中提取；经过验证后可升级为 RiskPattern 或 RuleProposal。 未评估前不是正式知识。
Evaluation 候选变化是否真的更好？ EvalSpec、EvalDataset、EvalRun、EvalReport、AcceptanceRecord。 连接 observation 与正式策略发布，是自进化的闸门。 没有评估的进化只是自信，不是治理。

3.2 为什么必须这么分

核心风险：如果把规则、案例、经验、索引、trace 都塞进一个“memory”，短期看系统简单，长期会在审计、申诉、误杀追责、策略回滚时还债。风控系统需要的是单一事实源 + 可重建索引 + 派生记忆 + 受控工具。
第二部分 · 运行机制：长期记忆、Runtime、旁路 Agent、评估闭环怎么串起来

4. 长期记忆系统为什么仍是核心

前面强调 artifact，是为了避免风控系统把正式规则、案例和策略放进一个不可追责的“记忆黑箱”。但这不等于 memory 不重要。恰恰相反：长期记忆系统是 Agent 能持续变强的核心，它让 Agent 不再每次从零推理，而是能复用历史经验、团队偏好、风险模式、工具路径和失败教训。

修正后的判断：Artifact 是可信运行的核心，Memory 是能力进化的核心。Artifact 解决“正式依据是什么、能不能追责”；Memory 解决“Agent 如何从历史中学习、下次更快更准、更少重复犯错”。

4.1 Memory 是能力层，不是事实源

长期记忆系统在风控里不应该被设计成“另一个规则库”或“另一个案例库”。它更像一层经验操作系统：从正式 artifact、trace、人工反馈和申诉后验中提炼出可复用信号，再在下一次任务中按需召回。

问题应该由谁回答例子原因
这个内容为什么被限流？ Artifact RiskCase@case_10086 引用 RuleSet.sexual@v18 和证据包。 这是正式解释、申诉和审计问题。
下次遇到类似内容，Agent 应该更注意什么？ Memory “近期软色情+私域引流误放上升，重点检查加群、福利、私聊话术。” 这是经验复用和注意力引导问题。
这个策略能不能上线？ Evaluation Artifact EvalReport 比较 baseline 与 candidate 的召回、误杀、申诉、人工压力。 这是验证与验收问题。
Agent 应该怎么查证更高效？ Procedural Memory “政治背景类先跑主体性检测，再检索误伤样本，再进专项复核。” 这是工具路径和流程经验问题。

4.2 风控里的四类长期记忆

Episodic Memory

情景记忆：发生过什么

不是完整案卷本体，而是对历史案卷的可召回摘要。例如：“上周政治背景误伤集中在低清视频 OCR 片段，源案卷为 case_9032、case_10086、case_10112。”它必须回指 case artifact。

Semantic Memory

语义记忆：一般性风险模式

例如：“软色情+私域引流常见链路是低风险互动 -> 暗示福利 -> 加群/私聊转化。”它能指导 Agent 关注模式，但要影响生产策略，必须转成 RiskPattern、RuleProposal 或 RuleSet。

Procedural Memory

程序记忆：怎么做事

例如：“多风险叠加内容先拆分证据，再分别跑色情/政治/引流规则，最后由 ActionMatrix 合并处置。”这类记忆提升 Agent 执行效率；正式 SOP 仍应 artifact 化。

Preference / Policy Context

组织偏好：业务怎么取舍

例如：“社区业务线边界内容优先限流+人审，不直接删除；政治背景类误伤要单独看新用户和低清 OCR slice。”它影响排序、解释和升级策略，但不能替代规则。

4.3 Memory 生产、检索与评估闭环

Memory 机制风控设计要求否则会出什么问题
写入过滤只有来自案卷、trace、人审、申诉、评估报告的候选记忆才能进入提炼；单个 case 的猜测不能直接持久化。一次误判被写成长期经验，后续持续污染判断。
来源引用每条 memory 必须有 source_artifacts、生成时间、置信度、适用范围和 TTL。Agent 引用了“看似熟悉”的经验，却无法解释来自哪里。
冲突消解新旧经验冲突时，按评估结果、时间、业务线、风险类型降权或失效。过期策略和新策略同时被召回，导致输出摇摆。
检索策略根据任务类型召回不同 memory：审核 runtime 召回风险模式和工具路径；策略旁路召回误杀/漏杀经验；评估 Agent 召回历史退化点。无关记忆挤占上下文，带来噪声。
Memory 评估比较开启/关闭 memory 后的人工接受率、误杀/漏杀、处理时长、误引用率、升级人审准确率。系统只是在“记更多”，但不知道是否真的变好。

5. Runtime 与旁路 Agent 的记忆边界：判断与修正

5.1 对原判断的修正

你昨天的三点判断大方向是对的，但纳入 artifact 后需要收窄“memory”的职责。

原判断是否成立修正后说法原因
Agent runtime 做两件事：情景记忆生产、语义记忆消费。 基本成立 Runtime 生产 case-level artifacts 与 episodic traces，消费正式 artifacts 与 semantic context。 风控的“情景记忆”很多是正式案卷、证据、处置结果，不能只放 memory。
Agent runtime 不负责语义记忆生产。 成立，但需补充 Runtime 可产生 observation candidate，但不能直接写入正式语义规则或生产策略。 运行时面对的是单个 case，容易过拟合；正式知识需要跨样本、评估、验收。
旁路 Agent 负责情景记忆提炼转化和语义记忆。 成立，但要加 gate 旁路 Agent 从案卷/trace/反馈中提炼 insight，生成候选 artifact，经评估和审批后才发布。 旁路 Agent 不是“总结后自动改规则”，而是候选变化生产者。

5.2 新模型：Artifact-first Memory

最终理解：Runtime 像一线审核员，负责把当前案子处理清楚并留下完整卷宗；旁路 Agent 像策略分析师，负责从大量卷宗和反馈里发现模式；Evaluation 像质检和发布委员会，决定候选变化能不能进入生产。

6. 详实示例：用户上传图文/短视频，按色情、政治等规则审核

场景：用户上传一条 15 秒短视频，画面有人物出镜，背景疑似政治标语，文案和口播含“加群看更多”，整体有成人暗示营销倾向。系统需要按色情、政治、广告引流、未成年人等风险类型审核。

6.1 审核执行链路：读什么、调什么、写什么

步骤Agent 动作调用工具读取 artifact / memory写入 artifact / trace关键边界
1. Intake 创建审核案卷，冻结上传对象快照。 媒体元数据读取、文件指纹、上传上下文查询。 RiskTaxonomy@v8：色情、政治、引流、未成年人等风险分类。 RiskCase@case_10086：content_type、upload_time、scene、user_id hash、status。 一开始就创建 case artifact，后续所有证据都挂到案卷上。
2. 内容解析 提取视频可审查信号。 抽帧、OCR、ASR、封面识别、音频分类、图像安全模型、文本安全模型。 读取模型配置 artifact：ModelRegistry.img_safe@2026_05、PromptTemplate.review@v6。 EvidenceBundle@case_10086/e01：关键帧、OCR 文本、ASR 文本、模型分数、模型版本。 工具原始返回先是 trace；被采纳后才进入 EvidenceBundle。
3. 检索上下文 检索相似案例和相关规则说明。 similar_case_search、policy_search、reranker。 RAG index 指向 RiskCase、RuleSet、PolicyDoc；memory 提醒“近期软色情+私域引流上升”。 RetrievedContextRecord：source_id、version、slice、score。 RAG 结果必须回指 artifact_id/version，不能只保存一段无来源文本。

5. 决策草案 生成结构化风险结论和处置建议。 LLM decision composer、schema validator、risk gate。 读取处置约束：高风险封禁需人审；政治类不确定进入专项复核。 DecisionDraft：risk_types、risk_level、recommended_action、basis、required_review_queue。 AI 输出是建议草案；高损害动作由审批/规则/人审 gate 决定。

第三部分 · 行业先进经验：只用公开资料做校验

7. 行业先进经验与论证

公开资料很少直接使用“artifact-first memory/RAG”这套说法，但成熟系统公开展示了相同的治理逻辑：政策/规则、casework、人审、自动化、申诉、审计、QA、评估和透明度指标被放在一个闭环里。这些公开证据足以校验我们的设计方向。

Agent Memory 研究：长期记忆让 Agent 从无状态生成器变成可适应系统。

近期综述把 memory 定义为跨交互持久化、组织并选择性召回信息的能力，并强调写入过滤、冲突处理、延迟预算和隐私治理等工程问题。对应我们设计里的 memory 写入过滤、source 引用、TTL、冲突消解和 memory eval。来源：Memory for Autonomous LLM Agents。

Cinder：Trust & Safety 的 system of record。

公开资料称其把 policy、casework、auditability、analytics 放在同一操作面，并记录 policy update、enforcement decision、agent action；还用同一批 labeled data benchmark human reviewers 与 AI agents。对应我们设计里的 artifact store、casework、audit trail、evaluation/QA。来源：Cinder Platform。

YouTube：自动系统 + 人审 + 申诉复审。

YouTube 公开说明自动系统利用历史人工审核数据识别潜在违规；高置信时可自动决策，多数情况下会标记给训练过的人审；申诉后由人工逐案复审。对应我们设计里的 runtime、human review、appeal feedback、评估数据飞轮。来源：How YouTube reviews content。

TikTok：每条上传视频自动审核，不确定进入人工。

TikTok 公开说明通过技术和审核团队执行社区规则；每个上传视频由自动系统审核，高置信违规会移除，置信不足时进入人工审核；也支持举报和申诉/救济。对应我们设计里的工具层、risk gate、人工复核和申诉记录。来源：TikTok Content Moderation。

NIST AI RMF：Govern / Map / Measure / Manage。

NIST AI RMF 是 AI 风险管理公开框架，强调在 AI 系统设计、开发、使用和评估中纳入可信风险管理。我们的“先评估再发布”对应 Measure/Manage，策略治理和责任归属对应 Govern。来源：NIST AI RMF。

OpenTelemetry：trace/log/metric 是可观测性信号。

OpenTelemetry 将 traces、metrics、logs 等定义为观测系统活动的 signals。它支持我们把 trace 定位为工程复现和观测层，而不是业务判罚依据。来源：OpenTelemetry Signals。

W3C PROV：来源追溯可标准化表达。

W3C PROV 提供 provenance 数据模型、约束和序列化，用于表达实体、活动和来源关系。风控案卷应能追溯模型、规则、工具、人工动作和版本。来源：W3C PROV Overview。

OWASP LLM Top 10：LLM 应用存在 prompt injection、data poisoning 等风险。

OWASP 把 prompt injection、training data poisoning、insecure output handling 等列为 LLM 应用关键风险。这支持我们“不让 memory/RAG 直接改生产策略”“工具副作用要受 gate 控制”的边界。来源：OWASP Top 10 for LLM Applications。

行业校验结论：成熟实践并不是“AI 自己判、自己学、自己改规则”，而是自动化识别、人审、case 管理、政策/规则治理、QA/评估、申诉、审计和透明度一起构成闭环。我们讨论的 artifact-first 方案，本质上是把这套成熟风控治理能力用 AI Native 架构语言重新表达。
