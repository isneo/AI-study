名词解释 · 把技术语言垫清楚
贯穿全文 · 编排与 Agent 基础概念

多 Agent 的名词常被混在一起，其实分属两个层次：宏观看多个 Agent 怎么编排，微观看单个 Agent 怎么运转。先建立这张全景，再看单个术语就不会乱：

# 宏观 · 多个 Agent 怎么编排 Orchestrator-Worker / Sequential / Fan-out / Handoff ... ↓ 其中每个 Agent 内部 # 微观 · 单个 Agent 怎么运转 Agent Loop └─ ReAct（推理 → 行动 → 观察，循环） └─ Tool Calling（调用具体工具）
核心模式 · 本系统采用
Orchestrator-Worker（编排者-执行者）— 一个调度者负责拆任务、派活、汇总、仲裁，多个执行者各自干一件专门的事；worker 之间不直接对话，都经调度者中转。优点是有单一问责点、流程可追踪，是生产环境最常用的模式。本系统的"一主三辅"即此模式——调度 Agent 是 orchestrator，识别 / 研判 / 处置是三个 worker。
其他编排模式 · 平级的选择
Sequential Pipeline（顺序流水线）— Agent A→B→C 固定顺序，每步依赖上一步输出。比 orchestrator-worker 更死板，无中央动态调度。
Fan-out / Fan-in（扇出 / 扇入，并发）— 一个任务同时分给多个 worker 并行干，再收拢汇总；适合子任务互不依赖。风控里"多个 Agent 同时从不同策略维度查同一条内容"即此模式。
Handoff / Swarm（交接 / 蜂群）— 无中央协调者，每个 Agent 自行判断该不该把任务转给更合适的 Agent；更灵活但更难调试。
Hierarchical（分层）— orchestrator 下的 worker 自身又是小 orchestrator，层层嵌套，处理超复杂任务。
协调机制 · 编排里的零件
Router / Routing（路由）— 决定"任务该交给哪个 Agent"的分发逻辑；orchestrator 内部通常含一个 router。
Handoff（交接）— 一个 Agent 把控制权和上下文移交给另一个 Agent 的动作。
Shared State（共享状态）— 多个 Agent 间传递、共享的数据（如已查到的证据）；由 orchestrator 维护。
单 Agent 内部 · 更底层的运转
ReAct（推理-行动循环）— 单个 Agent 在"推理 → 行动 → 观察"间循环，动态决定下一步，适合解法路径不固定、要边查边判的任务。注意层级：它是"一个 Agent 内部怎么思考"，与 orchestrator-worker（多个 Agent 怎么配合）不在同一层，可叠加使用。
Agent Loop（智能体循环）— Agent "思考-行动-再思考"直到任务完成的运行循环；ReAct 是其中一种实现。
Tool Calling（工具调用）— Agent 调用外部工具（查库、调模型、搜索）的能力；worker 干活主要靠它，ReAct 的每一次"行动"就是一次 tool calling。

用本系统串一遍：宏观上是 orchestrator-worker（调度 + 一主三辅）；其中识别 / 研判 worker 内部可跑 ReAct 循环（自主决定先查评论还是先查行为数据）；ReAct 的每次"行动"就是一次 tool calling（调用某个数据源）。三层各司其职，不冲突。

编排类 · Multi-Agent Orchestration
关联正文 · 03 系统骨架 / harness 基座
Q1为什么用多个 Agent，而不是一个大 Agent 全干？
职责隔离——识别要快要全、研判要准要可解释、处置要严要可审计，三种诉求塞进一个 Agent 会互相拖累；拆开后每个 Agent 上下文更干净、可独立优化和验证。但我不盲目堆 Agent：多 Agent 不是越多越强，它会带来额外的编排成本、上下文传递成本和分歧仲裁成本，还会拉高延迟、增加调试难度。所以我只在三类场景才拆——职责目标不同、权限边界不同、评估指标不同。主链路拆成识别 / 研判 / 处置，正是因为它们分别追求召回、定性准确和动作安全，不是为了显得复杂。 〔Anthropic 踩坑印证〕这个"不盲目堆 Agent"的判断，和 Anthropic 做 Claude Code 的踩坑一致：他们最初设想让 Claude 把任务拆成多个子 Agent 并发执行，实际发现 Claude 并不擅长并发拆解，子 Agent 真正有用的反而是"探索类任务"（让它去消耗上下文读资料、只回传结论）；他们甚至主动下架工具——给模型少一点选择、给上下文少一点干扰。多 Agent 和工具都不是越多越好，关键看是否真有职责隔离的收益。（来自 Anthropic 公开分享，非一手经验。） 思路：这题答的是"克制地用多 Agent"，呼应 04"看代价不看能力"。拆是为职责干净，不是为数量——这句可作为这块的金句。
Q2你的主链路具体怎么编排？一主三辅怎么配合？
orchestrator-worker 模式（业界生产环境最常用的多 Agent 编排）：一个调度 Agent做编排协调，下挂三个 worker——识别 Agent（看出有什么信号）、研判 Agent（定性、判风险等级）、处置 Agent（执行动作）。 待补：调度 Agent 怎么协调三辅？ 需你决策：是①严格顺序（识别→研判→处置一条线走完，调度只串联），还是②调度 Agent 动态决定调用谁、调几次？两种讲法不同——顺序更简单可控，动态更灵活但难追踪。告诉我你的真实做法。
Q3三个 Agent 判断有分歧时怎么办？谁说了算？
多 Agent 编排公认最大的挑战就是"Agent 分歧时谁裁决"——比如识别说涉政、研判说误报。 待补：你的分歧仲裁机制？ 需你决策：分歧/低置信时是①升级人审（最稳，勾回 harness 授信和人审兜底），还是②调度 Agent 按规则拍板，还是③两者结合（低风险调度拍板、高风险升级人审）？这题是 P8 加分位，因为它连回了 harness 授信。告诉我真实机制，我帮你串成"分歧→升级"的闭环表达。
Q4为什么是一主三辅、不多不少？
待补：你拆成四个的真实考虑 需你决策：这题考的是"你的拆分判断"。我的推测是——识别/研判/处置对应风控的三个天然环节（看信号→定性→执行），调度负责串联和仲裁，刚好覆盖、不冗余。但"为什么不把研判合进识别"这种取舍是你的真实决策，你补一句真实理由，比我推测有说服力。
Q5为什么用会推理的 Agent（ReAct），而不是固定 workflow？
因为风控的取证路径是不固定的，没法事先写死成流程图。workflow 适合"步骤已知、有唯一正确路径"的任务；而风控判一条内容，路径取决于边查边发现的线索。
举个真实场景：一条视频被举报，光看视频内容判不准——还要看发布者的行为数据。过去人工取证是"先去评论区找线索 → 验证 → 不够再回到视频里找 → 验证 → 直到形成结论"，下一步查哪里，取决于上一步查到了什么。这是典型的 ReAct 模式（推理 → 行动 → 观察，再循环），不是固定 workflow 能覆盖的——workflow 把路径写死了，遇到"该去哪找证据因案而异"就僵住。 我做的，是把这套"边推理边取证"的过程交给 Agent：Agent 自主决定下一步调用什么工具、查哪个数据源，像资深审核员一样顺着线索找，直到证据链完整。 〔Anthropic 踩坑印证〕这条判断 Anthropic 用 Claude Code 验证过：他们早期接触的客户多用 workflow 架构（有客户串联了 50 个 prompt），踩了两个坑——只能处理事先编码好的场景、中途出意外不会自愈反而继续跑出垃圾，最终转向 Agent 架构（让模型在循环里自己判断何时完成）。风控取证比写代码更"路径不可预知"，更要避免把链路写死成 workflow。（来自 Anthropic 公开分享，非一手经验。） 思路：基于你的视频举报案例，查证确认了 ReAct vs workflow 术语准确（ReAct=路径依赖中间结果的动态循环，workflow=预定义固定步骤）。这是编排块的"灵魂论证"——解释你的多 Agent 为什么是"会推理的 Agent"而非"自动化流水线"。需你确认：识别/研判 Agent 内部是否真跑 ReAct 多轮循环（自主取证），还是目前固定几步？据实调整。
harness 类 · 怎么把 Agent 管控住
关联正文 · 03 系统骨架（基座二·运行时）/ 04 演进与取舍（约束分级）

harness 是包裹 Agent 的运行时，做三件事：编排、约束、授信。编排见前面 Q1–Q5；这里讲最核心的约束与授信——也就是"怎么让 Agent 在低风险动作上放手、在高风险动作上被死死兜住"。这是我在策略运营试点线上投入最大的一块。

Q6harness 的"约束"具体怎么做？凭什么敢让 Agent 动手？
约束不是一道墙，是三层梯子，一层比一层往后兜：
层拦什么怎么拦
事前 · 治理该不该做权限（RBAC）· 审批分级 · 合规扫描（歧视变量 / 脱敏）· 影响半径 · 限速
事中 · 验证做得对不对5 道关：AST → 单测 → 影子回测 → 冲突检测 → 业务影响；失败结构化回灌、触发自愈循环
事后 · 观测出事能不能追多维度大盘 + 5W 不可篡改审计（谁 / 何时 / 改了什么 / 为什么 / 影响谁）
再加两道工程隔离：沙箱——Agent 产出的规则先在沙箱里只解析不执行（parse-time 校验，杜绝危险表达式）；灰度通道——Shadow（全量不生效）→ Canary（1/5/20%）→ Full，任何阶段触达回滚条件立即降回 shadow。 关键区分：回测 ≠ 测试——回测看业务影响（这条规则上线会误伤多少），测试看代码正确（语法对不对）。 落地状态（诚实交底）：治理 / 验证 / 观测三层为骨架（最小可用版）已落地；工具执行沙箱（7 条硬约束）设计已写、代码未起；灰度通道规划中。生产端到端闭环依赖沙箱 + 灰度补齐。 思路：本块素材来自真实试点项目。面试时先讲"三层梯子"心智模型（事前拦 / 事中抓 / 事后追），被深挖某一层再展开。诚实标注未落地项——附件原则"roadmap 当 achievement 讲会崩"。
Q7"授信"怎么给？凭什么决定哪个动作能让 Agent 自决、哪个必须人审？
授信的标尺是动作的可逆性 × 影响范围——约束强度随动作风险递增：
分析 / 查询（无副作用）→ 放手　|　产规则草稿（可丢弃）→ 放手　|　影子回测（不生效）→ 放手　|　发布生效（不可逆 · 影响海量分发）→ 必须人审
所以在试点线上，Agent 当前真正放开的是"分析 → 规则草稿"段（走偏的代价只是人工弃用一次输出）；"发布到生产"这种不可逆的高风险动作仍走人审、还在补齐。这不是保守，是把放权对齐到代价——低风险充分自治、高风险死守人审。等沙箱和灰度把"发布"这段的代价也兜住了，再逐步抬升放权。（这是"按什么标尺分级"；至于"怎么验证才敢往上抬一档"，见评估类 Q9。） 思路：呼应 04"约束随动作风险递增、看代价不看能力"。这里把"为什么发布还没让 Agent 自主做"讲透——是代价决定的，不是技术不行，且诚实说明在补。

关联正文 · 03 系统骨架 / 04 演进与取舍
Q8怎么证明 Agent 比传统规则系统更好？
不靠主观感觉，靠对比评估：同一批 golden set（标注好的真实案例集）上，比 Agent 和规则系统的 precision / recall / 误杀率 / 漏放率；线上用 shadow-run（Agent 跟跑但不生效）对比判定差异；再灰度 A/B。 待补：你实际用的核心指标和真实数值 需你决策：这块通用方法论我写好了，但"你实际看哪几个指标、当时的真实数字、怎么定的放权阈值"必须你来——这正是面试官最爱深挖、也最容易问穿的地方。没有真实数字就别写具体数字，宁可只讲方法。
Q9上线前怎么决定某类动作能不能放权给 Agent？
分级验证：离线 golden set 跑通 → 线上 shadow-run 比对人工判定一致率 → 小流量灰度盯指标 → 达标才升一档放权。放权是"挣来的"——用证据，不是拍脑袋。 思路：这题呼应 04"先在赔得起的地方验证再扩张"。可直接引用 04 的演进逻辑，附录这里给出具体的验证四步。
Q10怎么证明系统"真的变好了"？只看准确率够吗？
不够——单一准确率高，掩盖不了误杀上升、飞轮没转、线上扛不住、出事查不清。我把指标分成六层，共同判断"能不能继续放权"：
指标层回答的问题典型指标
业务效果风险有没有真变少拦截率 · 漏放率 · 误杀率 · 申诉成功率
Agent 判断判得准不准precision · recall · 等级准确率 · 证据命中率
放权治理能不能扩大自治各档自决率 · 升级率 · 人审推翻率
策略迭代飞轮转没转发现到上线耗时 · 回归集通过率
系统工程线上扛不扛得住P95/P99 延迟 · 成本 · 工具成功率 · 超时率
安全审计出事能不能兜住case replay 覆盖率 · 版本可追溯率 · 回滚耗时
六层不是并列，是一套健康度框架：业务层证明有价值、Agent 层证明判得准、治理层证明能放权、迭代层证明飞轮转、工程层证明扛得住、审计层证明兜得住。任何一层不达标，都不该继续放权。 思路：这张分层表整理自外部评审建议，已用业界审核 KPI 框架（准确率/运营/质量/业务结果多维）佐证，归类站得住。面试时不必背全表，记住"六层 = 结果·能力·权限·成长·稳定·底线"即可。真实数值需你按项目填，没有就只讲框架、不编数字。
Q11人调了一次意图，怎么知道它真的达成了目的？
这是飞轮"微观"那一圈——每次调意图都要能验证"这次到底有没有用"，否则就是改完不管的开环。一条完整的验证链：
绑定目标+护栏指标 → 事前历史流量预判 → 灰度（Shadow→Canary→Full）实测"调前 vs 调后"的差 → 自动比对：达标放量 / 护栏破线回滚
关键不是看"现在拦截率多少"，而是看调这次意图前后的差（拦截 75%→80% 即 +5，代价误杀 +0.5）——只有对比能证明变化是这次意图带来的，这就是 A/B / champion-challenger 的内核。 两个判断最容易被追问穿，我的答案是：
  • 灰色地带怎么权衡目标达成但护栏轻微越界（误杀涨了但拦截收益更大）／ 用代价量化裁决——把"漏过的代价"和"误杀的代价"折算成可比的量（风险分/影响人数/申诉成本），不拍脑袋
  • 底线兜底高危品类（涉未成年 / 暴恐等）／ 护栏一票否决，不允许任何"收益更大"的权衡，宁可误杀也不放过
宏观整体效果看 Q10（六层指标 + 大盘趋势），防脏数据带歪看 Q12（反馈治理）；这三个 Q 合起来，就是飞轮"怎么转、转得好不好、不转歪"的完整回答。 落地状态（诚实交底）：这套"意图→效果验证"的闭环，目前设计在推进中——影子回测有雏形、灰度通道规划中，完整的"自动比对预期 vs 实际 + 自动回滚"还在建设。讲的是想清楚的设计，不是已跑通的产线。 思路：本块沉淀自"意图效果评估"专项讨论。代价量化（用钱/风险表达权衡）和高危一票否决，都是查证支撑的业界做法（OEC+护栏、非劣性检验）。和 Q9 区分：Q9 是"某类动作能否放权给 Agent"（放权决策），本 Q 是"单次意图变更有没有达成"（效果验证）。
Q12数据飞轮怎么防止被"脏反馈"带歪？（防投毒）
风控系统最怕的不是没反馈，而是把脏反馈当成真相吸收。一条"用户申诉说我没违规"，可能是真误杀、可能是违规者狡辩、可能是黑产组织化攻击。全自动喂回飞轮，Agent 会学会放过违规、策略边界被恶意拉宽、越迭代越松。

所以反馈进飞轮前必须过一道治理层——它不只是"洗数据"（去重、清噪、修标签），更要判断这条反馈有没有资格成为系统学习的"真相"：
反馈 → 来源识别 → 可信度打分 → 去重/聚类 → 冲突检测 → 对抗投毒识别 → 人审仲裁 → 分流入库
最终按来源可信度分流，而非一锅烩：
反馈来源可信度处理方式
高级审核员仲裁 / 多人一致复核高可进案例库、eval 候选集
用户申诉成功中高聚合 + 抽检后入
普通用户申诉中低不直接进，先判真伪
被处罚账号集中申诉低隔离观察，疑组织化对抗

Q13"策略意图"具体长什么样？怎么从自然语言变成可执行？
策略意图是一份结构化的 YAML 规则对象——人类可读、机器可解析。它不是一段 prompt，而是经过静态校验、冲突检测、权限映射后，再被编译进 Agent instructions、工具权限、处置 guardrails 和回归评估集。它把"什么算违规、看哪些信号、分几档、各档怎么处置、要什么证据"一次性写清，让策略从"写在脑子里和文档里"变成"可下发、可校验、可灰度、可回滚的对象"。以色情低俗品类为例：
规则编号: P-SEX-001 品类: 色情低俗 # 1 · 违规定义 违规定义: 以挑逗、刺激、引诱为主要目的，描绘或暗示性行为、 性器官、裸露身体敏感部位的图文 / 音视频内容。 # 2 · 判断信号（内容 / 行为 / 关系 三维） 判断信号: 内容信号: [明确性行为描述, 裸露生殖器官, 含性挑逗动作视频] 行为信号: [同用户 24h 发 5 条以上类似内容, 头像简介含性暗示] 关系信号: [关注/被关注集中在同类发布者, 评论区高频"加我""私聊"引流] # 3 · 例外情形（不视为违规，但仍需分级处理） 例外情形: [医学科普, 认证机构的性教育, 严肃人体艺术, 母婴哺乳, 新闻事实性描述] # 注：即便例外，未成年人可见环境下仍需打"成人内容"标签 # 4 · 处置档位 处置档位: L1: { 特征: 软色情擦边·暗示不明确, 动作: 内容降权·不可见于推荐流 } L2: { 特征: 明确性暗示·软色情图文, 动作: 删除内容·用户警告 } L3: { 特征: 露骨色情内容, 动作: 删除·7 天禁言 } L4: { 特征: 露骨色情且批量传播·引流, 动作: 永久封号·设备拉黑 } L5: { 特征: 涉未成年人色情内容, 动作: 立即封号·上报执法机关 } # 5 · 证据要求（与放权强度挂钩） 证据要求: L1-L3: Agent 可直接处置，需保留完整推理链抽检 L4: 需 Agent 置信度 ≥ 0.9 且至少 2 类信号，否则升级人工 L5: 必须人工复核 + 法务备案，Agent 仅做识别冻结
说明：以上是脱敏的示例规则，用于演示策略意图的写法，不是任何具体平台的真实策略；实际落地会依据《网络信息内容生态治理规定》等法规和业务场景大量调整。

这份模板里藏着三个设计判断：其一，信号分内容 / 行为 / 关系三维——单看内容会漏判，结合行为（批量发布）和关系（引流话术、团伙关注）才能识别隐蔽违规，这正是 Agent 比纯规则强的地方；其二，证据要求与放权强度挂钩——L1-L3 Agent 自决、L4 要双信号高置信、L5 锁死人工，把"放权边界"直接编进了策略；其三，例外情形显式声明——避免医学、艺术、母婴等正常内容被误伤。 〔Anthropic 踩坑印证〕"不是一段 prompt、而是结构化对象"这点也有教训佐证：Anthropic 提到有客户把几十页的 SOP 直接塞进 system prompt，结果把模型搞蒙了。指令不是越多越细越好——这正是我把策略意图设计成结构化对象（可校验、可编译、可灰度）、而不是一长串自然语言 prompt 的原因。（来自 Anthropic 公开分享，非一手经验。）
