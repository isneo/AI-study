为什么是 AI Native

我们不是现在才用 AI——审核、策略早就在用；真正的转变，是让 AI 从每个环节的工具，变成打通链路的主体。

这四个死结，不是规则写得不够多、人手不够多能解决的。

对抗是小时级的变异，人写规则是天级，规则天生追不上；"鞋带都不会系"这种嘲讽没有关键词，规则和分类器判不了；判一条内容要边查边找证据、路径因案而异，固定流程覆盖不了；而 4w+ 策略、13w+ 特征、人均维护 200+ 的判断力全锁在人脑里，靠堆人根本维护不过来，还在持续熵增。这些都是传统方案的能力边界之外。

enabled 阶段我们早就做了——审核辅助、以及 AI 规则校验、特征推荐等能力。但这里我自己的一个判断是：这种"咨询师式"的问答辅助，做到头也只是个咨询师——要让 AI 真正替人干活，必须把"动作"也交给它。这正是从 enabled 跳到 native 的动因。

AI-native 这一阶段的核心追问是：能不能让 AI 自己打通整条链路——自己找证据、识别风险、处置风险、跑成闭环，而不再是每个环节各用一点 AI、靠人把它们串起来？因为我们走过 enabled、发现"光用 AI 当工具，链路还是断的"，才被逼着往前走的下一步。事后看，这个方向和行业大势是一致的——业界也在从 workflow 架构转向 agent；但我们不是看着趋势做的，是被自己的问题推到这一步的。

诚实交底：native 的目标是全生命周期到生产，目前已落地低风险段（规则草稿 / 数据探查 / 影子分析）+ 风险地基骨架（治理 / 验证 / 观测的最小可用版），生产闭环还在沙箱 + 灰度 + 端到端补齐中。至于 AI 打通链路后、人该站到哪里，是下一章的事。

面试口播 · 业务背景
讲 AI Native 之前，我想先说清楚它的由来。/ 内容风控有几个传统方案补不上的死结——对抗速度追不上、隐晦语义判不了、取证路径不固定、判断力锁在人脑里。// 面对这些，我们先走了 AI-enabled：审核辅助、策略分析都上了 AI，/ 但发现一个根本问题——AI 是无处不在了，可链路还是靠人的经验串起来的，链路本身是断的。// 所以我们往前走到 AI-native：让 AI 自己打通链路、找证据、识别、处置、跑成闭环，而不再是靠人把各环节串起来。/ 所以这一步不是追风口，是被问题逼出来的——我们不是现在才用 AI，是发现光把 AI 当工具不够。

AI Native 是什么 · 认知规模化

传统风控是流水线，AI Native 是闭环；传统风控用规则代码，AI Native 用策略意图。

第一部分 · 是什么（范式之别）

差别不在"是否用 AI"，而在人站在哪里。

传统风控里，人在流水线的每一环，强依赖人的经验判断——审核员点删留、工程师写规则、复审员兜底。机器只是其中的工具，没有谁端到端为"这条内容到底有没有风险"负责。

AI Native 里，中间环节交给 Agent 端到端自主闭环，人从执行者变成目标制定者和监督者，退到两端：上游定策略意图、下游判疑难案。三个核心特征一句话收束——意图到结果闭环交付、策略意图作为新表达层、人机分工根本重构。

面试口播 · 是什么
最简单的判断标准：看人站在哪里。/ 传统风控里人在流水线上——审核员点删留、工程师写规则。/ AI Native 里人退到流水线之上——定策略意图、设决策边界、只处理 Agent 搞不定的疑难案。// 所以它不是"AI 更强的传统风控"，是人机关系被重新设计的风控。
面试官大概率追问 · 和 AI-enabled 的区别
"业界大部分做的是 AI-enabled 风控，你说的 AI Native 核心区别到底是什么？"
一句话：区别不是"AI 做得更多"，而是角色互换——人从执行者变成目标制定者和监督者，Agent 从工具变成发起者。// 最硬的判定——把 AI 拿掉：enabled 退回人工还能跑，Native 整个流程不成立。/ 注意：用没用大模型不是判据，有没有"发起权和闭环权"才是——只做识别打分、人来决策，是 enabled；能自主发起处置并闭环，才是 Native。
  • 谁发起enabled：人写规则、人发起迭代 ／ Native：Agent 提案策略、自动迭代
  • 谁审核enabled：人在每环兜底执行 ／ Native：人退到上游定目标、下游审 Agent 决策
  • 知识形态enabled：规则 + 样本，需翻译成代码 ／ Native：策略意图，自然语言直接表达
第二部分 · 核心价值（认知规模化）

那它最大的价值是什么？把判断力变成可批量供应的资产。

过去，风控能力等于资深策略运营的数量——靠人脑判断风险、分析案例、手写规则，一个人能覆盖的策略量有上限，经验还会随离职带走。规模化只能靠堆人头。

AI Native 把这件事彻底改写：专家的判断模式被沉淀成策略意图、注入 Agent，再用机器规模施加到全平台。判断力第一次脱离个体、像算力一样被批量调用——这就是认知规模化。

面试口播 · 核心价值
一个资深策略运营，判断风险、分析案例、写规则、发布拦截，全靠他自己的经验和工时——慢、有上限、还不可复制。/ AI Native 做的，是把这套判断模式沉淀成策略意图、注入 Agent，让机器以全平台规模并发执行、策略从周级迭代变成小时级生效。// 同一套判断力，从一个人的产出放大成全平台的认知供给——这就是认知规模化。/ 人少、自进化、飞轮，都是这件事的衍生，不是它本身。

理想态 · 掌舵人的终局（这里有点冗余，删掉或3句话带过）

终局不是全自动审核，而是人在环上掌舵放权、Agent 在环内自治执行的认知系统。

我要做的，不是管一支审核团队，而是设计一套会自进化的认知系统，并亲手定义它的放权边界。

这套系统的格局，藏在放权的尺度里——这正是我作为负责人最核心的判断：风险越低，Agent 越自治（L1-L2 全自治、海量内容秒级处理）；风险越高，我把人拉得越近（L5 红线，Agent 仅识别冻结，处置权人专属）。

我借鉴自动驾驶 L0-L5 的分级思想，自定义了一套内容风控的放权等级——不是照搬 SAE 标准，而是用类似的风险分层语言，表达"什么动作可以自治、什么必须人审"。我的位置不在每一环的执行里，而是在环上（On the Loop）校准这条放权曲线——这是"操盘者"和"操作者"的分水岭。

"在环上"具体落在三个动作上：入口定意图（表达要拦什么、容忍度多少）、出口抽检校准（随机复核 Agent 的处置样本，校准指标真值、防止 Agent 自己评自己导致失真）、看大盘掌舵（整体效果不行时，决定收放权、触发意图调整）。人不再处理每一条内容，而是定方向、做校准、掌全局——这就是"在环上"和"在环内"的根本区别。

放权回答了"管什么"，三层协作回答"谁来管"。

我把人和 Agent 分成三层：治理层（人定意图、设边界、担责任）、协作层（人机共治，Agent 处置疑难、人复核反哺）、执行层（Agent 7×24 自主判定）。

关键设计是双向回路：治理层向下游下发策略意图与决策权，协作层向上游反哺判例与对抗样本——人始终在"喂养"和"校准"这套系统，而不是在流水线上做操作。

面试口播 · 理想态
我对终局的判断是认知有机体——人在环上掌舵、Agent 在环内自治。/ 它的格局不在"全自动"，而在放权的尺度：低风险放给 Agent、高风险收到人手里，我借鉴 L0-L5 的分级思想、自定义了一套放权等级来校准这条曲线。// 落到执行，人和 Agent 三层协作——人退到治理和协作层，执行层交给 Agent，我在环上做三件事：定意图、抽检校准、看大盘掌舵。/ 我的价值不是审得多快，是设计这套系统、并掌握它的放权边界。

AI Native 风控系统骨架

一张骨架图收口怎么落地：一条主线治风险、策略运营回路反哺、三纵托底座、数据飞轮自进化。

好架构的功夫，主线越简单越好、基座越通用越好——难的是让它自己转起来的那个飞轮。

一横是治风险主线：识别 → 研判 → 处置 → 反馈，一个风险事件的完整生命周期。三纵是所有业务共用的基础能力——知识管理、harness、行为上下文。横纵交织，构成系统的四大基座。

但骨架真正的灵魂在两处：一是横纵之间那条策略运营回路——案件分析、批量复盘、反哺策略，这是我作为负责人最核心的阵地；二是托底的数据飞轮，它让四大基座越用越强，把"静态系统"变成"会自己长大的认知系统"。

其中数据飞轮是这套架构最难、也最值得单独讲清的一块——它到底怎么转、靠什么转、怎么不转歪，放到 04 章作为核心挑战专门拆开。

面试口播 · 系统骨架
我用一张核心骨架图收口整套系统。/ 一横是治风险主线：识别 → 研判 → 处置 → 反馈，一个风险事件的完整生命周期。/ 三纵是三大基座：AI 原生知识管理、harness 运行时、行为上下文，所有业务共用的基础能力。/ 横纵之间一条策略运营回路反哺策略——这是我最核心的阵地；/ 底座一个数据飞轮，让四大基座越用越强。// 飞轮怎么转我也想清楚了：微观验证每次意图调整、宏观监控整体效果，而且要靠数据策展（洗筛校）防止脏数据把飞轮带歪。/ 每一块背后都有具体的工程落地和踩过的坑，您想从哪一块深入？

演进路径 · 与核心挑战

落地的成熟度，不看你敢让 AI 做多少，看你敢拦住它不做什么。

回到开头那条演进线——enabled 我们已经走过，native 的落地从这里开始。

AI-enabled 阶段（AI 辅助审核、辅助策略）已经落地，这里不再展开。真正要讲的是 AI-native 怎么一步步落地——它不是一次性把整条链路交给 Agent，而是先在策略运营试点线验证核心，再向主链路扩张。下面的决策点，都是这条 native 落地路径上的核心挑战与判断。

试点线已经让 Agent 真正"动手"——自己跑 ReAct 多步循环，做规则草稿、数据探查、影子分析，但不触发上线动作。就算 Agent 走偏，代价是人工弃用一次输出，不会变成生产事故；这正是"先在容错高的子集跑通"的工程选择。

〔实证可再补 · B：试点的真实规模、跑了多久、可量化效果（如草稿采纳率、分析提效），按真实项目填入；当前已埋点 / 未埋点据实标注。〕

风控是高敏场景，可控、可信先行；轮子先转起来，自进化才越跑越顺。

这是演进路径的判断，比具体方案更上层。我不会一上来就改主链路——那一旦出错就是线上分发事故。我先走策略运营试点线：它从分析到发布动作风险递增，但有人审兜底，整体代价可控。

在这条线上，我先把两块地基夯实——harness 把"可控可信"立起来（约束随风险递增、发布必经人审），数据飞轮让"自进化"转起来（规则在反馈中迭代）。地基被真实流量证明可靠，再抬升放权、扩张到主链路。先在赔得起的地方验证，再向赔不起的地方走。

约束不是越严越好，是该松的松到底、该严的严到人。

harness 是这条线上最核心的决策点：约束不能一刀切——分析无代价就放手，发布要生效就死守，约束强度随动作风险递增，到发布必须人工审核。这让 Agent 在低风险动作上充分自治、在高风险动作上被死死兜住。

数据飞轮则多一层警觉——喂给飞轮的每条数据都得先过一关：这是真相，还是有人想让我以为的真相？规则发布后效果反馈回来，可信的自动迭代下一版，可疑的（数据不稳、恶意申诉、对抗投毒）必须人校验。脏反馈一旦进了飞轮，规则只会越迭代越歪。

不迷信大模型万能——规则扛得住的，不必惊动 AI。扩张到主链路时还有两个决策点：主链路不追求大模型完全替代规则引擎，而是互补分层——规则扛海量明确违规、大模型啃疑难隐晦；行为上下文则在丰富度和成本间分层——高频判定用离线特征，疑难才上实时全量。

先回答一个尖锐质疑：这套验证流程，传统风控不也有吗？

流程本身确实通用。AI Native 不在"流程长什么样"，在"流程里每个环节谁干活"——环内的活全是 Agent 自主干、且在护栏内动态决策（ReAct，不是写死的固定流程）；人退到环上，只定意图、抽检校准、最终权衡。

飞轮转的是"意图 → 效果"这一圈：微观每调一次意图用"调前 vs 调后"验证有没有达成，宏观 7×24 监控大盘、退化触发再调。这才叫闭环，不是改完不管的开环。

交底：这是飞轮的意图侧（人驱动，设计在推进）；另一侧系统自进化（数据反哺让 Agent 自己变强）是下一步，还没做。

飞轮不能只是口号——得讲清它靠什么才转得起来。

往下拆，评估靠三根支柱、缺一不可：机器评估（六层指标，飞轮的"仪表盘"）、人抽检（校准真值，防 AI 自己评自己，飞轮的"校准器"）、数据建设（trace / 埋点 / 抽检反馈，飞轮的"燃料"）。

再往下是数据建设和技术实现的细节。P8 的功力是知道飞轮依赖这套地基（第 2 层），而不是只喊"我有飞轮"——越往下越是实现，越往上越是判断。

飞轮也会"越用越歪"——如果喂进去的数据是脏的。

内容审核场景人工标注错误率可高达 40%，还有恶意投毒。所以飞轮和指标之间要夹一条数据策展链路：洗 → 筛 → 校（清洗错误标签 → 优先复核最可能错的样本 → 人抽检校准真值），有 NVIDIA / Google / cleanlab 的成熟方案背书。

关键认知：数据不是越多越好，是越干净越好——脏数据进飞轮，规则只会越迭代越歪。这条链路的详细治理（来源可信度分流、防投毒）见附录 Q12。

面试口播 · 演进与核心挑战
我落地分两层想。/ 上层是演进路径——风控是高敏场景，可控、可信必须先行，轮子也要先转起来。/ 所以我不直接碰主链路，先走策略运营试点线，在这条线上把 harness 和飞轮两块地基夯实，再扩张，先在赔得起的地方验证，再向赔不起的地方走。// 下层是每个部件的决策点。/ harness 的约束不一刀切，随动作风险递增——分析放手、发布必须人工审核。/ 飞轮也是，可信反馈自动迭代、可疑反馈人校验，否则越迭代越歪。/ 主链路我不追求大模型替代规则引擎，而是互补分层；行为上下文按风险分层用。// 一以贯之的标尺，就是看代价、不看能力——落地的成熟度，不在敢让 AI 做多少，而在敢拦住它不做什么。
面试官大概率追问 · 为什么先做策略运营、不直接做主链路
"主链路最有价值，为什么绕开它、先做策略运营这条线？"
因为价值最大的地方，错误代价也最大。/ 主链路直接决定海量内容分发，Agent 判错就是不可逆的线上事故。// 策略运营线能让我在真实写线上的压力下，把最难的 harness 约束、规则迭代飞轮验证扎实，而代价被发布环节的人工审核兜住。/ 这不是挑最安全的，是挑最能验证核心、且代价可控的。/ 诚实说边界：这条线验证了授信的"人批准"档，Agent 自主处置那档要到接主链路时才真正验证。
  • harness一刀切 ／ 按动作风险分级，发布人审
  • 飞轮全自动迭代 ／ 可信自动、可疑人校验
  • 主链路大模型替代 ／ 与规则引擎互补分层
  • 行为上下文一律实时全量 ／ 按风险分层，高频用特征

项目当前进展

前面讲的是判断与设计；这里如实交代两个阶段各自落地到了什么程度——已落地的讲清楚，在补的不夸大。

一 · AI-enabled（已落地，稳态运行）

AI 作为审核与策略的辅助，已在生产稳定运行：

  • 智能审核AI 辅助审核员、标注回流优化模型 ／ 审核人员错审率降低约 26%
  • 大模型策略诊断规则页内嵌大模型做策略校验与分析 ／ 策略错误发现率约 40%

这一阶段 AI 已无处不在，但仍是"咨询师"——所有动作由人执行，链路靠人串起来。

二 · AI-native（试点中，地基已起）

在策略运营试点线上，已让 Agent 从"问答辅助"走到"低风险段动手"——能自主跑 ReAct 多步循环，做规则草稿、数据探查、影子分析，并跑通了 harness 的治理 / 验证 / 观测骨架。到生产的完整闭环（沙箱、灰度、端到端）仍在补齐中——这是下一步，不当作已完成来讲。
