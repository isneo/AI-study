Claude Code 建设中的坑点

基于 Anthropic 官方工程博客与团队负责人公开演讲，整理出 12 个真实失败案例与设计教训。

⚠ 阅读前必读 · 关于"坑点"的定性

官方语境里几乎没有正式的"踩坑清单"，教训散落在复盘里，表述多为"我们试了 X 不行、改成 Y"。本文档把它们反向工程成坑点，并为每条标注两个属性 —— ● CLAUDE-CODE 特有（官方明确是建 Claude Code 或同源 harness 时踩的）与 ● 通用 AGENT 教训（广义经验，别误以为它独有）。

00
坑点全景：踩在哪个环节
核心 agent 循环为「收集上下文 → 执行动作 → 验证结果 → 重复」，绝大多数坑都落在这条循环上。
FIG.0 — agent 循环与坑点分布（循环结构源自官方博客 "Building agents with the Claude Agent SDK"）
01
上下文工程：最深的几个坑
官方明确：agent 时代核心已从 prompt engineering 转向 context engineering，而这正是踩坑最集中处。
P1
上下文压缩（Compaction）极难做对，迭代约 100 次用户仍嫌弃
● CLAUDE-CODE 特有长任务 / Long-horizon

对话接近 200K token 上限时，系统发一条特殊消息让 Claude 总结全部进度，再清空对话、塞回摘要、继续。听着简单，实际是整个产品里最难调的部分之一。

Claude Code 在 compaction 提示词上大约迭代了 100 次，用户被"压缩"时的体验依然让人沮丧。—— Cal, Anthropic Applied AI 团队负责人（公开演讲）
为什么是坑

摘要必然丢信息。丢错了，agent 会在长任务中途"失忆"，把之前的决策、约束、踩过的雷全忘掉，导致返工甚至跑偏——这是个没有干净解的工程难题。

官方的应对

① 持续迭代压缩提示词（但承认体验仍不完美）；

② 转向记忆系统：让 agent 自己往文件里写 markdown 笔记，清空对话后直接读自己的笔记，而非依赖外部摘要；

③ 正尝试把"主动记笔记"直接训练进模型，让其自动发生。

P2
"上下文腐烂"：没到 200K 上限性能就垮了
● 通用 AGENT 教训可靠性

很多人以为塞满前都安全。真相是：在 5 万–15 万 token 区间，模型表现就开始退化（阈值随任务而定），远早于撞上硬上限。

为什么是坑

"窗口大"是种陷阱式安全感。盲目塞东西会在你没察觉时悄悄降准确率，且不会给明确报错。

应对

把 context engineering 当一等公民：只在必要时加载（P9）、用子 agent 隔离（P3）、主动压缩与记忆（P1）。核心是少即是多。

P9
老 RAG 思路：开局把所有文档一次性塞进去
● 通用 AGENT 教训渐进式披露

2024 初主流是先检索一堆文档全塞进首个 prompt。Claude Code 走反路——启动时不读目录里所有文件，只告诉模型"在哪个目录"，给它列目录/读文件工具，需要时再读。

容易忽略的反例

渐进式披露有例外：永远有用的信息要前置。如 CLAUDE.md（用户指令）开局直接加载，不让 agent 再调工具读——因它与任何任务都相关。这条边界很多人没掌握：要么全前置撑爆，要么全延迟连必读的也不读。

P8
系统提示词写得"太具体"或"太模糊"，都会翻车
● CLAUDE-CODE / 客户案例Goldilocks 区间

真实案例：有客户把一份 32 页的 SOP（标准操作流程）PDF 直接灌进系统提示词，把模型压垮了。但反过来写太模糊，模型又缺指引。

官方的判定方法
"好朋友测试"：把指令给一个不懂你领域的朋友看，若他看不懂该干嘛，模型多半也看不懂。—— Anthropic Applied AI 团队

团队建议初期宁可偏模糊：先让 agent 跑、看它在哪崩，再逐条加必要指令。一上来写太细，根本分不清哪些规则真有用、哪些只是噪声拖累性能。

02
工具设计与架构：预期落空的地方
有些功能上线后才发现，根本不是按当初设想的方式起作用的。
P3
子 agent（Sub-agent）：原本的设想完全落空了
● CLAUDE-CODE 特有最典型的"预期 vs 现实"坑

最初设想：Claude 把任务拆成多个并发原子单元，分发给子 agent 同时干、再汇总。结果 Claude 在"拆并发原子单元"上表现很差，设想直接落空。

意外发现的真正用途

子 agent 在"探索"场景里反而极有价值。Claude Code 要理解代码库时，读大量文件吃掉巨量上下文；把"调研"丢给子 agent——让它把自己的上下文窗口炸满、最后只返回一份精简报告——主 agent 就能把宝贵上下文留给真正的实现工作。

💡 全文档最值得记的一条：一个功能的实际价值常和你设计它的初衷不一样。

P4
工具混淆：分不清"用网页搜索还是网盘搜索"
● CLAUDE.AI（同源 harness）工具命名

线上真实问题：Claude.ai 同时有「网页搜索」和「Google Drive 搜索」，模型会把它们搞混——东西明明在网盘里却跑去搜网页，反之亦然。

根因与修复

工具描述被悄悄拼进系统提示词，所以写工具描述本质就是写 prompt。修复：在描述里讲清"什么数据存在哪里、何时用哪个工具"，并直接放 few-shot 示例。CC 和 Claude.ai 都大量用此模式。

P6
Artifacts 早期：每次改动都把整个 HTML 从头重写
● CLAUDE.AI 演进坑状态管理不足

Claude 3.5 Sonnet 擅长写带内嵌 JS/CSS 的 HTML，催生 Artifacts。但早期版本不会原地局部编辑，而是整份 HTML 从头重写。

这暴露了什么

说明当时模型还搞不定复杂状态管理——无法在已有结构里做精准增量修改。这是随模型能力提升才逐步解决的能力型坑，非单纯工程可补。

P11
无意中"打破" Prompt Caching，悄悄推高成本与延迟
● 通用 AGENT 教训成本 / 延迟

Agent 会连续发很多次 API 调用。若系统提示词、工具定义、历史对话在调用间保持不变（只在末尾追加），prompt caching 能大幅降本提速。

容易踩的坑

context engineering 的某些操作会不小心把缓存打破：比如没必要地频繁增删可用工具集、重排本应静态的内容顺序。缓存一旦失效，成本和延迟都会悄悄上去，而你可能根本没注意到原因出在这。

03
架构、评估与抽象层
更高层面的教训：选错架构范式、用错评估口径、依赖看不懂的框架。
P7
用 Workflow（写死的链式调用）做开放任务，脆弱且无法纠错
● 通用 AGENT 教训架构范式

2024 初模型较弱时，主流是 workflow：多个 LLM 调用链在一起、中间夹写死逻辑。官方见过客户为客服场景把 50 个 prompt 串成一条链。两个致命限制：

① 只能处理被显式编码进流程的场景，开放任务无能为力；

② 缺乏错误恢复——中途出意外，系统通常"跑完流程但产出垃圾结果"。

Claude Code 走 agent 路线：让 LLM 在带工具的循环里跑、自己判断何时完成。它能处理没预料的边界情况，也擅长识别异常工具返回并自我调整。

P10
按"每 token 单价"比成本，是误导性口径
● 通用 AGENT 教训成本评估

官方在 Claude Opus 4.5 评测里发现：更贵的模型用明显更少的 token 就拿到更高的 SWE-bench 分。

结论

生产部署要按"每个完成任务的成本"算，而非 token 单价。list price 更高的模型，因效率更高，单任务反而可能更便宜。只盯单价选便宜模型是常见算账误区。

P12
用了自己根本不理解的框架 / 抽象层
● 通用 AGENT 教训框架抽象

官方观察的常见翻车原因：很多团队栽在使用了理解不够深的库上。生产级框架须给足控制力与灵活性——能换 prompt、定制工具、需要时上多 agent——而非过度封装、自作主张的脚手架。

注脚

这正是 Anthropic 把 CC 底层 harness 抽成 Claude Agent SDK 的逻辑：暴露可控原语，而非黑盒。

P5
90% 的失败，根因就是 prompt 写得烂
● 通用 AGENT 教训头号失败模式

这是 Cal 在企业客户处遇到的最常见失败模式，单独压轴因为它太普遍：

90% 的情况下，系统表现不如预期时，问题就是指令本身——让一个不懂这领域的人来读，它根本讲不通。—— Anthropic Applied AI 团队，"2025–2026 头号 prompt 建议"

换句话说：在怪模型、怪框架、怪架构之前，先回去读一遍自己的 prompt。

04
我的批判性补充（不全顺着资料说）
几点反向思考，帮你判断该如何看待和使用这些资料。
1

这些"坑"带明显的幸存者 / 公关滤镜

官方分享的都是"踩了坑然后优雅解决"的故事。真正难看的失败（彻底砍掉的功能、大规模事故）几乎不进工程博客。所以这是偏乐观的样本，别当成 Claude Code 问题的全貌。

2

区分"能力型坑"与"结构型坑"

P6（整文件重写）是能力型坑，靠模型迭代就解决；P1（压缩）、P11（缓存）是结构型坑，再强的模型也得靠工程管。学习时要分清：哪些值得你投工程精力，哪些等下一代模型就好。

3

P3 才是最有迁移价值的一条

它的元教训不是"子 agent 怎么用"，而是 "先有能力、再找用途，而非先定用途、再硬塞能力"。这个反直觉发现适用于几乎所有 AI 产品设计——先观察模型实际擅长什么。

4

建议的学习路径

① 读官方 Building agents with the Claude Agent SDK 建框架；② 看 Cal 演讲原视频补失败细节；③ 再读 Effective context engineering 与 Writing tools for agents。务必看一手来源，二手整理（含本文档）都有损耗。
