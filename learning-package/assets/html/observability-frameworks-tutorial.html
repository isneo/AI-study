Observability for LLM Apps

OpenTelemetry × LangSmith × LoongSuite 简易教程

这份单文件手册把三个工具放在同一条链路里讲：从“采集什么数据”、到“怎么发出去”、再到“在哪里看 LLM/Agent 的每一步”。你可以直接照着代码跑一个最小 demo，也可以用最后的决策面板选择生产组合。

核验日期：2026-05-27 语言：Python + Java 目标：traces / LLM runs / evals / OTLP

1三者定位：先别把它们当同一类东西

一句话心智模型

OpenTelemetry 是开放标准和通用遥测管道；LoongSuite 是基于 OpenTelemetry 的增强采集套件；LangSmith 是面向 LLM/Agent 的可视化调试、评测和协作平台。

最实用的组合 LoongSuite 或 OpenTelemetry 负责埋点和采集，OpenTelemetry Collector 负责分发，LangSmith 负责看清 prompt、模型调用、tool call、RAG、Agent step 和评测结果。

别混淆的边界

OpenTelemetry 和 LoongSuite 本身不等于一个完整的“观测后台”。它们通常把数据导到 Jaeger、Grafana Tempo、Datadog、Prometheus、阿里云可观测或 LangSmith 这类后端。

LangSmith 可以接 OpenTelemetry trace，但它的强项不是替代所有 APM，而是把 LLM 应用的输入、输出、上下文、工具调用和评测组织起来。

STANDARD / PIPELINE

OpenTelemetry

统一 traces、metrics、logs 的 API、SDK、自动 instrumentation、Collector 和 OTLP 协议。

COLLECTION DISTRO

LoongSuite

阿里开源的统一可观测采集套件，Python Agent 是 OTel Python Agent 的定制发行版，并增强 GenAI/Agent 框架支持。

LLM PLATFORM

LangSmith

LLM 应用 tracing、prompt 调试、数据集、experiment、evaluator、人工反馈和监控平台。

2OpenTelemetry 简易教程

APISDK

你在代码里创建 span

比如一次 RAG 检索、一次 LLM 调用、一次工具执行，都可以成为 span。

Instrumentation

自动捕获常见框架

Flask、FastAPI、requests、数据库客户端等可以用自动 instrumentation 生成边界 span。

Collector

统一接收和分发

应用发 OTLP 到 Collector，Collector 再转发给 Jaeger、Tempo、Datadog、LangSmith 等。

2.1 最小上手路径

先本地打印

用 console exporter 看到 trace 结构，确认 span 和属性正确。

再接 Collector

把 exporter 切成 OTLP，默认可以发到本地 localhost:4317。

最后接后端

Collector 负责 fan-out 到 Jaeger/Tempo/Datadog/LangSmith 等不同目标。

OpenTelemetry 手动 span + Flask 自动 instrumentation
# app.py
# 目标：演示“一个 HTTP 请求”里如何手动补充 LLM/RAG 语义 span。
# 运行时再用 opentelemetry-instrument 包住 Flask，
# HTTP server span 会自动生成；下面的 rag.retrieve / llm.call 会挂在它下面。

from flask import Flask, request, jsonify
from opentelemetry import trace

app = Flask(__name__)

# tracer name 建议用稳定命名：公司/项目/模块。
# 后端 UI 里可以用它过滤“这些 span 是哪个模块产生的”。
tracer = trace.get_tracer("ai-study.otel-demo")


def retrieve_context(question: str) -> list[str]:
    # span 命名建议用“领域动作”，不要用随机函数名。
    # 好名字：rag.retrieve / vector.search / tool.payment_check
    # 坏名字：func1 / test / call
    with tracer.start_as_current_span("rag.retrieve") as span:
        span.set_attribute("rag.query.length", len(question))
        span.set_attribute("rag.top_k", 3)
        span.set_attribute("db.system", "mock-vector-store")

        # 生产环境不要把完整用户问题、完整文档内容直接写进 attribute，
        # attribute 会进入观测后端，可能涉及隐私、合规和成本。
        docs = [
            "OpenTelemetry records traces, metrics, and logs.",
            "LangSmith focuses on LLM app debugging and evaluation.",
        ]
        span.set_attribute("rag.documents.count", len(docs))
        return docs


def call_llm(question: str, context: list[str]) -> str:
    with tracer.start_as_current_span("llm.call") as span:
        # gen_ai.* 是 OpenTelemetry GenAI 语义约定常见字段。
        # 好处是后续可以跨平台聚合：按模型、provider、token、错误率看趋势。
        span.set_attribute("gen_ai.operation.name", "chat")
        span.set_attribute("gen_ai.system", "demo")
        span.set_attribute("gen_ai.request.model", "demo-model")
        span.set_attribute("gen_ai.usage.prompt_tokens", 42)
        span.set_attribute("gen_ai.usage.completion_tokens", 16)

        # 这里用 mock，真实项目里替换成 OpenAI / Anthropic / 本地模型调用。
        return f"mock answer for: {question}; context_count={len(context)}"


@app.get("/ask")
def ask():
    question = request.args.get("q", "what is tracing?")

    # 这个顶层业务 span 会成为 HTTP request span 的子 span。
    # 它把 RAG 和 LLM 两个内部步骤聚合成“处理一次 AI 请求”。
    with tracer.start_as_current_span("assistant.answer") as span:
        span.set_attribute("app.feature", "observability-demo")
        span.set_attribute("app.question.length", len(question))

        context = retrieve_context(question)
        answer = call_llm(question, context)

    return jsonify({"answer": answer})

python -m venv .venv
source .venv/bin/activate

# 2. 安装 Flask 和 OTel 发行包。
# opentelemetry-distro 提供 opentelemetry-instrument 命令。
# opentelemetry-exporter-otlp 用于把数据发给 Collector 或 SaaS 后端。
pip install flask opentelemetry-distro opentelemetry-exporter-otlp

# 3. 自动安装当前环境里可识别库的 instrumentation。
# 例如 Flask / requests / SQLAlchemy 等会被自动补上边界 span。
opentelemetry-bootstrap -a install

# 4. 本地调试：先把 traces / metrics / logs 打到终端。
# 这一步不用部署 Collector，适合确认 span 名称、父子关系、attribute 是否正确。
opentelemetry-instrument \
  --traces_exporter console \
  --metrics_exporter console \
  --logs_exporter console \
  --service_name ai-study-otel-demo \
  flask --app app run -p 8080

# 5. 触发请求后，在终端里应该能看到：
# - Flask 自动生成的 HTTP server span
# - assistant.answer
# - rag.retrieve
# - llm.call
curl "http://127.0.0.1:8080/ask?q=hello"

# 6. 接 Collector：把 exporter 从 console 切成 otlp。
# 默认 OTLP/gRPC 端口是 localhost:4317；OTLP/HTTP 常用 localhost:4318。
opentelemetry-instrument \
  --traces_exporter otlp \
  --metrics_exporter otlp \
  --logs_exporter otlp \
  --service_name ai-study-otel-demo \
  flask --app app run -p 8080
otel-collector-config.yaml
# otel-collector-config.yaml
# 目标：本地启动一个 Collector，把收到的数据打印出来。
# 等你确认数据正确后，再把 debug exporter 换成 Jaeger / Tempo / Datadog / LangSmith。

receivers:
  otlp:
    protocols:
      grpc:
        # SDK 通过 OTLP/gRPC 上报时用这个端口
        endpoint: 0.0.0.0:4317
      http:
        # SDK 通过 OTLP/HTTP 上报时用这个端口
        endpoint: 0.0.0.0:4318

processors:
  batch:
    # batch 会把小批量遥测数据攒成批次再发出，
    # 生产环境几乎都应该开启，能减少网络开销和后端压力。

exporters:
  debug:
    # detailed 会打印完整 payload，适合本地调试；
    # 生产环境不要开 detailed，可能输出 prompt、completion、用户数据。
    verbosity: detailed

service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [batch]
      exporters: [debug]
    metrics:
      receivers: [otlp]
      processors: [batch]
      exporters: [debug]
    logs:
      receivers: [otlp]
      processors: [batch]
      exporters: [debug]
启动 Collector
# 启动 Collector。
# - 4317：OTLP/gRPC
# - 4318：OTLP/HTTP
# - --rm：退出后自动删除容器，适合本地实验
docker run --rm \
  -p 4317:4317 -p 4318:4318 \
  -v "$PWD/otel-collector-config.yaml:/etc/otelcol/config.yaml" \
  otel/opentelemetry-collector:latest \
  --config=/etc/otelcol/config.yaml
核心理解 OpenTelemetry 不负责替你判断“回答是否正确”。它负责把一次请求里发生了哪些步骤、耗时多少、错误在哪、上下游如何关联，按标准格式记录下来。

3LangSmith 简易教程与原理

LangSmith 的入门重点不是“采集所有系统指标”，而是让你看到 LLM 应用的语义链路：用户输入、检索上下文、prompt、模型响应、工具调用、评测分数、人工反馈。

创建 LangSmith API key

app.py：业务 span + LLM span
# app.py
# 目标：让 LangSmith 看到一棵清楚的 run tree：
# assistant
#   ├─ get_context       run_type=retriever
#   └─ OpenAI chat call  由 wrap_openai 自动记录

import os
from openai import OpenAI
from langsmith import traceable
from langsmith.wrappers import wrap_openai

# wrap_openai 会自动捕获模型调用的 input / output / token / latency。
# 这样你不需要手动给每次 client.chat.completions.create 加 trace。
client = wrap_openai(OpenAI())


@traceable(
    name="get_context",
    run_type="retriever",
    metadata={"component": "mock-knowledge-base"},
)
def get_context(question: str) -> str:
    # 真实项目里这里会查向量库、知识库、搜索引擎或业务数据库。
    # run_type="retriever" 会让 LangSmith 把它识别为“检索步骤”，
    # 方便后续分析：是检索质量差，还是模型基于正确上下文答错了。
    return (
        "OpenTelemetry is a vendor-neutral telemetry standard. "
        "LangSmith is focused on LLM app observability and evaluation."
    )


@traceable(
    name="assistant",
    run_type="chain",
    tags=["tutorial", "observability"],
    metadata={"app_version": "demo-v1"},
)
def assistant(question: str) -> str:
    # 这个函数会成为顶层业务 run。
    # 它下面会自动挂 get_context 和 OpenAI 调用，形成父子结构。
    context = get_context(question)

    messages = [
        {
            "role": "system",
            "content": (
                "Answer with the context below. "
                "If the context is insufficient, say you do not know.\n\n"
                f"Context: {context}"
            ),
        },
        {"role": "user", "content": question},
    ]

    response = client.chat.completions.create(
        model=os.environ["OPENAI_MODEL"],
        messages=messages,
        temperature=0,
    )
    return response.choices[0].message.content


if __name__ == "__main__":
    # 运行后去 LangSmith 项目里看 trace：
    # 重点看 inputs / outputs / child runs / latency / token usage。
    print(assistant("OpenTelemetry 和 LangSmith 有什么区别？"))

3.2 做一个最小评测

Tracing 回答“发生了什么”，evaluation 回答“这次改动比上次好吗”。LangSmith 的基本评测三件套是：dataset、target function、evaluators。

dataset.py
创建数据集
# dataset.py
# 目标：创建一组可重复运行的“题库”。
# 每个 example 通常包含：
# - inputs：给应用的输入
# - outputs：参考答案或业务期望
# - metadata：场景、难度、来源，方便后续筛选

from langsmith import Client

client = Client()

dataset = client.create_dataset(
    dataset_name="Observability QA",
    description="Tiny examples for comparing observability answers.",
)

# 如果重复运行提示 dataset 已存在：
# 1. 换一个 dataset_name，例如 Observability QA v2

client.create_examples(
    dataset_id=dataset.id,
    examples=[
        {
            "inputs": {"question": "OpenTelemetry 是什么？"},
            "outputs": {
                "answer": "OpenTelemetry 是开放、厂商中立的可观测标准和遥测工具链。"
            },
            "metadata": {"topic": "otel", "difficulty": "easy"},
        },
        {
            "inputs": {"question": "LangSmith 主要解决什么？"},
            "outputs": {
                "answer": "LangSmith 主要用于 LLM 应用 tracing、调试、评测和实验对比。"
            },
            "metadata": {"topic": "langsmith", "difficulty": "easy"},
        },
        {
            "inputs": {"question": "LoongSuite 和 OpenTelemetry 是什么关系？"},
            "outputs": {
                "answer": "LoongSuite 可以理解为基于 OpenTelemetry 的增强采集套件。"

eval.py
运行实验
# eval.py
# 目标：用同一批 dataset 测一个应用版本，生成 experiment。
# 以后你换 prompt / 换模型 / 换 RAG 参数，只要复用同一 dataset，
# 就可以在 LangSmith 里横向比较多个 experiment。

from langsmith import Client


def target(inputs: dict) -> dict:
    # target 就是“被评测的应用”。
    # 真实项目里替换成你的 assistant(question)、RAG pipeline 或 agent.invoke。
    question = inputs["question"]
    return {"answer": f"demo answer for: {question}"}


def keyword_evaluator(inputs: dict, outputs: dict, reference_outputs: dict):
    # evaluator 负责把一次输出转成结构化反馈。
    # 返回的 key 会成为 LangSmith UI 里的指标名。
    # score 建议用 0/1 或 0~1，方便做均值、趋势和阈值告警。
    expected = reference_outputs["answer"]
    got = outputs["answer"]

    required_keywords = expected.split()[:2]
    passed = any(word in got for word in required_keywords)

    return {
        "key": "contains_reference_keyword",
        "score": 1 if passed else 0,
        "comment": f"required_keywords={required_keywords}; got={got[:80]}",
    }


client = Client()

client.evaluate(
    target,
    data="Observability QA",
    evaluators=[keyword_evaluator],
    experiment_prefix="observability-demo",
    max_concurrency=2,  # 并发数；真实调用大模型时要结合限流和预算调低
)

3.4 调试和评估到底怎么工作的

调试：把一次回答拆成可回放的执行树

LangSmith 的调试不是“读心”，而是把一次用户请求记录成一棵 run tree。顶层是一次 trace，里面挂着 prompt formatting、retriever、LLM call、tool call、parser、agent step 等子 run。

对象记录什么用来回答什么问题
trace一次完整请求这次对话整体走了哪条路径？
run一个离散步骤是检索错、工具错，还是模型输出错？
metadata/tags版本、环境、用户分组、实验标签哪个版本、哪类用户、哪个场景出问题？
feedback用户或 evaluator 的分数这条链路质量如何？

评估：把应用版本放到同一套题库里比较

Evaluation 的关键是可重复比较：同一批 dataset，分别跑不同 prompt、模型、RAG 参数或 agent 版本，再用 evaluator 打分并形成 experiment。

Dataset

准备输入、参考答案、上下文或业务期望。

Target

调用你的真实应用，而不是只测孤立 prompt。

Evaluator

规则代码、人类标注、LLM-as-judge、pairwise 对比都可以。

Experiment

保留每个样本的输出、分数和 trace，用于版本对比。

判断口径 调试看路径，评估看质量。trace 告诉你“发生了什么”，evaluator 告诉你“这件事好不好”。生产线上还可以对真实 runs/threads 做 online evaluation，用于发现质量下降、异常回答和安全风险。
核心理解 LangSmith 的价值在“语义可观测”：它让 trace 不只是耗时树，而是能读出 prompt、上下文、模型、工具、反馈、实验版本之间的关系。

4LoongSuite 简易教程

LoongSuite 可以理解为“更偏采集落地的 OTel 增强套件”。它包含 LoongCollector、Python/Go/Java Agent 等组件。Python Agent 的 README 说明它是上游 OTel Python Agent 的定制发行版，并增强了 AI Agent 框架和 GenAI semantic conventions 支持。

当前状态 GitHub README 显示 LoongSuite Python Agent 最新 release 为 0.5.0，发布日期为 2026-05-11。这个生态还在快速发展，实际生产接入前要固定版本并验证依赖解析。

4.1 推荐接入：loongsuite-instrument

安装 distro

loongsuite-distro 提供 loongsuite-instrument 和 loongsuite-bootstrap。

安装 instrumentation

用 bootstrap 从 release 安装 LoongSuite wheel，并配套安装兼容的 OTel instrumentation。

运行你的应用

用 console exporter 本地检查，再切 OTLP 发给 Collector 或后端。

安装与本地运行
# 1. 创建隔离环境。
python -m venv .venv
source .venv/bin/activate

# 2. 安装 LoongSuite Python distro。
# distro 里包含 loongsuite-instrument / loongsuite-bootstrap 等命令。
pip install loongsuite-distro

# 3. 安装 LoongSuite instrumentation。
# --latest 会从 LoongSuite release 安装较新的 instrumentation wheel。
# 生产建议固定版本，不要每次构建都追 latest。
loongsuite-bootstrap -a install --latest

# 4. 轻量环境可以先 auto-detect：
# 只为当前环境里已安装的库安装 instrumentation，减少依赖体积和冲突概率。
loongsuite-bootstrap -a install --latest --auto-detect

# 5. 本地调试：先打印到 console。
# 观察点：
# - 是否生成了预期的 LLM / Agent / tool / RAG span
# - service_name 是否正确
# - 有没有把敏感 prompt 或用户数据打出来
loongsuite-instrument \
  --traces_exporter console \
  --metrics_exporter console \
  --logs_exporter console \
  --service_name ai-study-loong-demo \
  python demo.py

LoongSuite 发送 OTLP
# 1. 安装 OTLP exporter，用来把遥测数据发给 Collector。

5业界产品地图：开源、商业和 OTel-first 怎么看

LLM observability 现在不是单点工具竞争，而是三条路线并行：一体化 LLMOps 平台、开源自托管平台、OpenTelemetry-first 采集/协议路线。选型时先看团队约束，而不是先看功能清单。

产品 / 项目类型强项适合场景注意点
LangSmith商业平台，LangChain 生态Tracing、prompt、dataset、experiment、online eval、协作LangChain/LangGraph 或需要成熟评测闭环平台绑定较强，费用和数据留存要提前评估
Langfuse开源 + 商业开源自托管、prompt/evals/observability 一体，OTel 支持较积极想要 LangSmith 类能力但更重视自托管不同语言 SDK 能力不完全一致，Java 通常走 OTel
Arize Phoenix开源，背后有 Arize AXRAG/LLM debugging、OpenInference、OTLP 友好偏数据科学、RAG 质量诊断、开源 PoC企业治理能力通常看 Arize AX
Braintrust商业平台生产流量打分、回归评测、告警、把失败 trace 转 eval评估和质量闭环优先级很高的团队商业平台路线，需看预算和数据合规
Comet Opik开源 + 商业Tracing、evaluation、monitoring、CI/CD eval，自托管友好想要开源 LangSmith 替代，并需要比较完整的 eval 面板生态成熟度和集成深度要按语言验证

MLflow Tracing开源 / Databricks 生态GenAI trace 纳入 experiment、model registry、MLOps 流程团队已经重度使用 MLflowLLM 专用交互体验和 agent UI 需实测
OpenLIT / OpenLLMetry / TraceloopOTel-first 开源/商业基于 OpenTelemetry 自动采集，后端可替换不想被某个 LLM 平台锁定，已有 OTel 后端评估闭环通常需要另配平台或自研
Datadog / Grafana / Elastic / Honeycomb通用 APM系统级 traces、metrics、logs、告警、SRE 工作流公司已有统一可观测体系LLM prompt、RAG、eval 语义层通常要补仪表盘或插件

如果你更关心“AI 输出质量”

优先看 LangSmith、Braintrust、Opik、Langfuse。它们把 trace、dataset、evaluator、experiment、人工反馈放在同一个工作流里，适合做 prompt 迭代和回归测试。

优先把 OpenTelemetry、Collector、Grafana/Datadog/Elastic 打牢，再把 LLM 平台作为分支后端接入。这样日志、指标、链路和 AI 语义层不会割裂。

6.3 Java 手动 span：补齐 RAG / Agent 语义步骤

AssistantService.java：业务步骤手动埋点
import io.opentelemetry.api.GlobalOpenTelemetry;
import io.opentelemetry.api.trace.Span;
import io.opentelemetry.api.trace.Tracer;
import io.opentelemetry.context.Scope;

import java.util.List;

public class AssistantService {

    // tracer name 建议用稳定模块名，方便在后端过滤。
    // 如果应用通过 OTel Java Agent 启动，GlobalOpenTelemetry 会拿到 agent 初始化的实例。
    private static final Tracer tracer =
            GlobalOpenTelemetry.getTracer("ai-study.java.assistant");

    public String answer(String question) {
        Span span = tracer.spanBuilder("assistant.answer").startSpan();

        // makeCurrent() 会把当前 span 放进上下文；
        // retrieveContext / callModel 里创建的 span 会自动成为它的子 span。
        try (Scope scope = span.makeCurrent()) {
            span.setAttribute("app.feature", "observability-demo");
            span.setAttribute("app.question.length", question.length());

            List<String> context = retrieveContext(question);
            return callModel(question, context);
        } catch (Exception e) {
            // 记录异常，后端 UI 里才能看到错误栈和失败位置。
            span.recordException(e);
            span.setAttribute("error.type", e.getClass().getSimpleName());
            throw e;
        } finally {
            // span 必须 end，否则后端会看到未完成或丢失的链路。
            span.end();
        }
    }

    private List<String> retrieveContext(String question) {
        Span span = tracer.spanBuilder("rag.retrieve").startSpan();
        try (Scope scope = span.makeCurrent()) {
            span.setAttribute("rag.top_k", 3);
            span.setAttribute("db.system", "mock-vector-store");

            List<String> docs = List.of(
                    "OpenTelemetry is a telemetry standard.",
                    "LangSmith focuses on LLM debugging and evaluation."
            );
            span.setAttribute("rag.documents.count", docs.size());
            return docs;
        } finally {
            span.end();
        }
    }

    private String callModel(String question, List<String> context) {
        Span span = tracer.spanBuilder("llm.call").startSpan();
        try (Scope scope = span.makeCurrent()) {
            span.setAttribute("gen_ai.operation.name", "chat");
            span.setAttribute("gen_ai.system", "demo");
            span.setAttribute("gen_ai.request.model", "demo-model");
            span.setAttribute("gen_ai.usage.prompt_tokens", 42);
            span.setAttribute("gen_ai.usage.completion_tokens", 16);

            // 真实项目里这里替换成 Spring AI ChatClient 或 LangChain4j ChatModel。
            return "mock answer for: " + question + "; context_count=" + context.size();
        } finally {
            span.end();
        }
    }
}

ChatController.java：最小聊天接口 + 可观测边界
import io.opentelemetry.api.GlobalOpenTelemetry;
import io.opentelemetry.api.trace.Span;
import io.opentelemetry.api.trace.Tracer;
import io.opentelemetry.context.Scope;
import org.springframework.ai.chat.client.ChatClient;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RequestMapping;
import org.springframework.web.bind.annotation.RequestParam;
import org.springframework.web.bind.annotation.RestController;

@RestController
@RequestMapping("/ai")
public class ChatController {

    private final ChatClient chatClient;
    private final Tracer tracer =
            GlobalOpenTelemetry.getTracer("ai-study.spring-ai-alibaba");

    public ChatController(ChatClient.Builder builder) {
        // defaultSystem 可以放稳定的业务角色说明。
        // 不建议把易变 prompt 全散落在各处，后续评估和回滚会很痛。
        this.chatClient = builder
                .defaultSystem("You are a concise assistant for observability learning.")
                .build();
    }

    @GetMapping("/chat")
    public String chat(@RequestParam String input) {
        Span span = tracer.spanBuilder("spring-ai-alibaba.chat").startSpan();

        try (Scope scope = span.makeCurrent()) {
            span.setAttribute("app.framework", "spring-ai-alibaba");
            span.setAttribute("gen_ai.system", "dashscope");
            span.setAttribute("gen_ai.request.model", "qwen-plus");
            span.setAttribute("app.input.length", input.length());

            // ChatClient 内部是否自动生成模型 span，取决于 Spring AI / 观测配置版本。
            // 这里外层手动 span 的价值是稳定表达“业务调用边界”。
            return chatClient.prompt()
                    .user(input)
                    .call()
                    .content();
        } catch (Exception e) {
            span.recordException(e);
            span.setAttribute("error.type", e.getClass().getSimpleName());
            throw e;
        } finally {
            span.end();
        }
    }
}

9常见坑和生产提醒

OpenTelemetry

  • 只采 HTTP 边界不够：LLM/RAG/Agent 内部步骤需要手动 span 或专用 instrumentation。
  • attribute 命名要稳定：生产里不要随意生成高基数字段，比如把完整用户输入当 label。
  • Collector 是分界点：采样、脱敏、路由、batch、重试最好在 Collector 层统一治理。

LangSmith

  • 注意敏感信息：prompt、context、completion 可能包含用户数据，生产前要确认脱敏策略。
  • 评测别只靠 LLM judge：能用规则、精确匹配、业务指标时，优先补上确定性 evaluator。
  • 项目隔离要清楚：用 LANGSMITH_PROJECT 或 OTLP header 区分环境和应用。

LoongSuite

  • 固定版本：LoongSuite 和 OTel 依赖关系要锁定，避免解析出不兼容版本。
  • 先 auto-detect：轻量环境可先让 bootstrap 只装当前项目需要的 instrumentation。
  • site-bootstrap 谨慎用：它会影响启用环境里的 Python 进程，生产前要评估边界。

组合架构

  • 先画数据流：应用 -> Agent/SDK -> Collector -> 后端，任何一步都要能本地验证。
  • 先 trace 后 metric：LLM 应用早期最需要解释链路，metrics 可以在核心路径稳定后补齐。
  • 不要双重上报：LangSmith SDK 和 OTel exporter 同时开时，要确认不会重复计费或重复 trace。

10术语卡：定义 + 举一反三 + 类比

Trace / Span

定义：Trace 是一次请求的完整链路，Span 是链路里的一个步骤。

举一反三：一次 AI 问答 trace 里可能有 HTTP request span、retriever span、LLM span、tool span、database span。

类比：Trace 像一张外卖订单全流程小票；Span 是“接单、备餐、骑手取餐、送达”这些独立环节。

OTLP

定义：OpenTelemetry Protocol，遥测数据在应用、Collector 和后端之间传输的标准协议。

举一反三：LangSmith、Jaeger、Grafana Tempo、很多商业 APM 都可以接收或间接接收 OTLP。

类比：像统一快递面单格式。不同快递站都能读懂，包裹就能被转运。

Instrumentation

定义：把观测逻辑接入应用的过程，可以手写，也可以由框架自动完成。

举一反三：Flask instrumentation 自动记录 HTTP 请求；LangSmith @traceable 手动标记业务函数；LoongSuite instrumentation 针对 Agent 框架自动补充 GenAI span。

类比：像在工厂流水线上装传感器，有些位置标准化可以自动装，有些关键工位要你自己指定。

Evaluation / Experiment

定义：Evaluation 是对 LLM 应用输出的量化评估；Experiment 是一次评测运行及其结果记录。

举一反三：换 prompt、换模型、换 RAG top-k 后，都应该用同一批 dataset 跑 experiment 比较。

类比：像考试和成绩单。Dataset 是题库，target 是答题学生，evaluator 是阅卷规则，experiment 是这次考试成绩。
