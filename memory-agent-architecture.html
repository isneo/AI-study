Agent Memory Architecture Brief

Memory 是 Agent 的外部上下文管理系统

本页把本次对话整理成一个结构化地图：从 Claude Code Auto Memory、Dream、Mem0，到企业级客服、风控、销售、推荐和工程后端里的记忆设计。

总定义 记忆类型 Claude 边界 算法管线 企业场景 风控库判断 架构选型 质量评估 结构化输出

Memory = 为未来任务准备的可召回上下文

它不是模型权重，也不是第二套业务数据库。它是 agent runtime 围绕 LLM 搭建的一层外部系统：记录、整理、检索、注入、治理过去有用的信息。

Store markdown、SQL、向量库、图数据库、memory store；负责持久化。
Curator 通常由 LLM 执行抽取、总结、冲突判断、合并和重写。
Runtime 决定何时写入、何时召回、召回多少、放进哪段上下文。
Governance 版本、来源、权限、审计、过期、回滚、敏感信息过滤。

四类记忆

个人助手和企业 Agent 都可以用这四层来拆，只是可信度、权限和来源要求不同。

taxonomy
Working

当前桌面

当前任务状态、刚跑过的命令、刚确认的事实、待执行步骤。生命周期最短。

Semantic

事实与偏好

用户偏好、客户画像、服务拓扑、账户关系、产品事实。要求可追溯。

Episodic

经历与证据

过去的对话、工单、案件、事故、会议、工具轨迹。最好保留 raw source。

Procedural

做事方法

SOP、代码规范、销售 playbook、排障步骤、合规流程。写入要最克制。

  • 本地、项目级、文件化记忆，典型路径是 ~/.claude/projects/<repo>/memory/。

Managed Agents Memory Store

  • 属于 Claude Platform Managed Agents API，不是 Claude Code 本地目录。
  • 它是 workspace 级记忆容器，ID 形如 memstore_...。
  • 挂载到 agent session 后，agent 像访问文件系统一样读写。
  • 概念上对应“一个记忆容器”，而 MEMORY.md 更像容器里的索引页。

Dream / Dreaming

  • 可以理解为受控的 memory manager agent：夜间整理、合并、去重、纠错。
  • 官方 Managed Agents Dreams 输入 memory store + 1 到 100 个 sessions，输出新的 memory store。
  • 它不是每轮写一点，而是 batch consolidation。
  • Claude Code 内部若出现 dream-like 机制，可类比为整理本地 memory 文件；官方公开细节有限。

Memory 系统的常见算法管线

主流系统通常是 LLM 做语义判断，工程系统做约束、索引、权限、版本和评估。

pipeline

捕获

收集消息、工具轨迹、工单、日志、CRM 记录、代码变更、会议纪要。

候选筛选

过滤寒暄、一次性格式要求、敏感信息、明显无长期价值的内容。

检索旧记忆

用关键词、向量、实体、时间范围召回可能相关的已有记忆。

LLM 抽取合并

输出事实、episode、规则；判断 ADD、UPDATE、DELETE、NOOP。

工程校验

schema、source id、hash、去重、权限、版本、时间有效性、回滚。

召回注入

任务开始或执行中检索、重排，把最有用的几条放入 context。

风控规则库 / 案例库：是不是 Memory？

这个判断的关键不是“它是否存了过去的信息”，而是它是否被转化成未来任务里的可召回上下文。

risk memory
正方假设

它们是广义 Memory

如果 Memory 的本质是“过去沉淀的信息在未来任务中被召回并影响行动”，规则库和案例库都符合。

  • 规则库记录风险策略、阈值、处置流程，像 procedural / semantic memory。
  • 案例库记录历史欺诈、调查过程、审核结论，像 episodic memory。
  • Agent 调查当前案件时召回相似规则和相似案例，这就是记忆在工作。
反方假设

它们不是 Agent Memory 本身

如果把 Agent memory 定义为“agent runtime 专门维护的上下文层”，规则库和案例库还差一层转化。

  • 规则库通常是权威策略源和控制平面，不能由 Agent 随意总结覆盖。
  • 案例库通常是证据库和事实库，核心价值是可审计、可追溯、不可被压缩改写。
  • 只有被检索、过滤、摘要、重排并注入上下文后，才成为 Agent 可用记忆。
1. 存了过去 规则、案例、日志、数据库都存过去信息；这只是必要条件。
2. 服务未来任务 被用来解释、调查、建议、升级、生成证据链，才进入 memory 语义。
3. 经过上下文化 权限过滤、检索、摘要、冲突处理、时间归因、source 绑定。
4. 注入 Agent 工作区 变成当前案件可理解、可引用、可行动的上下文片段。

最终结论

规则库和案例库不是天然意义上的 Agent memory；它们是风控系统的权威知识资产和事实资产。 在 Agent 架构里，它们应作为 memory 的上游来源：规则库提供 procedural / semantic source of truth，案例库提供 episodic source of truth； memory layer 负责从中检索、摘要、重排、注入上下文。

企业级架构选型

Managed Agents 省的是 agent 基础设施，不是业务控制平面。自研 harness 适合把 agent runtime 当成核心能力的团队。

architecture
Managed Agents

快速验证业务价值

适合长任务、异步任务、工具调用多、团队不想先造 agent loop 的场景。

  • 现成 runtime、session、工具、文件和 memory。
  • 业务 prompt、权限、数据源、验收标准仍要配置。
  • 要关注 beta、数据留存、合规和平台边界。
自研 Harness

掌握核心控制权

适合多模型路由、强审计、强合规、深度可观测、成本治理、平台化 agent。

  • 可以自定义 agent loop、工具策略、状态机、回放和评估。
  • 工程成本高，容易在还没验证业务价值前过度建设。
  • 长期适合核心产品型 agent。
混合架构

自控平面 + 可替换 Runtime

大多数企业更稳的路径：自己管权限、审批、审计、任务队列和 eval。

  • 底层可接 Managed Agents、自研 harness 或其他 SDK。
  • 避免被单个平台锁死。
  • 先验证，再把高价值能力内化。

什么是好的 Memory

不要只看“这条记忆像不像对的”，要看它是否提升未来任务，并且不会污染上下文。

evaluation
Grounded

可追溯

每条记忆能追到原始消息、工具输出、工单、日志或人工结论。

Useful

有下游收益

减少追问、提高成功率、降低返工，而不是只是“总结得漂亮”。

Specific

具体

保留专名、数字、日期、条件，不写“用户喜欢高质量服务”这类空话。

Temporal

有时间边界

区分现在如此、曾经如此、试用期间如此、已被新事实否定。

Safe

可治理

权限、审计、版本、删除、回滚、敏感信息过滤都在系统层有安排。
